# UI 设计灵感参考

> 本页收录 Radish 后续 UI 设计治理可参考的现代产品界面样本与提炼原则。
>
> 这些截图只作为布局、信息密度、配色节奏、控件气质和页面协调性的灵感参考；后续设计必须结合 Radish 的产品内容、淡雅新中式主题、`@radish/ui`、主题 token 和 Pencil 设计稿重新创作，不直接复刻任何外部产品。

## 使用边界

- 本页服务于 `P3-8 多端功能补全与 UI 设计治理`，尤其是 UI 端点审计、Pencil 设计稿创建和页面级改造前的设计判断。
- 参考重点是“如何组织信息、控制密度、处理留白、建立层级、使用小面积强调色”，不是复制外部产品的品牌、图标、组件细节或页面结构。
- 后续页面级或客户端级 UI 优化，原则上先产出对应 `.pen` 设计稿，再进入实现。
- 若本页原则与 [视觉主题规范](/frontend/visual-theme-spec) 冲突，以 Radish 既有主题方向和语义 token 体系为准。

## 总体设计语言

Radish 后续 UI 可以吸收这些产品的共同优点：

- **稳定结构**：高频工作台优先使用清晰的侧栏、主内容区、可选辅助栏；不要让每个应用自创一套导航和内容骨架。
- **温和密度**：信息要足够可扫描，但不堆满屏幕；列表、表格、设置项和图表之间保留可感知的分区节奏。
- **浅层承托**：用浅底、弱边框、细分割线和低强度阴影承托内容，减少大面积重色块和厚重卡片。
- **小面积强调**：品牌色、状态色和危险色只用于按钮、状态点、标签、选中条、链接和关键数字，不铺满大面积背景。
- **圆润但克制**：控件可以温润圆润，但圆角、阴影和背景层级要服务于阅读和点击，不做玩具化视觉。
- **文本层级明确**：标题、说明、元信息和操作文案应有稳定字号、字重和颜色层级，不能只靠颜色区分重要性。

## 布局参考

### 工作台与文档

AFFINE 的三张图最适合参考到 `WikiApp`、文档工作台、资源管理和内容编辑类页面：

- 左侧导航保持窄而稳定，承载空间、目录、分类和常用入口。
- 中心内容区保持主导地位，列表或正文不要被辅助信息挤压。
- 右侧辅助栏只放日历、属性、目录、元信息、筛选或快捷动作，不承担主流程。
- 顶部工具条应轻量排列图标、视图切换和主动作，避免把说明性文案堆在工具栏。
- 文档正文页应控制阅读宽度，属性信息和正文之间通过分区线、标题和留白建立层级。

### 设置与权限

CodexApp 与 GitHub 设置页适合参考到 Console 设置页、桌面偏好设置、权限配置和账号治理页面：

- 设置页主内容不应铺满超宽屏，建议使用居中内容列或固定最大宽度。
- 左侧设置导航负责分组，正文只展示当前分组，避免“设置总表”式堆叠。
- 行级设置优先采用“标题 + 简短说明 + 控件”的横向结构。
- 多个开关、选择框、权限项应组成分组面板，用细分割线划分行，而不是每行一个重卡片。
- 危险操作使用弱红色文本、浅红底或独立分组表达风险，不抢占普通设置页面的整体语气。

### 管理后台与运维页面

Cloudflare 与 1Panel 适合参考到 `radish.console`、系统监控、附件治理、应用管理和运营分析页面：

- 仪表盘首屏优先展示关键指标卡、近期变化、待处理事项和常用动作。
- 表格型页面需要稳定的工具条、筛选区、表头、操作列和空态，不要用卡片替代表格。
- 图表页应先给出时间范围、指标含义和图例，再展示趋势；多个图表之间保持统一间距和标题样式。
- 卡片栅格适合应用市场、插件、商品或模板类对象，但每张卡只保留最关键的图标、名称、说明、标签和主动作。
- 管理后台可使用更高信息密度，但仍要通过行高、分割线、弱底色和固定列宽保持扫描效率。

### 社区与列表

Discourse 与 GitHub 列表适合参考到论坛、通知、公开搜索、榜单和个人公开页：

- 列表项应突出标题，元信息、标签、作者、时间、回复数和热度列作为可扫描辅助信息。
- 标签用低饱和色和短文本，不能让标签色盖过标题。
- 头像、状态点、徽标和计数可以帮助定位，但同一行内不应出现过多强视觉元素。
- 社区话题列表可以比内容阅读页更紧凑；详情页和编辑页则应优先保证阅读宽度和操作安静。
- 聊天页应保持频道侧栏、消息流和输入区的稳定关系，输入区贴底但不遮挡消息阅读。

## 色彩与视觉节奏

Radish 已有淡雅新中式方向，不应改成外部产品的蓝白 SaaS 风格。可吸收的不是具体颜色，而是用色方法：

- 背景保持低饱和、低对比，形成纸感或浅雾感。
- 主品牌色和模块辅助色只用于强调，不承担大面积铺底。
- 选中态可以使用浅底 + 细强调条 + 图标色，不必整块高饱和填充。
- 链接色应稳定，避免在同一页面同时出现多套蓝、绿、红强调。
- 图表和状态色必须优先表达含义；装饰性配色不能干扰数据判断。

## 信息密度规则

后续 UI 审计时，可以按页面类型判断密度：

| 页面类型 | 推荐密度 | 参考方向 |
|----------|----------|----------|
| 文档列表 / 文件列表 | 中高 | AFFINE 列表、GitHub 文件列表、1Panel 文件表格 |
| 文档阅读 / 编辑 | 中低 | AFFINE 编辑器，正文宽度优先 |
| 设置页 / 权限页 | 中 | CodexApp、GitHub 设置页，行级设置清晰 |
| 管理后台仪表盘 | 中高 | Cloudflare、1Panel，指标和图表可扫描 |
| 社区话题列表 | 中高 | Discourse、GitHub，标题和元信息分层 |
| 聊天页 | 中 | Discourse 聊天，消息流优先 |
| 移动公开页 | 中低 | 保留主路径，减少并列栏和次级操作 |

## 转化为 Radish 设计稿

`P3-8-A` 启动 UI 审计时，建议按以下顺序使用本页：

1. 先确定端点：公开页、移动 Web 公开视图、WebOS 工作台、Flutter、Tauri / PC 壳、Console。
2. 再选参考维度：布局、信息密度、设置表单、列表、仪表盘、聊天、编辑器。
3. 用 Pencil 画 Radish 自己的 `.pen` 设计稿，保留产品语义、主题 token 和组件边界。
4. 在设计稿中标注可复用的 `@radish/ui` 组件、需要新增的 token、需要收敛的历史页面。
5. 只选择 `1-2` 个一天级可验证批次进入实现，避免一次性大改所有 UI。

## 禁止事项

- 不直接复制外部产品的布局、图标、品牌色、文案或完整页面。
- 不因为参考了 CodexApp / AFFINE 的圆润浅色风格，就取消 Radish 的淡雅新中式主题方向。
- 不把所有页面都改成卡片堆叠；表格、列表、编辑器、设置页和仪表盘应各有合适结构。
- 不绕过 Pencil 设计稿直接大面积改 UI。
- 不用硬编码颜色快速模拟参考图效果；所有可持续复用的颜色必须沉淀为语义 token。
