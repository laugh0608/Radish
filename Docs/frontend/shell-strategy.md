# 前端多壳层策略

> 状态：执行中
>
> 最后更新：2026-05-06（Asia/Shanghai）
>
> 本页只保留壳层分工、归属判断和当前边界。公开入口落地流水、RC 记录和批次细节不写入本页。

## 结论

Radish 不再把 **WebOS 桌面** 当作所有用户、所有设备、所有场景的唯一入口。

第二开发阶段的前端策略固定为 **多壳层分工**：

| 壳层 | 主要用户 | 主要设备 | 主要任务 | 交互形态 |
| --- | --- | --- | --- | --- |
| 公开内容壳层 | 游客、轻登录用户、新用户 | 手机 / 平板 / 桌面浏览器 | 浏览帖子、查看详情、访问公开主页、阅读文档、轻互动 | 单任务、直达内容、弱窗口感 |
| 桌面工作台壳层 | 已登录重度用户 | 桌面浏览器 / Tauri 桌面安装包 | 聊天、通知中心、个人中心、创作、继续使用、多窗口切换 | Dock、窗口系统、工作台 |
| 移动原生客户端壳层 | 高频移动用户 | Android 已完成 MVP，iOS 后续评估 | 高价值社区访问、通知回流、个人使用链路 | 原生导航、原生手势、原生生命周期 |
| 桌面安装包壳层 | 桌面安装包用户 | Windows / macOS / Linux | 分发 WebOS 工作台、承接系统级桌面能力 | Tauri 系统壳 + WebOS UI |

这不是推翻 WebOS，而是把 WebOS 从“唯一入口”收缩为“桌面工作台”。

## 当前多端方向

1. **公开内容壳层**
   - 面向匿名访问、外链分享、搜索流量、PC 浏览器与移动浏览器
   - 已完成 forum / docs / `u/:id` / leaderboard / shop / discover 首批公开入口收口
   - 当前转入稳定维护，不继续扩页面细节
2. **桌面工作台壳层**
   - 面向桌面浏览器、Tauri 桌面安装包、已登录用户与高交互场景
   - 当前产品功能开发优先 WebOS 桌面工作台、继续使用、启动续接、高价值应用回流
3. **移动原生客户端壳层**
   - Flutter Android MVP 已完成第一轮 RC 验收
   - iOS 后续按移动端价值单独评估
   - 不把 Windows / macOS / Linux 作为 Flutter 默认扩平台目标
4. **桌面安装包壳层**
   - Tauri 负责系统窗口、系统浏览器 loopback 登录回跳、deep link 兼容与安装包承载
   - UI 默认复用 WebOS 桌面工作台，而不是公开内容壳层
   - 签名、自动更新、菜单、托盘、文件系统与公开分发链路后置到真实对外分发前

## 功能归属

### 优先进入公开内容壳层

- 论坛列表、帖子详情、公开评论阅读
- 轻回应墙与轻量互动展示
- 公开个人主页
- 文档阅读与搜索
- 公开榜单、公开商城浏览、社区发现分发

### 优先保留在桌面工作台壳层

- 聊天室
- 通知中心完整管理
- 个人中心完整能力
- 桌面首页“继续使用”复访入口，例如最近应用、最近浏览、我的轻回应
- 发帖创作器与更重的编辑体验
- 多窗口切换与工作台级使用场景

### 按价值再判断是否双端承载

- 商城购买与登录后链路
- 排行榜 / 经验展示
- 萝卜坑的部分公开 / 轻量能力
- 更深的社区分发页

原则不是“所有能力都双端同步”，而是“高价值路径在正确壳层里自然成立”。

## 路由与迁移策略

- 当前保持增量迁移，不立即推翻 WebOS 路由结构
- 不立即把 `/` 从桌面入口切成公开社区首页
- 公开内容直达路由已经覆盖 `/discover`、`/forum`、`/forum/post/:id`、`/docs`、`/docs/:slug`、`/u/:id`、`/leaderboard`、`/shop` 等首批路径
- 公开内容主路由坚持使用服务端可见的真实路径，不采用 `/shell#page` 或 `/desktop#...` 这类 hash shell 作为公开资源地址
- `/desktop` 继续作为 WebOS 桌面工作台入口
- WebOS 可在 `/desktop` 内部使用 hash、query 或本地状态表达窗口恢复和局内导航，但这些状态不构成公开 URL 契约
- Tauri 桌面安装包默认入口已切到 `/desktop`

## 当前明确不做

- 不做“移动版 WebOS”
- 不做“所有页面都同时支持桌面窗口和移动壳层”
- 不把公开内容强行塞回窗口系统
- 不在 Flutter MVP 阶段复刻桌面工作台
- 不把 Capacitor 作为移动端产品化路线
- 不把 Tauri 理解为原生 UI 重写路线
- 不把 `/docs` 公开阅读页作为 Tauri 桌面安装包正式默认入口
- 不在个人开发阶段继续消耗主线精力处理 Tauri 签名、自动更新、SmartScreen、生产 Auth 或公开分发链路

## 判断规则

讨论某个功能放哪里时，优先回答四个问题：

1. 它是“看内容”还是“做事情”？
2. 它主要发生在公开访问，还是登录后的长期使用？
3. 它在手机上是否应该一跳直达？
4. 它是否真的需要多窗口和工作台语义？

- 如果偏公开、偏直达，优先进入公开内容壳层
- 如果依赖多任务、持续操作和桌面组织方式，优先留在 WebOS 工作台壳层
- 如果需要移动安装包、原生返回、系统浏览器登录回跳和移动生命周期，优先进入 Flutter 移动客户端
- 如果需要桌面安装包、系统菜单、托盘、自动更新或文件系统能力，优先评估 Tauri 壳承接，但 UI 仍优先复用 WebOS 工作台
