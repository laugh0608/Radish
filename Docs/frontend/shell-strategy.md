## 1. 结论

Radish 不再把 **WebOS 桌面** 当作所有用户、所有设备、所有场景的唯一入口。

第二开发阶段的前端策略固定为 **多壳层分工**：

1. **公开内容壳层（Public Content Shell）**
   - 面向匿名访问、外链分享、搜索流量与移动端浏览
   - 重点承载“看内容、轻互动、低门槛进入”
2. **桌面工作台壳层（Desktop Workspace Shell）**
   - 面向桌面端、已登录用户与高交互场景
   - 重点承载“多任务、创作、消息、通知、沉浸式使用”
3. **原生客户端壳层（Flutter Client Shell）**
   - 面向 Android 起步，并逐步扩到 Windows / Linux
   - 重点承载“移动 / 原生环境中的高频社区访问”

这不是推翻 WebOS，而是把 WebOS 从“唯一入口”收缩为“桌面工作台”。

## 2. 为什么要这么做

如果继续把 WebOS 同时当成：

- 公开社区首页
- 桌面工作台
- 移动端入口
- 未来 Flutter 客户端参考形态

那么复杂度会随着“应用数量 × 设备类型 × 窗口状态 × 登录态”成倍增长。

当前仓库已经明显出现这个信号：

- `radish.client` 以桌面 / 窗口系统为核心
- 个别页面虽有响应式处理，但并不等于移动产品形态
- 继续把手机体验理解为“压缩版窗口系统”，会持续放大适配成本

因此，当前最重要的不是继续给 WebOS 叠适配，而是先划清职责边界。

## 3. 三类壳层的职责分工

| 壳层 | 主要用户 | 主要设备 | 主要任务 | 交互形态 |
| --- | --- | --- | --- | --- |
| 公开内容壳层 | 游客、轻登录用户、新用户 | 手机 / 平板 / 桌面浏览器 | 浏览帖子、查看详情、访问公开主页、阅读文档、轻互动 | 单任务、直达内容、弱窗口感 |
| 桌面工作台壳层 | 已登录重度用户 | 桌面浏览器 | 聊天、通知中心、个人中心、内容创作、多窗口切换 | 桌面、Dock、窗口系统 |
| Flutter 客户端壳层 | 高频移动用户 | Android 起步，后续 Windows / Linux | 高价值社区访问、通知回流、个人使用链路 | 原生导航、原生手势、原生生命周期 |

## 4. 功能归属建议

### 4.1 优先进入公开内容壳层

- 论坛列表
- 帖子详情
- 轻回应墙与轻量互动
- 公开个人主页
- 文档阅读

### 4.2 优先保留在桌面工作台壳层

- 聊天室
- 通知中心完整管理
- 个人中心完整能力
- 发帖创作器与更重的编辑体验
- 多窗口切换与工作台级使用场景

### 4.3 后续按价值判断是否双端承载

- 商城购买与登录后链路
- 排行榜 / 经验展示
- 萝卜坑的部分公开 / 轻量能力
- 更深的社区分发页

原则不是“所有能力都双端同步”，而是“优先让高价值路径在正确壳层里自然成立”。

## 5. 路由与迁移策略

### 5.1 当前阶段不做的事

- 不立即推翻现有 WebOS 路由结构
- 不立即把 `/` 从桌面入口切成公开社区首页
- 不因为移动端规划而重写整个 `radish.client`

### 5.2 推荐迁移顺序

1. **先冻结分工，不急着迁根路由**
   - 第二阶段先把“什么应该在公开内容壳层、什么应该留在桌面工作台”写清楚
2. **再新增公开直达路由**
   - 例如：`/forum`、`/forum/category/:id`、`/forum/tag/:tagSlug`、`/forum/question`、`/forum/poll`、`/forum/lottery`、`/forum/search`、`/forum/post/:id`、`/docs`、`/docs/search`、`/docs/:slug`、`/u/:id`、`/leaderboard`、`/leaderboard/:type`、`/shop`、`/shop/products`、`/shop/product/:id`
3. **最后再评估入口切换**
   - 等公开内容壳层稳定后，再决定 `/` 是否继续保留桌面入口，还是改成公开社区首页，并把桌面入口收束为 `/desktop`

### 5.3 当前建议

第二阶段前半程，优先采用 **增量迁移**：

- 保留现有 WebOS 入口不动
- 为公开内容逐步增加直达形态
- 避免“先做大搬家，再做业务”的高风险顺序

## 6. 与移动 Web 的关系

移动 Web 不是“给桌面窗口系统做更多窄屏补丁”，而是：

- 为公开内容壳层建立自然的移动访问形态
- 优先保证帖子列表、帖子详情、公开主页、文档阅读在手机上好用
- 登录后高交互能力再按价值逐步扩展

这意味着第二阶段的移动 Web 建设，应该从 **公开内容壳层** 起步，而不是从 `WindowManager` 起步。

## 7. 与 Flutter 的关系

Flutter 客户端不是 WebOS 的原样翻版。

Flutter 的职责是：

- 复用现有业务接口、认证契约与主题语义
- 用原生导航和原生交互承载高价值社区链路
- 优先覆盖 Android 高频场景，再扩展到 Windows / Linux

Flutter 不承担“把桌面工作台搬到原生端”的任务。

## 8. 第二阶段执行顺序建议

1. 冻结多壳层分工和迁移策略
2. 继续推进 `Phase 2-1` 的社区深化第一批，例如论坛轻回应墙
3. 在 `Phase 2-2` 中为论坛、公开主页、文档阅读补公开直达入口
4. 公开内容壳层稳定后，再评估根路由与桌面入口关系
5. 最后进入 Flutter MVP

- `Phase 2-1` 当前已完成论坛轻回应墙与最小回流链路收口
- 当前主线已正式切到 `Phase 2-2`
- `Phase 2-2` 首批只冻结 forum 公开列表与帖子详情的移动访问形态，不做“移动版 WebOS”

- 公开内容壳层当前已完成第一轮阅读体验收口：列表状态已支持 URL 回写与“详情 -> 返回列表”上下文保留，详情页已补齐只读阅读摘要
- forum 公开标签首批当前已通过 `/forum/tag/:tagSlug` 接入公开内容壳层，首批只承载标签上下文、帖子列表阅读、排序分页与帖子详情阅读
- forum 公开标签当前已继续收口到 canonical ASCII slug：旧中文标签名直链进入后，也会在公开壳层内回写到 `/forum/tag/:tagSlug`
- forum 公开结构化类型列表当前已通过 `/forum/question`、`/forum/poll` 与 `/forum/lottery` 接入公开内容壳层；帖子卡片与详情里的结构化徽标也已统一回落到对应类型列表
- forum 公开搜索首批当前已通过 `/forum/search` 接入公开内容壳层，首批只承载关键词检索、时间范围过滤、排序分页与帖子详情阅读；搜索结果中的标签点击会继续收口到公开标签直达链路
- 当前范围继续明确收紧：公开 forum 壳层仍是只读阅读，不开放发帖、评论提交、投票提交、点赞或编辑等桌面交互动作
- 公开 forum 列表当前已形成分层布局口径：手机窄屏保持单列 `publicCompact`，较宽窄屏与平板进入轻量双栏摘要布局
- 公开 forum 评论阅读当前已补齐根评论分页与子评论分页；“加载更多子评论”在公开壳层下已可正常触发
- 公开文档阅读首批当前已完成 `/docs` 与 `/docs/:slug` 落地，并在人工验收后补齐返回目录滚动位置保持、正文内链跳转与旧 `__documents__` 链接兼容
- docs 公开搜索首批当前已通过 `/docs/search` 接入公开内容壳层，首批只承载关键词检索、结果分页与“详情 -> 返回搜索结果”上下文
- docs 公开分享入口首批当前已补到详情阅读页，首批只提供“复制当前公开链接”的最小能力，不扩展海报、分享统计或其他治理动作
- 公开文档阅读当前仍保持只读阅读边界：只承载目录浏览、关键词搜索、正文阅读、复制公开链接、返回浏览态与文档内链跳转，不开放桌面文档应用里的编辑、发布、回收站或版本历史
- 个人公开页首批当前已完成 `/u/:id` 落地，并保持“资料 + 公开帖子 / 评论阅读”边界
- forum 公开分类首批当前已通过 `/forum/category/:categoryId` 接入公开内容壳层，首批只承载分类上下文、帖子列表阅读、排序分页与返回链路
- 公开榜单首批当前已通过 `/leaderboard` 与 `/leaderboard/:type` 接入公开内容壳层，首批只承载榜单阅读、分页与公开个人页跳转
- 公开商城浏览首批当前已通过 `/shop`、`/shop/products` 与 `/shop/product/:id` 接入公开内容壳层，只承载首页、商品列表与商品详情阅读；购买、订单、背包和其他“我的”链路继续留在桌面工作台
- 当前仍保持增量迁移口径，`/` 与 `/desktop` 的根入口关系未调整

## 9. 当前明确不做

- 不做“移动版 WebOS”
- 不做“所有页面都同时支持桌面窗口和移动壳层”
- 不把公开内容强行塞回窗口系统
- 不在 Flutter MVP 阶段复刻桌面工作台

## 10. 当前判断规则

以后讨论某个功能放哪里时，优先回答四个问题：

1. 它是“看内容”还是“做事情”？
2. 它主要发生在公开访问，还是登录后的长期使用？
3. 它在手机上是否应该一跳直达？
4. 它是否真的需要多窗口和工作台语义？

如果前两项偏公开、后两项偏直达，优先进入公开内容壳层。  
如果它明显依赖多任务、持续操作和桌面组织方式，优先留在 WebOS 工作台壳层。
