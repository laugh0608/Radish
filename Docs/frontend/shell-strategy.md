## 1. 结论

Radish 不再把 **WebOS 桌面** 当作所有用户、所有设备、所有场景的唯一入口。

第二开发阶段的前端策略固定为 **多壳层分工**。截至 `2026-05-04` 多端路线评估收口后，长期产品线进一步明确为：

1. **公开内容壳层（Public Content Shell）**
   - 面向匿名访问、外链分享、搜索流量、PC 浏览器与移动浏览器
   - 重点承载“看内容、轻互动、低门槛进入”
2. **桌面工作台壳层（Desktop Workspace Shell）**
   - 面向桌面浏览器、Tauri 桌面安装包、已登录用户与高交互场景
   - 重点承载“多任务、创作、消息、通知、沉浸式使用”
3. **移动原生客户端壳层（Mobile Native Shell）**
   - Android MVP 已由 Flutter 跑通第一轮 RC
   - iOS 后续按 Flutter 单独评估
   - 重点承载“移动安装包里的原生导航、原生生命周期、高频社区访问”
4. **桌面安装包壳层（Desktop App Shell）**
   - 由 Tauri 承接 Windows / macOS / Linux 的系统窗口、系统浏览器 loopback 登录回跳、deep link 兼容、菜单、托盘、自动更新、文件系统与分发能力
   - UI 默认复用 WebOS 桌面工作台，而不是公开内容壳层
   - Tauri 不是原生 UI 重写路线，而是 `WebOS + Rust 原生桥` 的桌面分发方式

这不是推翻 WebOS，而是把 WebOS 从“唯一入口”收缩为“桌面工作台”。

## 2. 为什么要这么做

如果继续把 WebOS 同时当成：

- 公开社区首页
- 桌面工作台
- 移动端入口
- 未来 Flutter 客户端参考形态

那么复杂度会随着“应用数量 × 设备类型 × 窗口状态 × 登录态”成倍增长。

当前仓库已经明显出现这个信号：

- `radish.client` 以桌面 / 窗口系统为核心
- 个别页面虽有响应式处理，但并不等于移动产品形态
- 继续把手机体验理解为“压缩版窗口系统”，会持续放大适配成本

因此，当前最重要的不是继续给 WebOS 叠适配，而是先划清职责边界。

## 3. 四类壳层的职责分工

| 壳层 | 主要用户 | 主要设备 | 主要任务 | 交互形态 |
| --- | --- | --- | --- | --- |
| 公开内容壳层 | 游客、轻登录用户、新用户 | 手机 / 平板 / 桌面浏览器 | 浏览帖子、查看详情、访问公开主页、阅读文档、轻互动 | 单任务、直达内容、弱窗口感 |
| 桌面工作台壳层 | 已登录重度用户 | 桌面浏览器 / Tauri 桌面安装包 | 聊天、通知中心、个人中心、内容创作、多窗口切换 | 桌面、Dock、窗口系统 |
| 移动原生客户端壳层 | 高频移动用户 | Android 已完成 MVP，iOS 后续评估 | 高价值社区访问、通知回流、个人使用链路 | 原生导航、原生手势、原生生命周期 |
| 桌面安装包壳层 | 桌面安装包用户 | Windows / macOS / Linux | 分发 WebOS 工作台、承接系统级桌面能力 | Tauri 系统壳 + WebOS UI |

## 4. 功能归属建议

### 4.1 优先进入公开内容壳层

- 论坛列表
- 帖子详情
- 轻回应墙与轻量互动
- 公开个人主页
- 文档阅读

### 4.2 优先保留在桌面工作台壳层

- 聊天室
- 通知中心完整管理
- 个人中心完整能力
- 发帖创作器与更重的编辑体验
- 多窗口切换与工作台级使用场景

### 4.3 后续按价值判断是否双端承载

- 商城购买与登录后链路
- 排行榜 / 经验展示
- 萝卜坑的部分公开 / 轻量能力
- 更深的社区分发页

原则不是“所有能力都双端同步”，而是“优先让高价值路径在正确壳层里自然成立”。

## 5. 路由与迁移策略

### 5.1 当前阶段不做的事

- 不立即推翻现有 WebOS 路由结构
- 不立即把 `/` 从桌面入口切成公开社区首页
- 不因为移动端规划而重写整个 `radish.client`

### 5.2 推荐迁移顺序

1. **先冻结分工，不急着迁根路由**
   - 第二阶段先把“什么应该在公开内容壳层、什么应该留在桌面工作台”写清楚
2. **再新增公开直达路由**
  - 例如：`/discover`、`/forum`、`/forum/category/:id`、`/forum/tag/:tagSlug`、`/forum/question`、`/forum/poll`、`/forum/lottery`、`/forum/search`、`/forum/post/:id`、`/docs`、`/docs/search`、`/docs/:slug`、`/u/:id`、`/leaderboard`、`/leaderboard/:type`、`/shop`、`/shop/products`、`/shop/product/:id`
3. **最后再评估入口切换**
   - 等公开内容壳层稳定后，再决定 `/` 是否继续保留桌面入口，还是改成公开社区首页，并把桌面入口收束为 `/desktop`

### 5.3 当前建议

第二阶段前半程，优先采用 **增量迁移**：

- 保留现有 WebOS 入口不动
- 为公开内容逐步增加直达形态
- 避免“先做大搬家，再做业务”的高风险顺序

## 6. 与移动 Web 的关系

移动 Web 不是“给桌面窗口系统做更多窄屏补丁”，而是：

- 为公开内容壳层建立自然的移动访问形态
- 优先保证帖子列表、帖子详情、公开主页、文档阅读在手机上好用
- 登录后高交互能力再按价值逐步扩展

这意味着第二阶段的移动 Web 建设，应该从 **公开内容壳层** 起步，而不是从 `WindowManager` 起步。

## 7. 与移动原生客户端的关系

移动原生客户端不是 WebOS 的原样翻版。

当前 Flutter Android MVP 已完成第一轮 RC 验收，说明 Flutter 能承载 Android 高频社区访问链路。但这不自动代表长期多端都应继续扩大 Flutter。

移动原生客户端壳层的职责仍是：

- 复用现有业务接口、认证契约与主题语义
- 用原生导航和原生交互承载高价值社区链路
- 优先保证 Android MVP 已完成成果可维护
- iOS 后续按 Flutter 移动端单独评估
- 不把 Windows / macOS / Linux 作为 Flutter 默认扩平台目标

Flutter 移动端不承担“把桌面工作台搬到原生端”的任务。

## 8. 与 Tauri 桌面安装包的关系

Tauri 不是 WebOS 的替代品，也不是原生 UI 重写路线。

Tauri 的合理角色是：

- 为 Windows / macOS / Linux 提供安装包、系统窗口、系统浏览器 loopback 登录回跳、deep link 兼容、菜单、托盘、自动更新、文件系统和外部链接能力
- 以 WebOS 桌面工作台作为默认 UI 入口
- 在桌面安装包中继续复用 React、`@radish/http`、`@radish/ui`、主题 token 与现有工作台能力

Tauri 不适合作为：

- 移动端替代方案
- 公开内容壳层的默认桌面产品形态
- 用 Rust 重写 Radish UI 的入口

若继续推进 Tauri，下一轮应优先验证：

1. WebOS Dock、桌面图标、窗口系统在 Tauri 固定窗口里的布局与滚动行为人工验收已通过，测试后暂未发现问题
2. 桌面登录 / 登出回跳已通过：系统浏览器 + `127.0.0.1:48801` loopback callback 主路径可用，`radish://` deep link 仅保留为兼容路径
3. Windows NSIS installer 首轮已通过；后续继续验证签名、Windows SmartScreen、自动更新、卸载 / 升级、托盘与系统菜单

## 9. 第二阶段执行顺序建议

1. 冻结多壳层分工和迁移策略
2. 继续推进 `Phase 2-1` 的社区深化第一批，例如论坛轻回应墙
3. 在 `Phase 2-2` 中为论坛、公开主页、文档阅读补公开直达入口
4. 公开内容壳层稳定后，再评估根路由与桌面入口关系
5. 进入 Flutter Android MVP；Android 第一轮完成后，按三端分工重新评估 Android 产品化、iOS 单独评估与 Tauri + WebOS 桌面安装包路线

## 10. 当前明确不做

- 不做“移动版 WebOS”
- 不做“所有页面都同时支持桌面窗口和移动壳层”
- 不把公开内容强行塞回窗口系统
- 不在 Flutter MVP 阶段复刻桌面工作台
- 不把 Capacitor 作为移动端产品化路线
- 不把 Tauri 理解为原生 UI 重写路线
- 不把 `/docs` 公开阅读页作为 Tauri 桌面安装包正式默认入口

## 11. 当前判断规则

以后讨论某个功能放哪里时，优先回答四个问题：

1. 它是“看内容”还是“做事情”？
2. 它主要发生在公开访问，还是登录后的长期使用？
3. 它在手机上是否应该一跳直达？
4. 它是否真的需要多窗口和工作台语义？

如果前两项偏公开、后两项偏直达，优先进入公开内容壳层。  
如果它明显依赖多任务、持续操作和桌面组织方式，优先留在 WebOS 工作台壳层。
如果它需要移动安装包、原生返回、系统浏览器登录回跳和移动生命周期，优先进入 Flutter 移动客户端。
如果它需要桌面安装包、系统菜单、托盘、自动更新或文件系统能力，优先评估 Tauri 壳承接，但 UI 仍优先复用 WebOS 工作台。
