# P3-12-E5-A 用户可恢复错误与反馈硬化首批记录

> 日期：2026-07-06（Asia/Shanghai）
>
> 范围：`Frontend/radish.client`、`Docs/planning`。本批不改后端接口、HTTP 错误模型、日志链路、数据库结构或跨端运行时行为。

## 目标

承接 `P3-12-E4-B` 后的下一顺位，先处理社区主路径中低风险、用户可见、能当天闭合的恢复反馈缺口：

- 发帖失败后，用户需要明确知道草稿没有丢失，并能稍后重试。
- 评论附件上传失败后，用户需要明确知道评论正文仍保留，不必重新输入。
- Workbench 活动同步部分失败时，用户需要知道通知 / 消息入口仍可直接进入，不把局部同步失败理解成整页不可用。

## 代码侧结论

- `PublishPostModal`：
  - 发布失败时新增局部错误条，说明“发布失败，草稿已保留”。
  - 错误信息追加恢复动作：草稿仍保存在本地，检查网络或稍后重试。
  - 保留原本发布成功后清理草稿的行为，不改变 `onPublish` 契约。
- `CreateCommentForm`：
  - 图片 / 文档上传失败时，错误条追加“当前评论草稿仍保留在编辑器中，请检查文件或网络后重试上传”。
  - 中英文文案均补齐，避免英文界面混入中文标点。
- `WorkbenchApp`：
  - 通知或消息活动同步部分失败时，在“今日队列”中新增可恢复项。
  - 用户可直接进入通知或消息处理；数量看起来不完整时再刷新工作台重试。

## 不纳入本批

- 不新增统一诊断编号生成、错误上报 API、Console 工单、后端日志关联或跨端错误模型。
- 不模拟弱网、断网、请求取消或后端异常注入平台。
- 不改变聊天发送失败的现有本地失败态和重试机制；后续可在第二批补诊断复制与更清晰反馈。
- 不改变通知目标缺失路由契约；目标缺失反馈已在 `E4-B` 收口，本批只承接 Workbench 局部同步失败。
