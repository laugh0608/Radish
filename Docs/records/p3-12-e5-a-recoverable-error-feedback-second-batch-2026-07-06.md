# P3-12-E5-A 用户可恢复错误与反馈硬化第二批记录

> 日期：2026-07-06（Asia/Shanghai）
>
> 范围：`Frontend/radish.client`、`Docs/planning`。本批不改后端接口、HTTP 错误模型、通知路由契约、日志链路、数据库结构或跨端运行时行为。

## 目标

承接 `E5-A` 首批的发帖、评论和 Workbench 可恢复反馈，继续补齐社区复访链路里的用户支持证据：

- 聊天消息发送失败后，用户可以重试、撤销，也能复制不含正文内容的诊断上下文。
- 通知目标缺失时，不再只是停留在通知中心说明，而是提供可复制的通知上下文，方便反馈给支持或治理人员。

## 修改摘要

### 聊天发送失败

- `ChatMessageList` 的本地失败消息增加恢复提示：消息仍保留在本地，可重试或复制诊断给支持人员。
- 失败消息动作增加“复制诊断”。
- `ChatApp` 生成诊断上下文时只包含：
  - `channelId`
  - `messageId`
  - `clientRequestId`
  - 本地状态、消息类型、创建时间、错误信息
  - 是否包含正文、是否包含附件
  - 当前页面路径
- 诊断上下文不复制聊天正文，避免把私域消息内容带出会话。

### 通知目标缺失

- 通知队列中有 Web 目标的条目继续使用跳转链接。
- 没有 Web 目标的条目改为手动处理项，不再使用 `#notification-center` 伪跳转。
- 手动处理项提供“复制上下文”按钮，复制内容包含通知 ID、类型、业务类型、业务 ID、时间、已读状态、目标状态、目标提示和当前路径。
- 复制通知上下文不包含通知正文，仅保留定位和支持所需元信息。

## 不纳入本批

- 不新增服务端诊断编号、错误上报 API、Console 工单、后端日志关联或审计查询入口。
- 不改变聊天发送失败的接口载荷、重试请求、幂等键或本地消息状态机。
- 不改变通知目标解析规则；目标缺失仍由现有 `notificationActionQueue` 和 `resolveWebNotificationNavigation` 判断。
- 不执行真实弱网、断网、服务端异常注入或 Gateway smoke；本轮只做前端代码侧验证。
