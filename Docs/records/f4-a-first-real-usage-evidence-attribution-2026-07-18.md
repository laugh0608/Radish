- 仓库内唯一明确的发布后生产 UX 证据，是 2026-07-12 首次管理员初始化入口未统一经过 `BootstrapGate`。该问题为 `P2` 维护项，已由 `54852c73` 在 `dev` 修复，但尚未随新版本部署复核。
- 仓库内存在登录授权、公开论坛、评论区和聊天视觉的明确用户反馈记录，但均来自发布前设计审阅；它们能证明反馈真实存在，不能证明生产用户采用、转化或复访价值。
- `Logs/` 与 `DataBases/*.db` 均被 `.gitignore` 排除，且当前样本明确来自 `Development`、种子账号和本地验收。它们只能证明链路、失败记录与审计入口存在，不能当作真实使用价值证据。
- 2026-07-12 的镜像、DbMigrate、OpenIddict 与 PostgreSQL DateTime 失败是高可信工程运行证据，但均已在 `v26.7.1.1204-release` 前后闭环，不构成当前未处置 `P0/P1`。
- 当前没有生产用户反馈原文、生产六链路聚合、非种子用户行为样本、反馈工单或可关联通知点击与目标回访的证据。

因此本轮不推荐 Docs、聊天、通知、治理、宠物或其他产品功能，不建立功能设计，也不进入大范围代码实现。下一批应继续补采生产证据，而不是从现有页面清单反推选题。

### 2.1 可计入证据

- 明确标注生产环境、时间、版本和影响路径的运行或部署记录。
- 保留原始语义、来源时间和复现上下文的用户反馈；必要时只保存脱敏摘要。
- 能排除种子账号、自动化、开发者 smoke 和人工验收动作的生产业务数据。
- 能把失败、恢复或用户结果关联到具体路径、目标对象、时间和责任模块的审计记录。

### 2.2 只作能力或失败参考

- local / Development 日志、SQLite 数据和本地浏览器运行态。
- 自动化测试、页面可达、构建通过、PC / mobile CSS 视口 smoke。
- Pencil / 页面审阅反馈、开发者受控写入和种子账号操作。
- 候选或发布门禁失败；它们是工程证据，不自动证明用户价值。

### 2.3 可信度

| 等级 | 定义 | 功能选题权重 |

## 6. 问题分组与维护线

### `P0/P1`

- 当前没有未处置 `P0/P1`。
- `OPS-001` 至 `OPS-004` 均为高可信历史失败，已经闭环；不重复进入当前维护线。

### `P2`：顶层入口编排责任分散

- 证据：`UX-001`。
- 共同成因：首次管理员门禁原先由多个私域 `*Entry` 分散包裹，公开入口与 Workbench 没有统一顶层责任。
- 当前状态：`54852c73` 已在 `dev` 完成根因修复；后续随正常批次发布后，从任一产品入口复核首次部署都进入同一初始化门禁。
- 停止线：不修改生产管理员、角色、`SystemBootstrapState` 或旧 release tag，不为该非阻断问题单独紧急补发。

### `P3`：信息密度与具体任务表达

- 证据：`FB-001 / FB-002`。
- 共同成因：旧设计偏大块展示、抽象说明或路由覆盖图，没有优先呈现用户要完成的具体决策与多记录内容。
- 当前状态：相关 Auth、Public、Forum 与 Chat 设计已分批收口；没有生产重复反馈，不再追加新的全站 UI 批次。
- 重新进入条件：同类问题在生产出现至少两个独立反馈，或与任务放弃 / 失败数据形成交叉证据。

### 不分级的证据治理缺口

生产六链路聚合、反馈归档和回流归因不足会阻碍 F4 决策，但当前没有证据表明它本身造成用户损失，因此不冒充 `P2/P3` 产品 bug，也不直接启动“大而全可观测性”。

## 8. 功能选题门槛

下一项完整功能专题至少满足以下条件：

1. 有两个相互独立的真实使用信号，例如两个不同非种子用户，或一条直接反馈与一条生产数据相互印证；单个已验证 `P0/P1` 仍可直接进入维护线。
2. 证据指向同一用户路径与长期结果，不只是页面样式偏好、单次运维失败或已能小范围修复的 bug。
3. 能明确现有数据模型缺口、接口责任、页面入口与权限边界。
4. 能写出停止线和可观测的验证口径，包括成功、失败、恢复与不做范围。
5. 候选之间只选择证据最强的一项；证据强度相同或都不足时继续观察，不以代码量或页面缺口破平。

当前六条链路均未达到上述门槛。下一顺位是 `F4-A` 生产证据补采与反馈归因，不是产品功能实现专题。
