## 背景

`P3-12-D35` 已完成 `radish.console` 表格交互代码侧收口，但这只是 Console 已迁移页面中的一类代码侧问题治理。`P3-12-D` 的完整目标仍包括 Pencil 设计源、设计说明、代码实现、PC / mobile 复核和阶段文档口径一致性。

因此，`P3-12-E` 正式版发布候选不能紧接 D35 启动。下一步继续留在 `P3-12-D`，先整理 UI 专题剩余差距与退出标准，再决定后续成组代码治理、设计源回拉或运行态验收。

## D36+ 退出条件

`P3-12-D` 至少需要满足以下条件后，才能转入 `P3-12-E`：

1. public / private / foundation / console 四类设计源均有页面族对齐矩阵，明确每个画板对应的代码入口、已落地范围和剩余缺口。
2. 对首轮落地页面逐项区分：只需文档确认、需要代码治理、需要 Pencil 回拉、需要 Gateway PC / mobile 运行态观察。
3. Console 表格固定列中宽交互、移动窄屏可读性、操作区换行、弹窗 / 抽屉内表格、分页与工具条布局完成成组判断，不把单页修补误写成专题完成。
4. 公开 Web 与私域 / 作者态页面完成首轮落地状态复核，确认是否还有明显偏离设计源的信息密度、状态槽、移动单列节奏或共享壳层问题。
5. 需要运行态确认的项目统一进入一次阶段性 Gateway PC / mobile 验收；验收前必须等待用户明确说明前后端已启动。
6. `Docs/planning/current.md`、P3-12 专题、相关 frontend 设计端点、记录和 changelog 的阶段口径一致，不把 `P3-12-E` 排在 D36+ 未完成之前。

## 下一步建议

1. 先建立 D36 差距矩阵，不写业务代码。
2. 若矩阵发现代码侧可确认的同类问题，按页面类型成组修复，并保持 API、权限、路由、保存动作和载荷不变。
3. 若剩余问题主要依赖真实视口、数据状态或登录链路观察，再集中申请一次 Gateway PC / mobile 真实联调。
4. 只有 D36+ 退出条件完成后，再准备 `P3-12-E` 发布候选验证矩阵、PR 范围、剩余风险和回滚口径。
