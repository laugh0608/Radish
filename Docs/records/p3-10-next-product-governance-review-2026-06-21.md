# P3-10 后续产品 / 治理增量评审记录

> 记录日期：`2026-06-21`（Asia/Shanghai）
>
> 关联入口：[当前进行中](/planning/current)、[开发路线图](/development-plan)、[P3-10 Web-first 信息架构与下一批开发任务选择](/planning/p3-10-cross-platform-information-architecture)、[论坛内容发布可靠性与编辑历史治理](/guide/forum-content-write-reliability-governance)

## 背景

`WOG-1` 至 `WOG-6` 首轮写操作治理已经完成，论坛内容发布可靠性与编辑历史治理也已覆盖当前确认范围：

- Web 发帖、评论、回答、帖子编辑和评论编辑已接入 `clientSubmissionId` / `ContentSubmissionRecord`。
- Flutter 原生纯文本发帖、根评论 / 回复、问答回答、作者帖子正文编辑和作者根评论编辑已承接同一提交意图字段。
- 创建类频率限制已覆盖发帖、评论 / 回复和回答。
- 编辑历史继续以 `PostEditHistory` / `CommentEditHistory` 为真值，不新增通用编辑历史表。

本轮评审的目标是判断是否继续把 Flutter 子评论编辑、回答编辑或其他跨端写入口作为下一批默认开发项。

## 评审结论

### 1. Flutter 子评论编辑不进入下一批默认代码

子评论编辑的用户价值低于作者帖子正文编辑和根评论编辑，且会引入更复杂的评论树定位、局部刷新、失败重试和实时事件一致性问题。当前 Flutter 已覆盖根评论 / 回复创建和根评论编辑，能够满足移动端主要讨论路径。

后续若真实移动端使用暴露强需求，应按“评论线程编辑能力”单独评审，先明确：

- 子评论定位与详情页局部刷新方式。
- 编辑时间窗、次数限制和历史展示入口。
- 与评论实时事件、回复对象和提交意图 key 的一致性。

### 2. Flutter 回答编辑不并入论坛发布可靠性批次

回答编辑属于问答产品能力，不只是论坛评论编辑的端侧补齐。它需要回答历史、排序、采纳 / 解决状态、通知和详情展示规则一起评审。

后续若要推进，应拆为“问答编辑能力”专题，先确认：

- 回答是否允许作者编辑、编辑时间窗和次数限制。
- 已采纳回答、已解决问题和通知回流的处理规则。
- 回答编辑历史是否复用 `PostEditHistory`、新增 Answer 级历史，或调整现有问答模型。

### 3. 下一批不继续扩大写操作平台

当前不启动 Flutter 转账、完整移动商城、服务端强制资产写入口 key、独立频率限制平台、完整反垃圾系统、完整审核平台、Redis 分布式锁或通用写操作中台。

写操作治理后续进入发布候选前回归线，只在新真实缺口、自动化验证命中或明确跨端承接需求出现时回拉。

### 4. P3-10 下一步转向阶段收束准备

当前更符合整体计划的下一步不是追加新功能入口，而是把 P3-10 已完成的产品 / 治理增量整理成可恢复合并、可进入发布候选整备的状态：

1. 保持 `P3-10-D` PR 准备记录可复用，但本轮不创建 PR。
2. 对 P3-10 已完成项做入口文档和专题文档口径同步，避免新会话误判下一步仍是旧候选。
3. 后续若恢复 `dev -> master` 或进入发布候选整备，再集中执行 `validate:baseline`、`validate:identity`、host runtime 检查和 Gateway PC / 移动真实页面复核。
4. 若新缺口出现，按同类问题成组治理，不把维护观察项扩成新的大范围专题。
