## 触发原因

`P3-12-E6` 原判断将 E1-E5 的阶段性证据解释为“具备进入 P3-12-F 正式版发布候选整备”。2026-07-06 用户人工抽查多个真实页面后指出：页面仍包含大量 debug / 内部说明内容，UI 信息密度偏低，大篇幅大卡片和重复说明不符合正式产品对高频重度使用者的效率要求。

该反馈推翻 E6 原判断。当前不能把 E 期首批硬化等同于正式产品成熟度完成，不能进入 `P3-12-F`。

## 纠偏结论

- `P3-12-E6` 不通过。
- `P3-12-E` 不能收口。
- `P3-12-F 正式版发布候选整备` 暂停进入。
- 下一步回拉 `P3-12-E7 正式 UI 与文案成熟度专项审计`，先形成阻断级缺口矩阵，再按页面族成组修复。

## 问题分类

| 分类 | 表现 | 处理要求 |
| --- | --- | --- |
| Debug / 内部标识 | `PUBLIC DOCS`、`公开入口`、`reaction`、阶段验证式标签。 | 正式页面不得出现开发、路由、调试或验收标识；需要替换为用户能理解的产品语言。 |
| 实现说明文案 | “公开壳层”“正式 Web 私域路由”“桌面治理流程”“这个公开壳层会……”等。 | 从页面正文和说明卡移除实现解释，改为用户任务、边界和下一步动作。 |
| 低信息密度 | 大标题、大卡片、大留白、重复提示占据首屏。 | 高频页面优先展示列表、筛选、关键状态、主要操作和上下文；说明信息收敛到短提示或帮助入口。 |
| 卡片滥用 | 页面区块、说明区、指标和入口均以大卡片呈现。 | 保留卡片用于单个重复项、模态或必要工具；页面分区优先使用紧凑布局和表格 / 列表 / rail。 |
| 移动效率不足 | Console 和 Docs 移动视图纵向堆叠过重，首屏信息少。 | 移动端需要任务化压缩：固定关键动作、减少说明、提高列表密度、避免大块指标卡。 |
| 高价值路径未按重度用户优化 | 论坛详情、Docs 目录、商城列表、Console 仪表盘缺少快速扫描效率。 | 以高频用户任务重新评估首屏、导航、筛选、操作路径和反馈。 |

## 分级清单

### 必须先修复的阻断项

- Public Web、Docs、Forum、Shop、Auth、Console 中的 debug / 内部术语和实现说明文案。
- Public Web 和 Console 代表页的大面积低密度卡片布局。
- Console 移动仪表盘的信息密度和运营任务入口。
- Docs PC / mobile 目录和阅读入口的信息层级。
- 论坛详情轻互动 / reaction / 登录回流区域的正式讨论页语义。

### 发布前建议项

- 建立页面族 UI 与文案成熟度验收矩阵，覆盖 PC 和 mobile。
- 梳理正式产品术语表，禁止在用户界面出现路由、壳层、阶段验收、debug 和内部实现词。
- 重新评估 Public 首页 / 发现、商城、Docs、论坛详情、Auth 授权和 Console 高频页的首屏任务优先级。
- 将说明卡、提示卡和右侧 rail 收敛为用户任务说明，不重复解释已可见导航。

### 可后置项

- 完整设计系统重绘、全站品牌升级、复杂动效和新视觉资产。
- 独立移动 Console App。
- 跨域搜索、推荐算法、完整聊天平台和完整钱包等非本次 UI / 文案成熟度核心问题。

## E7 执行顺位

1. 先做只读页面矩阵：Public Discover、Forum 列表 / 详情、Docs 列表 / 详情 / 移动目录、Shop 列表 / 详情、Auth 授权、Console 仪表盘 / 内容治理 / 订单 / 文档治理。
2. 将每页问题分成 `debug 内容`、`内部术语`、`低信息密度`、`大卡片滥用`、`移动效率`、`高频任务断点`。
3. 输出成组修复顺位：先清 debug / 内部术语，再收敛说明卡和首屏密度，最后处理移动效率和高频任务流。
4. 代码修复按页面族推进，修完一组跑对应 workspace build / type-check；真实 Gateway smoke 需用户当日确认服务已启动。
