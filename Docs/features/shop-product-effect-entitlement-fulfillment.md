# 商城商品效力与权益履约专题

> 状态：F1-A / F1-B / F1-C 已完成本地实现与静态回归，下一切片为 F1-D
>
> 当前阶段：发布后功能完成线 `F1`
>
> 复核日期：2026-07-13

## 摘要

Radish 已具备商品浏览、支付口令、购买幂等、萝卜币扣款、订单、库存、背包 / 权益发放、来源回访和 Console 订单排障。本专题不重写商城，而是补齐“购买完成后商品真正产生什么效果、效果如何持续、失败后如何安全恢复”的长期契约。

F1 先治理订单履约安全，再完成现有消耗品的幂等使用和审计，随后建立可被主题系统复用的权益选择模型，并首批开放低风险的徽章与称号。`F1-A` 已于 2026-07-13 完成订单失败阶段、支付证据门禁、订单快照发放、明确履约资源字段和 schema ledger 迁移；`F1-B` 已于同日完成改名卡、经验卡和萝卜币红包的原子使用、结果回放、正式 Web 幂等提交与 Console 流水回看；`F1-C` 已完成唯一权益指针、服务端 UTC 状态、过期物化、管理员撤销和通用流水。主题权益的实际页面切换归属 `F2`；F1-D 完成真实消费面前，徽章、称号及其余权益类型继续关闭。

## 一、现状判断

### 1.1 已完成基础

- `OrderService` 已把支付口令校验、购买请求幂等、库存扣减、订单创建、萝卜币扣款和发放放在购买主链路中。
- `OperationIdempotencyRecord` 已能绑定购买请求摘要和终态响应。
- `UserBenefit.SourceOrderId` 具备订单级唯一约束；消耗品通过 `UserInventoryGrantRecord` 保证同一订单只发放一次。
- 正式 Web 已承接 `/shop/orders`、`/shop/order/:orderId`、`/shop/inventory`，WebOS `/desktop` 只保留历史兼容入口。
- Console 已能查看订单、商品、用户和扣款流水，并重试失败订单的发放。
- 改名卡、经验卡和萝卜币红包已有后端使用逻辑；正式 Web 背包已有使用入口。

### 1.2 启动审计确认的缺口

1. `[F1-A 已治理]` `OrderStatus.Failed` 曾同时表示“扣款失败”和“已扣款但发放失败”，管理员重试也没有先证明订单已经支付。
2. `[F1-A 已治理]` 失败重试曾重新读取当前 `Product`，商品被修改或删除后可能偏离用户购买时的履约意图。
3. `[F1-A 已治理]` `Order.UserBenefitId` 曾同时承接权益 ID 和消耗品背包 ID。
4. `[F1-A 已治理]` 固定日期有效期曾未完整写入订单快照。
5. `[F1-B 已治理]` 消耗品扣减曾缺少“同一次用户动作”的幂等键、使用流水和业务结果回放。
6. `[F1-C 已治理]` 权益曾只用 `IsActive / IsExpired` 表示状态，同类型权益的唯一激活只靠逐条更新，无法形成跨并发请求的稳定唯一真相。
7. 所有 `BenefitType` 仍由可用性策略禁止购买和激活；后端、Client、Console 各自维护相似的不可用类型判断。
8. `[F1-C 已治理]` `UserBenefitVo.VoDurationDisplay` 曾自行使用本地当前时间计算，未统一使用服务端 UTC 状态口径。

### 1.3 旧文档使用口径

`Docs/guide/shop-*` 中的数据表、示例枚举和示例 Service 只作为历史设计背景；凡是与当前实体和本专题冲突的内容，以当前代码事实和本专题为准。后续实现完成后再回写商城入口文档，不在 F1 内重写全部历史说明。

## 二、目标与停止线

### 2.1 目标

- 订单必须能明确回答：是否已支付、是否已履约、失败发生在哪一阶段、能否安全重试、最终发放到哪个资源。
- 商品发放只能使用订单快照，不能因商品后续编辑而改变历史订单的履约结果。
- 每次消耗品使用必须具备原子扣减、效果写入、幂等回放和可追溯记录。
- 权益拥有记录与“当前启用哪一个”分离，同一用户同一权益类型最多存在一个当前选择。
- 页面展示、业务效果判断和 Console 排障读取同一状态口径。
- 只有真实消费页面和验证闭环已经完成的商品类型才允许上架。

### 2.2 停止线

- 不实现实物商品、购物车、优惠券、赠送、交易市场、提现或完整退款平台。
- 不恢复帖子置顶卡、帖子高亮卡；两者已经因社区公平性废弃，只保留历史枚举读取能力。
- 不在 F1 实现双倍经验卡。经验倍率必须先覆盖全部经验发放入口并保证一次结算只应用一次倍率。
- 不在 F1 实现抽奖券；它归属抽奖专题，不与普通背包使用混合。
- 主题权益只建立通用拥有 / 选择契约，实际主题资源、首屏加载、跨设备切换和失效回退归属 `F2`。
- 头像框、签名档、用户名颜色和点赞特效在对应公开消费面完成前继续禁止上架。
- 不为了 F1 解冻 Flutter 或扩展 Tauri；正式产品入口只实现纯 Web，WebOS 保持阻断级兼容。

## 四、真相源与状态模型

### 4.1 真相源分工

| 模型 | 唯一职责 |
|------|----------|
| `Product` | 当前可售配置；历史订单不得依赖它恢复履约 |
| `Order` | 购买、支付和履约意图的不可变快照，以及订单当前结果 |
| `CoinTransaction` | 扣款或补偿的资产事实 |
| `UserBenefit` | 用户拥有的一份持续权益 |
| `UserActiveBenefit` | 用户每种权益类型当前选择的唯一指针 |
| `UserInventory` | 用户某类同值消耗品的当前聚合数量 |
| `UserInventoryGrantRecord` | 每个订单的消耗品发放事实 |
| `ShopEntitlementOperation` | 使用、启用、停用、过期、撤销等动作及结果的业务流水 |

审计日志用于安全审阅，不替代以上业务真相；业务流水也不保存支付口令、完整异常堆栈或用户敏感输入。

### 4.2 订单状态

继续保留现有 `OrderStatus` 作为用户可见主状态，同时新增明确的失败阶段：

```text
Pending -> Paid -> Completed
   |
   +-> Cancelled
   +-> Failed(Payment)

Paid -> Failed(Fulfillment) -> Completed
```

- `FailureStage=None`：没有失败。
- `FailureStage=Payment`：没有形成有效扣款事实，不允许重试发放。
- `FailureStage=Fulfillment`：支付事实成立，但商品发放未完成，允许受权管理员重试。
- 只有同时满足 `Status=Failed`、`FailureStage=Fulfillment`、`PaidTime != null`，且 `CoinTransactionId` 指向同租户、同用户、`BusinessType=Order`、`BusinessId=OrderId`、金额等于订单总价的成功消费流水时，订单才可进入发放重试。
- `Completed` 表示支付和发放都完成，不表示权益一定处于启用状态。
- 通知失败不回退订单完成状态，由可靠 Outbox 独立重试。

订单增加明确的履约资源字段：权益订单写 `GrantedBenefitId`，消耗品订单写 `GrantedInventoryId`。不再用名为 `UserBenefitId` 的字段存放背包 ID。

### 4.3 权益状态

`UserBenefit` 表示“拥有”，`UserActiveBenefit` 表示“当前选择”。有效状态在服务层统一派生：

```text
RevokedAt != null                -> Revoked
ExpiresAt != null && ExpiresAt <= now -> Expired
存在有效 UserActiveBenefit 指针  -> Active
否则                              -> Available
```

- 过期判断和业务授权必须实时比较 UTC 时间，不能只依赖定时任务写入的 `IsExpired`。
- `IsExpired` 在迁移期只作为查询优化和兼容字段，不再是唯一授权真相。
- 现有 `IsActive / ActivatedAt` 迁移到 `UserActiveBenefit` 后停止作为当前选择真相。
- `EffectiveAt` 继续表示拥有开始时间；F1 有效期从订单履约完成时开始，不延迟到首次启用。
- 撤销是管理员治理动作，不等同于退款；撤销后不能重新启用，必须记录操作者、时间和原因。

### 4.4 消耗品状态

消耗品继续按 `(TenantId, UserId, ConsumableType, ItemValue)` 聚合数量：

- `Quantity > 0`：可用。
- `Quantity = 0`：保留聚合行和历史流水，但默认不出现在用户背包。
- 每个来源订单仍由 `UserInventoryGrantRecord` 独立记录，不把聚合行的 `SourceProductId` 误写成完整来源历史。
- 每次使用通过 `ShopEntitlementOperation` 记录数量、目标、效果资源和幂等结果。

## 五、购买与履约契约

### 5.1 下单快照

订单必须保存足以独立完成发放的快照：

- `ProductType / BenefitType / ConsumableType`
- `BenefitValue`
- `DurationType / DurationDays / FixedExpiresAt`
- 商品名称和图标附件快照
- 数量、单价、总价、库存类型

失败重试只根据订单快照构造发放命令。商品当前是否下架、改名、改值或被软删除，不改变历史已支付订单的履约内容。

### 5.2 发放规则

- 权益类订单一张订单产生一条 `UserBenefit`，通过来源订单唯一约束防重。
- 消耗品订单按购买数量增加聚合背包，并通过 `UserInventoryGrantRecord.SourceOrderId` 防重。
- 发放成功后先写清履约资源 ID，再把订单置为 `Completed`。
- 发放失败保留已支付事实，把失败阶段标为 `Fulfillment`，不静默退款、不重复扣款。
- 管理员重试必须复用同一订单和同一来源唯一键；重复点击返回既有履约结果。

### 5.3 重复购买

- 消耗品允许重复购买，数量叠加；仍受库存和 `LimitPerUser` 约束。
- 同一用户已经拥有未撤销、未过期且 `BenefitType + BenefitValue` 相同的持续权益时，默认拒绝再次购买。
- F1 不实现“购买后延长既有有效期”；需要续期时另行定义订单来源、时间叠加和退款边界。
- 商品下架只停止新购买，不撤销已拥有权益，不清除背包数量。

### 5.4 退款与补偿

- F1 不提供用户退款入口。
- 支付未成立时不产生权益；已经支付但无法发放时优先安全重试发放。
- 必须人工补偿时，萝卜币补偿、权益撤销和订单备注分别留下关联业务编号，不通过直接改余额或软删订单掩盖问题。

## 六、消耗品使用契约

### 6.1 请求幂等

正式 Web 每次使用生成 `inventory-use:{uuid}` 幂等键。服务端请求摘要至少包含：

- `InventoryId`
- `Quantity`
- 操作类型
- 规范化后的目标 ID；改名卡只在摘要中保存新展示名的安全摘要，不在幂等日志重复保存敏感输入

同一键同一请求返回第一次结果；同一键不同请求返回冲突；处理中请求提示用户查询背包或稍后重试，不再次扣减。

### 6.2 原子性

- 条件扣减和真实效果写入必须位于同一 Main 数据库事务。
- 效果写入失败时回滚背包扣减和使用流水。
- 经验卡使用业务键绑定使用操作，不能因服务重试重复增加经验。
- 萝卜币红包必须产生唯一资产流水，业务键绑定使用操作，不能用普通赠送接口形成不可识别的重复奖励。
- 改名卡必须复用用户展示名的完整格式、保留字、频率和唯一性规则；不能只检查 `UserName` 字段相等。

### 6.3 结果

使用结果返回：操作流水 ID、是否为幂等回放、剩余数量、效果类型、效果值和可回看的业务资源。前端成功后只刷新背包和直接受影响的用户 / 资产 / 经验状态，不刷新整个应用。

## 七、持续权益契约

### 7.1 当前选择

`UserActiveBenefit` 使用 `(TenantId, UserId, BenefitType)` 唯一键：

- 启用前校验权益归属、未撤销、未过期和类型已开放。
- 启用通过原子 upsert 把类型指针切换到目标权益。
- 重复启用同一权益返回成功，不制造重复流水。
- 停用只在当前指针确实指向目标权益时删除；重复停用返回已停用结果。
- 并发启用同类型不同权益后，数据库中最终只能保留一个指针。

### 7.2 过期

- 所有读取和效果消费都实时应用 `ExpiresAt <= now` 失效规则。
- 定时任务负责物化过期标记、清理当前选择、创建操作流水和发送到期通知，不承担正确性的唯一责任。
- 过期任务可重入；同一权益只能形成一条最终过期事实。

### 7.3 首批身份权益

F1 首批只开放 `Badge` 和 `Title`：

- 通过共享只读 `UserAdornmentVo` 向公开主页、帖子作者和评论作者暴露当前有效徽章 / 称号。
- 只返回展示所需的资源标识、名称和安全附件 URL，不暴露来源订单、价格或内部权益 ID。
- 页面无权益、权益过期或资源失效时回到普通身份展示，不保留幽灵占位。
- 移动视图必须允许称号换行或省略，不挤压作者名、楼层和操作按钮。

## 八、API 与页面边界

### 8.1 API

保留现有商城查询、购买、订单和背包接口，F1 调整或新增：

- 商品能力元数据：返回类型是否可售、是否可启用、配置约束和不可用原因，供 Console 与 Client 展示。
- `UseItem` 请求增加 `IdempotencyKey`。
- 改名卡改为 body DTO，包含 `InventoryId / NewDisplayName / IdempotencyKey`；旧 query 入口标记废弃并在后续 API 版本移除。
- 权益启用 / 停用返回结构化结果和当前选择，不只返回 `bool`。
- Console 增加受权的权益撤销和履约重试接口；两者必须写操作流水。

所有外部 ID 继续按字符串 LongId 传递；Controller 只编排用户身份和 HTTP 契约，履约规则留在 Service / Repository。

### 8.2 正式 Web

- `/shop/inventory` 同时展示 `Available / Active / Expired / Revoked` 状态及来源。
- 只有后端能力矩阵确认可用的项目显示“使用”或“启用”。
- 使用和启用操作具有处理中状态，重复点击不会生成新幂等键。
- 成功结果明确展示真实效果和剩余数量；失败保留当前背包上下文和可复制诊断。
- 徽章 / 称号启用后，当前页面和公开身份消费面按受控状态刷新。

WebOS 复用同一 API、类型和业务 Hook，只保持历史窗口导航；不为 `/desktop` 维护第二套履约规则。Flutter 继续只读展示已购订单和背包，不在 F1 新增使用 / 启用动作。

## 九、Console 与审计

### 9.1 商品管理

- 商品表单从后端能力元数据读取可售状态和配置说明。
- 新开放类型不会自动把历史种子商品重新上架；必须由管理员检查价格、库存、有效期和资源后显式上架。
- 已产生订单的商品继续禁止物理删除；下架不影响历史履约。

### 9.2 订单排障

订单详情增加：

- 支付证据、失败阶段、履约资源类型 / ID。
- 来源快照与当前商品配置差异提示。
- “重试发放”资格说明；未支付失败订单不显示或拒绝该动作。
- 重试、撤销和人工补偿的操作流水。

### 9.3 用户权益排障

Console 可按用户查看当前权益、当前选择、过期 / 撤销状态、来源订单和消耗品数量；高风险动作要求确认、权限和原因，不提供直接编辑数量或状态字段的万能表单。

## 十、迁移与兼容

F1 涉及数据库演进，必须通过 `Radish.DbMigrate` ledger 迁移，不使用运行时 Code First 静默补列：

1. 为订单补失败阶段、固定到期快照和明确履约资源字段。
2. 根据 `PaidTime / CoinTransactionId / ProductType / UserBenefitId` 迁移历史失败阶段与履约资源。
3. 新建 `UserActiveBenefit`，从合法且未过期的历史 `IsActive` 数据迁移；同类型冲突时只保留最近激活记录并输出 doctor 报告。
4. 为 `UserBenefit` 增加撤销信息。
5. 新建 `ShopEntitlementOperation` 及幂等唯一约束。
6. `doctor` 报告无法判定的失败订单、无支付证据的已完成订单、同类型多激活权益和无来源发放；`apply` 不猜测修复高风险异常。
7. `verify` 检查唯一键、历史数据映射和不可用商品仍处于下架状态。

## 十二、验证口径

### 12.1 后端与数据库

- 未支付 `Failed(Payment)` 订单不能重试发放。
- 已支付 `Failed(Fulfillment)` 订单可重试，重复重试只产生一份权益或一次消耗品增量。
- 商品修改 / 下架 / 软删除后，历史订单仍按快照发放。
- 固定日期和天数有效期都以订单快照和 UTC 计算。
- 同一消耗品使用幂等键并发请求只扣减和生效一次；不同请求复用同一键返回冲突。
- 经验或萝卜币效果失败时，背包数量和操作流水回滚。
- 同类型权益并发启用后只有一个 `UserActiveBenefit`。
- 到期边界、撤销竞争、重复过期任务和跨租户访问均有测试。
- SQLite migration 重入、verify 与隔离 PostgreSQL 并发用例通过。

### 12.2 前端

- Client 的 type-check、lint、test、production build 通过。
- Console 的 type-check、lint、test、production build 通过。
- 背包使用、启用、过期、撤销、失败重试提示具有精确组件测试。
- PC 与移动视图重点检查背包卡片、确认弹层、作者身份装饰和 Console 订单详情。

### 12.3 运行态

真实 Gateway smoke 只在 F1 完整专题准备验收时执行。执行前必须在当前任务中确认 API、Auth、Gateway、Client 和 Console 已启动，并按 [浏览器 smoke 指南](/guide/browser-smoke) 同时覆盖 PC 与移动视图。
