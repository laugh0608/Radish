## 目标

在现有论坛帖子主链路上增加一套**最小可用的抽奖能力**，满足以下目标：

- 用户可在发帖时附带一个抽奖活动。
- 其他登录用户可通过发布父评论参与抽奖。
- 发帖者可在发帖满 1 小时后至截止时间前手动提前开奖。
- 若发帖者未提前开奖，系统会在截止时间到达后自动开奖。
- 帖子列表可识别“抽奖帖”，但不引入独立活动页。
- 尽量复用现有发帖、详情、评论、身份与多租户链路，不新开独立应用。

本轮目标是“**小而快的活动闭环**”，不是一次性做成完整运营系统。

---

## 为什么现在做抽奖

在投票 MVP、问答 MVP 与 `P4-ext` 首轮都已完成收口后，当前最适合继续补一个**复用论坛主链、感知明确、体量可控**的新能力。

相比继续深挖投票或问答，抽奖作为 M12 内新增的主要功能更合适，原因是：

- **仍然复用论坛主链**：发帖、列表、详情、评论与权限链路都能继续复用。
- **产品感知清晰**：用户容易理解“发帖 -> 评论参与 -> 提前开奖 / 自动开奖”的闭环。
- **边界可控**：MVP 阶段不碰奖励发放、抽奖券、防刷审计等重治理能力。
- **和评论链路天然结合**：通过父评论参与，可以避免再开一套独立报名机制。

因此，本方案默认采用“**论坛帖子附带抽奖 + 父评论参与 + 发帖满 1 小时后可提前开奖 + 截止时间自动开奖**”的路径。

## MVP 范围

### 本轮包含

- 发帖时附带抽奖
- 抽奖标题与奖品说明
- 开奖时间
- 中奖人数
- 通过发布父评论参与
- 按用户去重统计有效参与人数
- 发帖者在发帖满 1 小时后可手动提前开奖
- 截止时间到达后系统自动开奖
- 帖子详情页展示中奖名单
- 帖子列表页显示“抽奖”标识与参与人数摘要

### 本轮不包含

- 独立抽奖应用或活动频道
- 报名按钮 / 抽奖券 / 多倍参与次数
- 防刷风控 / 黑名单 / 审计公示页
- 奖品库存、实物发放、商城联动
- 萝卜币 / 经验值奖励联动
- Console 抽奖管理页

## 产品形态

### 位置选择

抽奖不做独立应用，直接挂在论坛帖子体系内：

- **发布入口**：论坛发帖弹窗新增“附带抽奖”开关。
- **列表页**：帖子卡片显示“抽奖”徽标与参与人数摘要。
- **详情页**：正文下方、讨论区上方插入抽奖卡片。

### 用户流

#### 发帖者

1. 打开发帖弹窗
2. 输入标题、正文、分类、标签
3. 开启“附带抽奖”
4. 填写抽奖标题、奖品说明、开奖时间、中奖人数
5. 发布帖子
6. 发帖满 1 小时后可在帖子详情提前开奖；若未操作则等待系统到点自动开奖

#### 参与者

1. 在帖子列表识别抽奖帖
2. 打开帖子详情
3. 查看抽奖规则与开奖时间
4. 在讨论区发布一条父评论
5. 等待发帖者开奖并查看中奖结果

## 业务规则

### 发帖规则

- 抽奖标题必填。
- 奖品说明必填。
- 开奖时间必填，且必须至少晚于发帖时间 `1 小时`。
- 中奖人数必填，建议限制在 `1 ~ 20`。
- MVP 阶段继续保持“抽奖帖”和“投票帖 / 问答帖”互斥，不支持同一帖子叠加多种活动能力。

### 参与规则

- 仅登录用户可通过评论参与。
- 只有**父评论**计入参与资格；子评论 / 回复不计入。
- 同一用户在同一帖子下发布多条父评论，只保留一个参与资格，不叠加权重。
- 发帖者本人不计入中奖候选池。
- 开奖前被删除或软删除的参与评论，不计入有效参与资格。
- 开奖时间之后新发的父评论，不计入本次抽奖。

### 开奖规则

- 仅发帖者本人可手动提前开奖。
- 手动提前开奖窗口为：**发帖满 1 小时后** 到 **截止时间之前**。
- 到达截止时间后由系统自动开奖，不再要求作者手动操作。
- 已开奖后不可重复开奖。
- 候选池来源于有效参与用户集合，按用户去重后随机抽取。
- 若有效参与人数少于配置的中奖人数，则按“**全部有效参与者中奖**”处理，并固化实际中奖名单。
- 自动开奖时若没有有效参与者，也应将本次抽奖收口为“已开奖但无中奖名单”，避免任务无限重试。

### 编辑与删除规则

- 带抽奖的帖子在 MVP 阶段**允许编辑标题 / 正文 / 分类 / 标签**。
- **不允许编辑抽奖标题、奖品说明、开奖时间与中奖人数**，避免开奖语义漂移。
- 删除帖子时，抽奖随帖子一并失效。
- MVP 不提供独立“删除抽奖”入口。

## 后端设计

### 数据模型

建议新增 2 个实体。

#### 1. `PostLottery`

表示一个帖子附带的抽奖主体。

建议字段：

- `PostId`
- `Title`
- `PrizeDescription`
- `DrawTime`
- `WinnerCount`
- `ActualWinnerCount`
- `IsDrawn`
- `DrawnAt`
- `DrawnByUserId`
- `DrawnByUserName`
- `TenantId`
- `IsDeleted`
- 审计字段

#### 2. `PostLotteryWinner`

表示抽奖开奖结果中的中奖用户。

建议字段：

- `LotteryId`
- `PostId`
- `UserId`
- `UserName`
- `CommentId`
- `SortOrder`
- `TenantId`
- `CreateTime`

### 为什么 MVP 不单独建参与记录表

当前已确认参与方式为“发布父评论参与”，因此第一版直接复用现有 `Comment` 体系作为参与凭证：

- 是否参与：看是否存在符合条件的父评论
- 参与人数：按评论作者去重统计
- 开奖候选池：从符合条件的父评论作者集合中抽取

这样可以避免再引入一套新的报名数据链路，先把论坛主链闭环跑通。

### 与帖子 / 评论关系

- 一个帖子最多挂一个抽奖。
- 抽奖归属于帖子，不对外暴露为独立内容类型。
- 参与资格依附于评论区父评论，不额外创建报名动作。
- 开奖时需同时读取帖子、抽奖和评论数据。

## 接口设计

### 控制器划分

- 帖子主体继续由 `PostController` 承担。
- 抽奖动作建议新增 `LotteryController`，避免把帖子控制器继续堆大。
- 评论参与继续复用现有 `CommentController` 发布评论接口，不新增独立“参与抽奖”接口。

### 最小接口集

#### 1. 发布帖子（附带抽奖）

`POST /api/v1/Post/Publish`

- 继续复用现有发帖接口
- 请求中可选携带 `Lottery`

#### 2. 获取帖子详情（包含抽奖）

`GET /api/v1/Post/GetById/{id}`

- 返回现有 `PostDetail`
- 若帖子带抽奖，则附带 `VoLottery`

#### 3. 按帖子获取抽奖详情

`GET /api/v1/Lottery/GetByPostId?postId=`

用途：

- 前端局部刷新抽奖结果
- 评论后刷新参与人数
- 开奖后原位刷新中奖名单

#### 4. 手动开奖

`POST /api/v1/Lottery/Draw`

请求：

- `postId`

返回：

- 最新 `PostLotteryVo`

### 自动开奖任务

- 使用 Hangfire 按分钟扫描已到截止时间且尚未开奖的抽奖帖子。
- 自动开奖的候选池截止时间固定取 `DrawTime`，不取 Job 实际执行时刻，避免把截止后评论错误混入中奖池。
- 手动提前开奖的候选池截止时间固定取“实际手动开奖时刻”。

### 错误语义

建议统一给出明确失败消息：

- 当前帖子未配置抽奖
- 抽奖尚未到开奖时间
- 只有发帖者可以开奖
- 抽奖已完成开奖
- 当前没有有效参与者
- 当前状态不允许开奖

## Service / Repository 设计

### Service

建议新增：

- `IPostLotteryService`
- `PostLotteryService`

建议职责：

- 创建帖子抽奖
- 获取帖子抽奖详情
- 统计有效参与人数
- 从父评论作者集合中随机抽取中奖者
- 固化中奖名单与开奖状态

### Repository

优先策略：

- 抽奖主体与中奖记录先通过 `IBaseRepository<TEntity>` 完成
- 参与统计与开奖候选池查询通过评论相关仓储能力或必要扩展完成
- 仅当抽奖聚合查询复杂时，再拆专属仓储

### 事务边界

开奖必须放在同一事务内完成以下操作：

1. 校验帖子、抽奖与当前用户身份
2. 校验尚未开奖且已到开奖时间
3. 查询有效参与用户集合
4. 随机抽取中奖用户
5. 写入 `PostLotteryWinner`
6. 更新 `PostLottery.IsDrawn / DrawnAt / ActualWinnerCount`

## 接口返回建议

### 帖子列表

为了避免 N+1，帖子列表不返回完整抽奖明细，只返回轻量摘要：

- `VoHasLottery`
- `VoLotteryParticipantCount`
- `VoLotteryIsDrawn`

### 帖子详情

详情页返回完整 `VoLottery`：

- 抽奖标题
- 奖品说明
- 开奖时间
- 中奖人数
- 实际中奖人数
- 参与人数
- 开奖状态
- 中奖名单

### 为什么这样分层

- 列表页保持轻量，避免每条帖子都附带完整中奖名单。
- 详情页承担完整交互，符合当前论坛架构。

## 权限与安全

### 权限边界

- 抽奖帖发布权限继续复用现有发帖权限。
- 评论参与权限继续复用现有评论发布权限，不额外引入新的权限族。
- 开奖权限仅由发帖者本人持有。

### 风险点

#### 1. 同一用户多次评论刷权重

必须按用户维度去重，不能按评论条数累计权重。

#### 2. 回复评论误计入参与资格

必须明确只统计父评论，不能把楼中楼回复计入候选池。

#### 3. 开奖前后状态漂移

开奖时要固化中奖名单，不能每次详情查询都重新随机。

#### 4. 租户隔离

抽奖实体、中奖记录与候选评论都必须带租户口径，跟随帖子租户边界。

## 验收标准

满足以下条件即可认为抽奖 MVP 可进入验收：

- 可发布带抽奖的帖子
- 帖子列表可识别抽奖帖
- 帖子详情可正确展示抽奖信息
- 登录用户发布父评论后可被计入参与人数
- 同一用户多条父评论不会重复增加参与资格
- 发帖者可在发帖满 1 小时后至截止时间前成功提前开奖
- 到达截止时间后系统可自动开奖
- 已开奖结果可稳定回显
- 前后端字段口径一致，无匿名对象返回

### 最小人工验收顺序（2026-03-16 补齐）

1. 先使用 `Radish.Api.AuthFlow.http` 获取 `access_token`，必要时准备第二个账号的 token 用于验证“非发帖者不能开奖”。
2. 执行 `Radish.Api.Forum.Lottery.http`，先发布带抽奖帖子，再通过列表与详情确认抽奖摘要 / 明细已回传。
3. 发布一条父评论后再次查询 `Lottery/GetByPostId`，确认参与人数增加；再发布一条回复评论，确认参与人数不继续增加。
4. 在发帖 1 小时内尝试一次 `Lottery/Draw`，确认返回“不允许提前开奖”的失败语义；发帖满 1 小时后、截止时间前由发帖者执行一次提前开奖并再次刷新详情。
5. 另准备一条未提前开奖的抽奖帖，等待截止时间后观察系统自动开奖，再刷新详情确认中奖名单与参与人数。
6. 执行 `Radish.Api.User.Profile.http`，依次访问帖子、商品与 Wiki 详情，再查询 `User/GetMyBrowseHistory`，确认三类对象均已进入浏览记录。
7. 打开欢迎 App 的论坛详情、个人主页浏览记录页签与深链跳转，确认浏览记录点击后可正常打开帖子 / 商品 / Wiki 对应窗口。

- 发帖附带抽奖、父评论参与、发帖满 1 小时后提前开奖、截止后自动开奖、中奖名单展示主链路可完成。
- 回复评论不会误计入参与资格，同一用户重复父评论不会重复计数。

## 结论

本方案选择“**论坛帖子附带抽奖 + 父评论参与 + 发帖满 1 小时后可提前开奖 + 截止时间自动开奖**”作为第一版实现路径，核心原因是：

- 与现有论坛主链最贴合
- 复用现有发帖、详情与评论体系
- 体量可控
- 能形成明确的用户闭环

截至 **2026-04-10**，`P5` 已达到“可演示、可联调、可回归、可转维护”的本轮收口标准；如后续继续推进，应转入 `P5-ext`，而不是继续占用当前主线。
