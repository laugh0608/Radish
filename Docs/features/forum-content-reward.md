# 论坛内容赞赏（胡萝卜 +1）设计方案

> **状态**：专题方案已建立，尚未实现
>
> **更新日期**：2026-07-13
>
> **适用主线**：正式 Web 论坛
>
> **阶段关系**：作为后续完整功能专题储备，不改变当前 `F1 商城商品效力与权益履约` 第一顺位

## 摘要

“胡萝卜 +1”是基于现有萝卜币账本的**内容赞赏**：用户从自己的可用余额中送出 `1 胡萝卜`，内容作者收到 `1 胡萝卜`，并可附带一条简短理由。

它不是新积分、风评值或声望系统，也不是点赞、系统奖励或通用转账的别名。首版固定金额、免手续费、每人对同一内容最多赞赏一次，并由专用资产写接口、数据库事务、幂等记录和唯一约束共同保护。

产品文案统一使用：

- 入口按钮：`送胡萝卜`
- 成功反馈：`已送出 1 胡萝卜`
- 公开记录：`胡萝卜 +1`
- 产品与领域名称：`内容赞赏`
- 代码领域名：`ContentReward`

## 1. 产品定位

### 1.1 要解决的问题

论坛已有点赞、轻回应和评论，但仍缺少一种介于“轻量表达”和“正式回复”之间、能让作者感受到真实认可的互动：

- 读者可以明确表达“这条内容值得我付出一点已有资产”；
- 作者可以看到是谁、因为什么给予认可；
- 社区把萝卜币从静态余额延伸到真实内容互动；
- 资产流转保持可审计，不凭空增加胡萝卜总量。

### 1.2 与相邻能力的区别

| 能力 | 是否涉及资产 | 主要语义 | 是否附理由 |
|---|---:|---|---:|
| 点赞 / 轻回应 | 否 | 低成本表达态度 | 否 |
| 内容赞赏 | 是，固定 `1 胡萝卜` | 对具体内容的公开认可 | 是 |
| 系统奖励 | 是，系统增发 | 任务、活动或运营奖励 | 由系统决定 |
| 用户转账 | 是，金额可变 | 用户之间的通用资产流转 | 私域备注 |
| 精华 / 采纳 | 否 | 管理或问答状态认定 | 否 |

内容赞赏不得替代点赞、问答采纳或管理员精选，也不得借“赞赏”入口开放任意金额转账。

## 2. 首版范围

### 2.1 支持对象

首版仅支持正式 Web 论坛中的：

- `Post`：帖子正文；
- `Comment`：帖子评论；
- `PostAnswer`：问答回答。

目标必须属于当前租户、真实存在、未删除，并且当前访问者具备读取权限。接收者由服务端依据目标内容解析，客户端不得传入作者 ID。

### 2.2 暂不支持

- `PostQuickReply` 轻回应；
- 聊天消息、Docs、商品评价等非论坛对象；
- 匿名赞赏；
- 自定义金额、连续多次赞赏和批量赞赏；
- 赞赏排行榜、等级、徽章或新的声望体系；
- Flutter 新入口、Tauri 专用能力或 WebOS 专用交互；
- 法币充值、提现、兑换或任何现金价值承诺。

WebOS 历史入口若复用正式 Web 论坛组件，可自然获得同一能力，但不单独维护第二套业务实现。

## 3. 核心业务规则

### 3.1 资产规则

- 每次固定扣除发送者 `1 胡萝卜`，接收者增加 `1 胡萝卜`；
- 手续费固定为 `0`，不改变系统胡萝卜总量；
- 金额由服务端固定，创建请求不接收 `amount`；
- 接收者由目标内容解析，创建请求不接收 `recipientUserId`；
- 不使用系统奖励和增发链路；
- 发送者可用余额不足时直接失败，不允许负余额或先记后补。

现有通用用户转账继续保持独立接口及支付密码保护。内容赞赏是受“固定金额、固定业务对象、单次唯一、频率限制和公开记录”约束的低风险专用动作，不得把免支付密码扩展到任意金额转账。

### 3.2 资格与次数

- 必须登录后才能送出胡萝卜；公开内容的赞赏摘要可以匿名读取；
- 不允许给自己的内容送胡萝卜；
- 同一发送者对同一目标内容终身最多成功一次；
- 发送者每日成功赞赏上限默认 `20` 次；
- 同一发送者每日向同一接收者成功赞赏上限默认 `5` 次；
- 发送者或接收者的用户状态不允许资产写时拒绝交易；
- 次数上限属于可治理业务配置，但首版不开放普通 Console 用户随意修改。

“每日”使用系统统一业务日和 `TimeProvider` 口径，不以浏览器本地时区自行计算。

### 3.3 理由

首版提供预设理由，并允许附加一条简短自定义理由：

| `reasonCode` | 展示文案 |
|---|---|
| `Helpful` | 很有帮助 |
| `Insightful` | 很有启发 |
| `WellWritten` | 写得真好 |
| `Detailed` | 内容详实 |
| `Warm` | 温暖了我 |

规则如下：

- 必须选择一个预设理由；
- 自定义理由可选，去除首尾空白后长度为 `2-40` 个字符；
- 自定义理由按用户生成内容处理，复用敏感内容过滤、举报和治理边界；
- 理由违规时可以隐藏理由文本，但不得修改或删除已经成功的资产流水；
- 首版不支持匿名显示或隐藏赠送者身份。

## 4. 交互与视觉方案

### 4.1 详情页位置

内容赞赏位于帖子正文或对应评论 / 回答的主要操作区之后。帖子详情的整体层级为：

```text
正文与主要操作
  -> 内容赞赏摘要
  -> 轻回应墙
  -> 评论区
```

内容赞赏是有成本的明确动作，不与无成本点赞挤在同一个即时切换控件中。

### 4.2 默认展示

- 未赞赏时显示 `送胡萝卜` 和累计人数；
- 已赞赏时显示稳定完成态 `已送胡萝卜`，不可再次点击扣款；
- 有记录时默认展示最近 `3` 条，可展开分页查看；
- 固定金额下，赞赏人数与胡萝卜数量相等，但接口仍分别保留明确语义，避免未来展示口径混淆。

PC 可采用“用户 / 操作 / 理由 / 时间”的紧凑行式布局；移动端必须改为纵向卡片，不压缩成四列小表格。

### 4.3 确认与反馈

点击 `送胡萝卜` 后打开确认浮层或对话框，明确展示：

- `本次将花费 1 胡萝卜`；
- 当前可用余额；
- 预设理由与可选补充文字；
- `确认送出` 主按钮。

提交期间禁用重复操作，不做资产金额的乐观更新。服务端成功后再更新余额、累计数和公开记录。

成功动效可以使用一次约 `300ms` 的胡萝卜上浮与 `+1` 反馈；系统开启 `prefers-reduced-motion` 时只保留颜色与文字反馈。动效不得改变布局高度或遮挡正文。

未登录时进入登录流程并保留当前内容返回地址；余额不足时给出资产说明，并可跳转 `/me/assets`，不把购买或充值作为强制下一步。

## 5. 数据模型

### 5.1 `ContentReward`

建议新增独立业务实体，保存公开展示和目标关联，不把用户理由塞进财务流水备注后反向拼装页面。

| 字段 | 说明 |
|---|---|
| `Id` | 内部主键，雪花 ID |
| `TenantId` | 租户 ID |
| `TargetType` | `Post / Comment / PostAnswer` |
| `TargetId` | 目标内容内部 ID |
| `SenderUserId` | 赠送者用户 ID |
| `RecipientUserId` | 内容作者用户 ID 快照 |
| `Amount` | 固定为 `1` |
| `ReasonCode` | 预设理由编码 |
| `ReasonText` | 可选自定义理由 |
| `CoinTransactionId` | 对应资产流水 ID |
| `Status` | `Succeeded / ReasonHidden` |
| `CreateTime` | 成功时间，UTC |

必要约束与索引：

- 唯一约束：`TenantId + SenderUserId + TargetType + TargetId`；
- 目标列表索引：`TenantId + TargetType + TargetId + CreateTime`；
- 发送者审计索引：`TenantId + SenderUserId + CreateTime`；
- 接收者审计索引：`TenantId + RecipientUserId + CreateTime`；
- `Amount` 必须为 `1`；
- 资产事实不物理删除，理由治理只改变可见状态。

首版不在 `Post / Comment / PostAnswer` 上新增冗余计数。列表或详情通过批量聚合查询获得数量；出现明确性能证据后，再引入缓存或可重建聚合表。

### 5.2 萝卜币流水

成功赞赏应复用现有资产账本：

- `CoinTransaction.TransactionType = TIP`；
- `FromUserId = SenderUserId`；
- `ToUserId = RecipientUserId`；
- `Amount = 1`；
- `Fee = 0`；
- `BusinessType = ContentReward`；
- `BusinessId = ContentReward.Id`；
- 同时写入发送者扣减和接收者增加两条 `BalanceChangeLog`。

`RewardBusinessKey` 只用于系统发放 / 奖励业务，内容赞赏属于用户出资转移，不得占用该字段充当幂等键。

数据库结构必须通过 `Radish.DbMigrate` 的有序 schema ledger 迁移，不允许恢复 Code First 或运行时静默补表。

## 6. 事务、幂等与并发

### 6.1 原子事务

建议新增职责明确的 `IContentRewardService`。以下写入必须在 Main 数据库同一事务中成功或回滚：

1. 校验目标、作者、余额、资格和每日限制；
2. 创建 `ContentReward`；
3. 扣减发送者余额；
4. 增加接收者余额；
5. 创建 `TIP` 类型 `CoinTransaction`；
6. 写入双方 `BalanceChangeLog`；
7. 写入可靠通知 Outbox。

余额写继续复用现有乐观锁和冲突重试机制。任何一步失败都不得留下单边余额、孤立赞赏记录或无业务关联的流水。

### 6.2 操作幂等

内容赞赏是资产写，必须使用 `OperationIdempotencyRecord`，不得复用论坛内容发布的 `ContentSubmissionRecord`。

- 新增操作类型：`ContentReward`；
- 客户端键格式：`content-reward:{uuid}`；
- `ResourceType = ContentReward`；
- 请求摘要包含：`targetType`、`targetId`、`reasonCode`、规范化后的 `reasonText`；
- 同键同摘要重试返回同一最终结果；
- 同键不同摘要返回冲突；
- 处理中请求不得再次进入余额写；
- 终态业务失败按现有资产幂等治理保存和重放。

即使客户端使用不同幂等键并发请求，同一发送者 / 目标唯一约束也必须阻止第二次成功。唯一冲突应映射为稳定的“已经送过胡萝卜”，不能暴露数据库异常。

## 7. API 契约

### 7.1 创建赞赏

```http
POST /api/v1/ContentReward/Create
```

请求：

```json
{
  "targetType": "Post",
  "targetId": "123456789012345678",
  "reasonCode": "Helpful",
  "reasonText": "把来龙去脉讲清楚了",
  "idempotencyKey": "content-reward:7d8705ad-6cd4-48a0-a51d-a09e30047b91"
}
```

禁止接收 `senderUserId`、`recipientUserId`、`amount` 和 `paymentPassword`。

成功响应至少包含：

- 赞赏公开 ID 或字符串 ID；
- `targetType / targetId`；
- `totalCount`；
- `viewerRewarded = true`；
- 发送者最新可用余额；
- 供发送者资产自查的交易流水号。

所有 Long ID 在 JSON 中按字符串传输。

### 7.2 查询目标赞赏

```http
GET /api/v1/ContentReward/GetTargetRewards
    ?targetType=Post
    &targetId=123456789012345678
    &pageIndex=1
    &pageSize=20
```

公开目标允许匿名读取，响应包含累计数量、当前登录用户是否已赞赏，以及分页记录。公开记录只返回赠送者公开资料、展示理由和时间，不返回内部用户 ID、余额、交易流水号或幂等键。

### 7.3 稳定错误语义

至少区分：

- 目标不存在或不可访问；
- 不允许赞赏自己的内容；
- 已经赞赏过；
- 可用余额不足；
- 达到每日限制；
- 作者或用户状态不可用；
- 请求处理中；
- 幂等键摘要冲突。

接口继续使用项目统一 `MessageModel`、HTTP 状态、错误码和 `TraceId` 契约。

## 8. 通知、回流与资产自查

- 成功后向内容作者发送 `ContentRewardReceived` 通知；
- 通知正文可展示赠送者和理由，导航载荷指向目标帖子的 canonical forum 路由；
- 评论 / 回答赞赏必须携带对应定位信息，打开后滚动并高亮目标；
- 通知通过事务内 Reliable Outbox 记录、事务后异步投递；即时推送失败不回滚资产交易；
- 发送者和接收者都可在现有资产流水中看到 `TIP / ContentReward`；
- Console 资产流水可按 `TIP` 和 `ContentReward` 筛选，并回看目标内容。

首版不新增独立“赞赏中心”页面，先复用通知中心、内容详情和资产流水三条可追查路径。

## 9. 治理、安全与隐私

- 发送身份从认证上下文取得，目标作者从数据库解析；
- 复用资产写限流，并以每日成功次数和同目标唯一约束控制滥用；
- 自定义理由可举报、隐藏并保留治理证据；
- `ContentReport` 增加 `ContentReward` 目标类型，Console 现有 `Moderation` 页面承接理由回看、隐藏与审计，不另建平行治理系统；
- 内容被删除或下线后，公开页面不再展示赞赏列表，但资产流水和审计关联永久保留；
- 内容删除、理由隐藏或普通用户反悔均不自动退款；
- 只有重复扣款、系统错误或经确认的欺诈场景可进入受权补偿流程；补偿必须新增反向流水，不得修改或删除原成功流水；
- 日志不得记录认证凭据、完整幂等记录或不必要的完整理由正文；
- 观测指标至少覆盖成功量、失败原因、发送者 / 接收者集中度、唯一冲突和余额冲突。

## 10. 实施顺序

1. 定稿实体、接口、错误码和迁移契约；
2. 新增 `ContentReward` schema ledger migration；
3. 实现 Repository / Service、原子账本写和操作幂等；
4. 接入通知 Outbox、`ContentReward` 举报目标、资产流水筛选和治理回看；
5. 在 `@radish/http` 增加统一类型与 API 客户端；
6. 先接入正式 Web 帖子，再接入评论与问答回答；
7. 完成 PC / mobile 交互、无障碍与 reduced-motion；
8. 按资产风险执行定向测试、前端静态验证和成组浏览器验收。

进入代码实现前，应再次确认当前规划顺位；本专题本身不抢占已确定的 `F1 商城商品效力与权益履约`。

## 11. 验证基线

### 11.1 后端

- 正常赞赏后发送者 `-1`、接收者 `+1`、总量不变；
- 余额不足、自赞赏、目标不可见和作者不可用均不产生任何写入；
- 同一幂等键重试只产生一次业务记录和一次资产交易；
- 同键不同摘要稳定冲突；
- 不同幂等键并发赞赏同一目标仅一次成功；
- 并发赞赏不同目标不会导致超扣或负余额；
- `ContentReward`、`CoinTransaction` 和两条余额日志始终原子一致；
- 赞赏不写 `RewardBusinessKey`，流水类型固定为 `TIP`；
- 每日次数按统一业务日边界计算；
- Long ID 序列化为字符串；
- 通知 Outbox 重试不重复创建持久通知。

### 11.2 前端

- 未登录回流、余额不足、提交中、成功和重复赞赏状态清晰；
- 资产数量不做乐观更新；
- PC 行式布局与移动卡片布局均不溢出；
- 键盘操作、焦点返回、错误提示和 reduced-motion 可用；
- 帖子、评论、回答的累计数和当前用户状态一致；
- 从通知与 Console 回看能够稳定定位目标内容。

### 11.3 验收节奏

开发中优先执行定向单元 / 集成测试、type-check、build、changed-only lint、仓库卫生和 `git diff --check`。待帖子、评论、回答及通知形成完整专题批次后，再按 [浏览器 Smoke 规则](/guide/browser-smoke) 同时覆盖 PC 与移动视图；执行真实 smoke 前必须重新确认本任务服务启动授权。

## 12. 停止线

- 不创建“胡萝卜之外的萝卜币”或独立风评值；
- 不允许客户端决定金额、接收者或资产流水类型；
- 不把通用转账接口包装成无支付密码的快捷转账；
- 不引入手续费、匿名赞赏、任意金额或自动退款；
- 不在首版扩展排行榜、声望、勋章、任务奖励和复杂经济循环；
- 不为 Flutter、Tauri 或 WebOS 另建一套服务契约；
- 不因单独专题文档频繁创建 `dev -> master` PR，待形成完整功能或成组维护批次后统一集成。
