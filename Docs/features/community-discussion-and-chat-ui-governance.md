# 社区讨论与聊天室侧栏 UI 治理方案

> 适用范围：`radish.client` 桌面 / WebOS 论坛评论区、聊天室右侧成员栏
>
> 最后更新：`2026-04-10`
>
> 关联文档：
> [论坛应用功能说明](./forum-features.md) ·
> [聊天室 App UI 模块设计](./chat-app-ui-modules.md)

---

## 目标

本方案解决两个已经暴露出来的 UI / 交互问题：

1. **聊天室右侧成员栏收起入口重复**
   - 当前顶部已存在“收起成员”按钮，成员栏标题行内又存在一个箭头按钮。
   - 两个入口都在表达“收起右栏”，职责重叠，造成界面噪音。

2. **论坛两层评论中，子评论回复路线不清晰**
   - 现状采用“父评论 + 子评论”两层结构，不继续无限嵌套。
   - 更深层回复被压平成子评论平级项后，仅靠 `@用户名` 区分“回复谁”。
   - 当两个用户多次往返回复时，用户只能看到“谁 @ 谁”，看不出“具体接的是哪一条评论”。

本次治理目标不是恢复无限层级，而是在**保留两层结构**前提下，补齐“回复锚点”表达能力。

---

## 聊天室成员栏

### 设计结论

- 右侧成员栏只保留**一个**收起 / 展开入口。
- 删除频道标题区右上角的独立“收起成员”大按钮。
- 保留成员栏标题行右侧的箭头按钮，并把**整个标题行**扩展为可点击热区。

### 职责定义

- 标题行：负责成员栏整体展开 / 收起。
- 箭头图标：只承担当前状态提示，不再作为第二套重复入口。

### 交互规则

- 展开态显示：`成员 + 数量 + 折叠箭头`
- 收起态显示：成员栏收成 `40-44px` 的窄轨道，仅保留居中的展开箭头
- 收起态不再显示竖排“成员”文字、数量徽标或半截标题骨架
- 收起态边框与背景应弱化，让它看起来像“轨道入口”，而不是没收干净的侧栏
- 宽度变化由成员栏自身承担，不再依赖头部大按钮同步状态
- 移动窄屏继续沿用现有策略，直接隐藏成员栏

### 预期收益

- 降低认知成本，避免“一件事两个按钮”
- 让聊天室头部更干净，把注意力还给频道信息与消息流
- 为后续成员栏扩展状态展示、筛选或小工具预留头部空间

## 论坛评论回复路线

### 设计原则

- 评论树**始终保持两层**：父评论 + 子评论列表
- 子评论允许“回复某条子评论”，但**不新增三级、四级缩进**
- 回复关系要同时表达：
  - 回复的是**谁**
  - 接的是**哪条评论**
- “回复关系”是结构化元信息，不再依赖正文开头的 `@用户名`

### 数据模型

在现有两层评论模型上补充“回复锚点”字段：

- `ReplyToCommentId`
  - 指向当前评论真正要回应的那条评论
  - 顶级评论为空
  - 子评论直接回复父评论时，指向父评论本身
  - 子评论回复另一条子评论时，指向那条子评论，而不是仅记录根评论

- `ReplyToCommentSnapshot`
  - 保存被回复评论的轻量内容摘要
  - 作为稳定的阅读锚点
  - 即使被回复评论随后编辑、删除或暂未加载，当前回复仍然保留最小上下文

现有字段继续保留：

- `ParentId`
  - 仍然只用于控制两层结构归属
  - 子评论统一挂在根评论下

- `ReplyToUserId / ReplyToUserName`
  - 作为展示与兼容字段存在
  - 真正的回复路线以 `ReplyToCommentId` 为准

### 展示结构

每条子评论统一拆成四段：

1. `作者`
2. `回复 XX`
3. `↪ 被回复内容摘要`
4. `当前评论正文`

示例：

```text
张三 回复 李四
↪ 我觉得成员栏应该只保留一个收起入口
这样更清晰，也不会让操作职责重复。
```

### 摘要规则

- 优先取被回复评论正文前 `18-30` 个可读字符
- 超长截断为 `...`
- 压缩空白和换行
- 图片 / 附件 / 贴图优先转为轻量占位：
  - `[图片]`
  - `[附件]`
  - `[表情]`

### 交互规则

- 点击 `↪ 摘要`：
  - 若目标评论已在当前列表中，直接滚动并高亮
  - 若目标评论尚未加载，先通过评论定位接口加载对应根评论页和子评论页，再滚动并高亮

- 回复输入框顶部：
  - 不再只显示“回复 @某人”
  - 改为显示“回复对象 + 被回复摘要”
  - 用户提交前即可确认自己正挂在哪条评论下

### 服务端行为调整

为了保证业务语义和通知语义一致，子评论发布时需要区分：

- **结构父节点**：`ParentId`
  - 决定评论在两层结构中的归属

- **真实回复目标**：`ReplyToCommentId`
  - 决定通知、奖励与阅读锚点的实际指向

因此：

- 回复父评论：通知 / 奖励指向父评论作者
- 回复某条子评论：通知 / 奖励指向那条子评论作者
- 不再因为“所有子评论都挂在根评论下”，就把所有回复都误判为回复根评论作者

---

## 兼容策略

- 老评论如果没有 `ReplyToCommentId / ReplyToCommentSnapshot`
  - 继续按原有 `ReplyToUserName` 显示最小兼容信息
  - 不强制回填历史数据

- 新评论从本方案落地后开始写入回复锚点
  - 后续阅读体验逐步自然收敛

---

## 验收要点

### 聊天室

- 不再出现两个“收起成员”入口
- 成员栏展开 / 收起入口唯一且可发现
- 收起后不破坏消息主区宽度与输入区布局

### 论坛评论

- 仍保持两层评论结构
- 子评论可明确看出“回复谁”与“回复哪条评论”
- 同一对用户多轮往返时，列表中仍能通过摘要区分每一跳
- 点击摘要可回到原评论，原评论会短暂高亮
- 回复某条子评论时，通知与奖励指向真实被回复者，而不是默认指向根评论作者

---

## 不做项

- 不恢复无限评论树
- 不引入多级缩进伪层级
- 不把回复关系继续塞回正文开头的 `@用户名`
- 不通过重新排序评论列表来模拟回复树
