## P0 任务清单（当前优先）

后端依赖：
- `Channel` / `ChannelMessage` / `ChannelMember` 最小接口可用。
- `ChatHub` 事件：`MessageReceived`、`MessageRecalled`、`ChannelUnreadChanged`。

前端任务：
- 建立 `apps/chat` 目录与 `ChatApp` 入口。
- 实现 `chatStore`（频道、消息、未读）。
- 实现 `chatHub.ts`（连接、重连、事件分发）。
- 实现频道切换 + 历史分页 + 文本发送 + 撤回。

验收标准：
- 两账号同时在线，发送消息双方可见。
- 任一端撤回消息，双方 2 秒内同步。
- 频道切换后未读数可正确清零。
- 已通过编译与静态契约核对（2026-03-03）：后端 `Radish.Api` 编译通过，前端 `radish.client` type-check 通过。

## P1 任务清单

- 输入中状态：2 秒节流上报，3 秒超时消失。
- `@mention`：输入检索、插入格式、通知触达。`已完成`
- 引用回复：消息气泡内嵌 `QuotedMessage` 预览，输入区展示 `ReplyPreview`，发送时携带 `replyToId`。`已完成`
- 图片消息：上传、进度、缩略图、灯箱。`已完成`
- 右侧成员列表：在线状态与人数展示。`已完成`
- 断线重连 UI：状态条 + 自动恢复频道订阅（重连后清空当前频道缓存并重新加载最新 50 条）。`已完成`
- 乐观更新升级：发送时先插入 `status: 'sending'` 临时消息，Hub 回推后替换为服务端真实消息，超时标记 `status: 'failed'` 并允许重试。`已完成`
- 草稿持久化：MessageInput 内容按 `channelId` 存 `localStorage`，切换频道时自动保存与恢复，包含当前的 `replyToId` 引用状态。`已完成`

验收标准：
- 输入中与 mention 在多端场景表现一致。
- 引用回复在发送方和接收方均能正确展示引用来源。
- 图片发送成功率和重试体验可接受。
- 重连后 1 次自动恢复即可继续收发消息。
- 乐观消息在发送失败时可重试或撤销。
- 切换频道后返回，原频道的未发送内容与引用状态可自动恢复。

## P2 任务清单

- 私聊分区与会话列表。
- 频道内消息搜索（关键字 + 时间范围）。
- Reaction 集成（复用表情系统）。
- 消息置顶（Pin）：Moderator/Owner 可置顶单条消息，频道顶部展示 `PinnedMessageBar`，Hub `MessagePinned` 实时同步。
- 消息阅读回执（轻量版）：消息气泡底部展示已读在线成员头像（至多 3 个），Hub `MemberReadUpdated` 实时推送，基于 `ChannelMember.LastReadMessageId`。
- 菜单/按钮级权限细化（只读频道、禁言等）。

验收标准：
- 私聊与频道消息链路互不干扰。
- 搜索结果可跳转到消息上下文。
- 置顶消息在多端实时同步，点击可跳转至原始消息位置，无置顶时预览条自动收起。
- 阅读回执头像仅显示在线成员的已读状态，离线成员不占位。
- 权限变更在前后端均可生效并可审计。

## 风险与应对

1. Hub 与 REST 顺序不一致 / 消息重复
- 采用"REST 成功后插入 + Hub 回推"双通道。`addMessage` 按 `messageId` 去重，已存在则忽略。
- P1 乐观更新阶段：临时消息使用负数 ID，REST 成功后以真实 ID 替换。

2. 长列表性能波动
- 默认启用虚拟滚动与动态高度测量。

3. 多端未读不一致
- 以服务端 `LastReadMessageId` 为权威来源，不使用纯前端累计。

4. messageMap 内存增长
- 切换频道时裁剪非活跃频道消息至最新 50 条。
- 全局消息总数上限 2000 条，超出按 LRU 淘汰最久未访问频道。

5. 重连后消息缺失
- 重连成功后清空当前频道缓存，重新加载最新 50 条（简单可靠）。
- 后续若需精细补全可扩展 `afterMessageId` API 参数（P2 酌情处理）。
