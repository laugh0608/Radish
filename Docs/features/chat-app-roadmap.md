## P1 人工验收模板（样板）

> 结构遵循：[人工验收模板](/records/manual-acceptance-template)

### 适用改动范围

- 频道列表、历史分页、发送、撤回、未读同步改动后
- `@mention`、引用回复、图片消息、成员面板、草稿恢复改动后
- ChatHub 重连、补拉、乐观发送、失败重试改动后

### 前置条件

- 已启动 `Radish.Api`、`Radish.Auth`、`Radish.Gateway`
- 已准备两个可登录账号，便于验证多端收发 / 撤回 / 未读同步
- 已通过 `Radish.Api.AuthFlow.http` 获取必要 token，或直接在前端完成登录
- 已能打开 `radish.client` 中的聊天室 App

### 联调资产

- `Radish.Api.Tests/HttpTest/Radish.Api.Chat.http`
- [聊天室 App 文档总览](./chat-app-index.md)
- [专题回归索引](/guide/regression-index)

### 人工验收顺序

1. 两个账号分别打开聊天室 App，进入同一频道，确认频道列表与最近消息可正常加载。
2. 使用 `Radish.Api.Chat.http` 或前端界面发送一条文本消息，确认发送方即时回显、接收方在合理时间内收到消息。
3. 在发送方对刚发送的消息执行引用回复，再切换到接收方确认引用来源展示正确。
4. 发送一条 `@mention` 消息，确认接收方在消息正文中能看到 mention 标记，且不影响正常收发。
5. 选择一张图片，确认其先进入输入区待发送附件，不会立即出现在消息列表；此时可继续输入文字或移除待发图片。
6. 点击发送，确认图片可作为纯图片或图文消息正常发出，缩略图、预览与消息列表展示正常。
7. 撤回一条已发送消息，确认两端都在合理时间内同步为撤回状态。
8. 主动断开一端网络或刷新窗口后重新进入频道，确认状态条、自动重连与最近消息补拉正常。

### 预期结果

- 文本发送、引用回复、撤回、图片草稿发送、未读同步主链路可正常完成。
- 多端消息状态一致，不出现一端成功另一端缺失或重复。
- 重连后可以继续收发，且不会因为补拉导致明显乱序或重复消息。

### 可跳过项

- 仅调整静态样式或文案时，可跳过双账号实时链路，至少保留单端打开与基础渲染确认。
- 未涉及上传或图片渲染时，可跳过图片消息验收步骤。
- 未涉及重连 / 乐观发送逻辑时，可跳过断网恢复与失败重试观察。

## P0 任务清单（当前优先）

后端依赖：
- `Channel` / `ChannelMessage` / `ChannelMember` 最小接口可用。
- `ChatHub` 事件：`MessageReceived`、`MessageRecalled`、`ChannelUnreadChanged`。

前端任务：
- 建立 `apps/chat` 目录与 `ChatApp` 入口。
- 实现 `chatStore`（频道、消息、未读）。
- 实现 `chatHub.ts`（连接、重连、事件分发）。
- 实现频道切换 + 历史分页 + 文本发送 + 撤回。

验收标准：
- 两账号同时在线，发送消息双方可见。
- 任一端撤回消息，双方 2 秒内同步。
- 频道切换后未读数可正确清零。
- 已通过编译与静态契约核对（2026-03-03）：后端 `Radish.Api` 编译通过，前端 `radish.client` type-check 通过。

## P1 任务清单

- 输入中状态：2 秒节流上报，3 秒超时消失。
- `@mention`：输入检索、插入格式、通知触达。`已完成`
- 引用回复：消息气泡内嵌 `QuotedMessage` 预览，输入区展示 `ReplyPreview`，发送时携带 `replyToId`。`已完成`
- 图片消息：上传、进度、缩略图、灯箱，以及“先入待发送附件区、点击发送后再落消息”。`已完成`
- 右侧成员列表：在线状态与人数展示。`已完成`
- 断线重连 UI：状态条 + 自动恢复频道订阅（重连后清空当前频道缓存并重新加载最新 50 条）。`已完成`
- 乐观更新升级：发送时先插入 `status: 'sending'` 临时消息，Hub 回推后替换为服务端真实消息，超时标记 `status: 'failed'` 并允许重试。`已完成`
- 草稿持久化：MessageInput 当前按 `userId + channelId` 在 `radish.chat.drafts.v1` 中保存并恢复，包含 `content + replyTarget + pendingImage`。`已完成`

验收标准：
- 输入中与 mention 在多端场景表现一致。
- 引用回复在发送方和接收方均能正确展示引用来源。
- 图片草稿发送、移除待发图与发送后落消息体验可接受。
- 重连后 1 次自动恢复即可继续收发消息。
- 乐观消息在发送失败时可重试或撤销。
- 切换频道后返回，原频道的未发送文本、引用状态与待发送图片可自动恢复。

- 消息搜索已按 [F4-C 专题设计](./chat-message-search-design.md) 完成 A-D 批并关闭。

- 消息置顶进入 [F4-E 专题](./chat-message-pin-design.md)，采用独立状态表、频道 revision 和最多 20 条共享置顶。

以下能力继续作为独立后续专题，不并入 F4-C：

- 消息置顶（Pin）：按 F4-E 权威专题分 A-D 批推进；管理权限同时覆盖频道角色和 Accepted Direct，不采用旧单条覆盖草案。
- 消息阅读回执（轻量版）：消息气泡底部展示已读在线成员头像（至多 3 个），Hub `MemberReadUpdated` 实时推送，基于 `ChannelMember.LastReadMessageId`。
- 菜单/按钮级权限细化（只读频道、禁言等）。

验收标准：
- 一对一私聊按专题设计的权限、请求、恢复和 PC / mobile 矩阵验收，且不破坏公开频道消息链路。
- 搜索同时覆盖当前会话和全部当前可见会话，结果可复用现有消息窗口协议跳转到上下文；权限、cursor、删除 / 阻断和跨库行为符合 F4-C 专题设计。
- 置顶集合以带 revision 的完整快照多端同步，点击可跳转至原始消息位置，无置顶时预览条自动收起。
- 阅读回执头像仅显示在线成员的已读状态，离线成员不占位。
- 权限变更在前后端均可生效并可审计。

## 风险与应对

1. Hub 与 REST 顺序不一致 / 消息重复
- 采用"REST 成功后插入 + Hub 回推"双通道。`addMessage` 按 `messageId` 去重，已存在则忽略。
- P1 乐观更新阶段：临时消息使用负数 ID，REST 成功后以真实 ID 替换。

2. 长列表性能波动
- 当前使用服务端历史分页和普通 DOM 渲染；F4-C 搜索结果采用独立 cursor 分页，不把全量命中灌入 `messageMap`。
- 若真实长会话回归暴露 DOM 压力，再单独评估虚拟列表，不把尚未落地的方案写成现状。

3. 多端未读不一致
- 以服务端 `LastReadMessageId` 为权威来源，不使用纯前端累计。

4. messageMap 内存增长
- 切换频道时裁剪非活跃频道消息至最新 50 条。
- 全局消息总数上限 2000 条，超出按 LRU 淘汰最久未访问频道。

5. 重连后消息缺失
- 重连成功后清空当前频道缓存，重新加载最新 50 条（简单可靠）。
- 后续若需精细补全可扩展 `afterMessageId` API 参数（P2 酌情处理）。
