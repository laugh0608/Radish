# Radish 电子宠物开发计划

> 状态：`规划中`
>
> 创建日期：2026-06-08（Asia/Shanghai）
>
> 本文档用于承载 Radish 电子宠物的产品定位、阶段边界、经济系统约束和跨端路线。当前仅作为规划文档，不代表已进入代码实现。

## 产品定位

Radish 电子宠物是面向长期复访、轻互动和社区陪伴感的 Web-first 玩法系统。它不应只是一个独立小游戏，也不应成为萝卜币或经验系统的附属 UI，而是把日常访问、内容互动、关系链、轻任务和资产消耗组织成更可持续的用户回访场景。

首批定位：

- 默认优先进入纯 Web 入口，后续再评估 Flutter 原生承接。
- 可以和萝卜币、经验、签到、评论、轻回应、关注、公开主页产生联动。
- 不直接影响核心社区阅读和内容创作路径。
- 数值、奖励和消耗必须先设计清楚，避免制造刷分和通胀风险。

## 核心循环

建议首版围绕低压力日常照顾展开：

1. 用户获得一只 Radish 宠物。
2. 每日访问、喂养、清洁、互动或完成轻任务提升宠物状态。
3. 宠物状态影响外观、心情、成长阶段和少量展示权益。
4. 社区互动可给宠物提供成长材料，但不得强迫用户刷评论或刷点赞。
5. 萝卜币可用于购买装饰、食物或玩具，但不直接购买核心等级优势。

## 首批能力范围

| 能力 | 首版建议 | 说明 |
| --- | --- | --- |
| 宠物档案 | 支持命名、头像 / 形态、成长阶段、心情、饱食度、清洁度、精力 | 数值应可配置，避免写死 |
| 日常照顾 | 喂食、清洁、互动、休息 | 每日次数和冷却必须受控 |
| 成长系统 | 成长值、阶段解锁、外观变化 | 不与用户经验等级强绑定 |
| 轻任务 | 每日访问、阅读、评论、轻回应、关注等低强度任务 | 任务奖励要有每日上限 |
| 装饰系统 | 背景、玩具、饰品 | 可与商城 / 萝卜币后续联动 |
| 展示入口 | 首页小组件、个人主页展示、独立宠物页面 | 首页只展示状态摘要，不抢信息流主功能 |

## 与现有系统关系

| 系统 | 关系 | 边界 |
| --- | --- | --- |
| 萝卜币 | 可用于购买食物、装饰、玩具 | 不直接购买成长等级，不制造付费优势 |
| 经验等级 | 可作为轻任务输入或展示联动 | 不让宠物成长反向放大经验收益 |
| 论坛 / 评论 | 评论、轻回应、问答、投票等可提供低额材料 | 必须有防刷、每日上限和异常检测 |
| 关系链 | 好友 / 关注可以访问宠物展示或轻互动 | 首版不做偷取、攻击、排行榜攀比 |
| 商城 | 后续可上架宠物装饰和消耗品 | 首版不做复杂优惠、交易或赠送 |
| 通知 | 可提醒状态异常或任务完成 | 不制造高频打扰 |

## 数据模型草案

后续实现前应先确认实体边界：

- `PetProfile`：用户宠物主档，包含名称、形态、成长阶段、状态数值和当前装扮。
- `PetStatLog`：状态变化流水，用于审计和异常排查。
- `PetItem`：食物、玩具、装饰等物品定义。
- `UserPetInventory`：用户拥有的宠物物品。
- `PetTaskProgress`：每日任务进度和奖励领取状态。

所有对外展示对象应预留 `PublicId`，不直接把内部 long 主键作为公开链接或分享标识。

## 阶段路线

### Phase A：方案冻结

- 明确宠物定位、数值范围、奖励来源和消耗边界。
- 明确 Web 入口：独立页面、首页小组件、个人主页展示的优先级。
- 明确是否接入萝卜币、经验和轻任务，以及各自上限。
- 明确不做范围：复杂养成战斗、交易市场、排行榜攀比、强付费成长。

### Phase B：Web 首版

- 实现宠物档案、日常照顾、基础成长和状态展示。
- 首页只展示轻量宠物状态，不影响信息流阅读。
- 个人主页可选择展示宠物，但默认不暴露过多私域状态。
- 补后台配置或种子数据，避免关键数值散落在前端。

### Phase C：经济与治理

- 接入萝卜币消耗和宠物物品。
- 补状态变化流水、异常检测、每日上限和后台查询。
- 评估是否把部分任务与评论、轻回应、关注等社区行为联动。

### Phase D：跨端承接

- Flutter 只在 Web 首版稳定后承接日常照顾、状态提醒和轻量展示。
- 不把 Flutter 作为首发实现端。
- PC/Tauri 如有安装包价值，只复用纯 Web 能力，不单独做原生宠物系统。

## UI 与入口建议

- 首页：只放轻量状态入口，例如宠物头像、心情、一个当前可执行动作。
- 独立页面：承载完整照顾、装扮、任务和成长记录。
- 个人主页：展示宠物名片和可选装扮，不展示敏感行为记录。
- Console：后续承载数值配置、物品管理、异常用户查询和状态流水。

视觉方向应服从 [视觉主题规范](/frontend/visual-theme-spec) 和 [颜色参考](/frontend/visual-color-reference)，保持淡雅新中式，不做过重游戏化界面。

## 风险与约束

- 防刷优先：任何把社区行为转化为成长材料的规则都必须有每日上限。
- 经济稳定优先：萝卜币消耗只做装饰和轻量便利，不做核心成长捷径。
- 用户压力控制：不做高频掉状态、高频通知或惩罚式流失机制。
- 隐私边界：默认不公开用户所有照顾行为，只展示用户主动选择公开的宠物状态。
- 公平性：不做强竞技排行榜，不把宠物系统变成攀比压力源。

## 当前不做

- 不在本批直接实现完整电子宠物代码。
- 不做战斗、繁殖、交易市场、抽卡或强付费成长。
- 不把宠物系统塞进 WebOS `/desktop` 作为新增主线。
- 不把首页信息流改造成宠物页。
- 不绕过 PublicId 规划直接暴露内部 long 主键。
