## 产品定位

Radish 电子宠物是面向长期复访、轻互动和社区陪伴感的 Web-first 玩法系统。它不应只是一个独立小游戏，也不应成为萝卜币或经验系统的附属 UI，而是把日常访问、内容互动、关系链、轻任务和资产消耗组织成更可持续的用户回访场景。

首批定位：

- 默认优先进入纯 Web 入口，后续再评估 Flutter 原生承接。
- 可以和萝卜币、经验、签到、评论、轻回应、关注、公开主页产生联动。
- 不直接影响核心社区阅读和内容创作路径。
- 数值、奖励和消耗必须先设计清楚，避免制造刷分和通胀风险。

## 核心循环

建议首版围绕低压力日常照顾展开：

1. 用户获得一只 Radish 宠物。
2. 每日访问、喂养、清洁、互动或完成轻任务提升宠物状态。
3. 宠物状态影响外观、心情、成长阶段和少量展示权益。
4. 社区互动可给宠物提供成长材料，但不得强迫用户刷评论或刷点赞。
5. 萝卜币可用于购买装饰、食物或玩具，但不直接购买核心等级优势。

## B8 开发前冻结口径

本节作为进入代码前的评审基线。后续若要改变入口、经济消耗、公开展示或任务奖励口径，应先更新本文档，再进入实现。

首版目标：

- `Phase B` 首批代码先完成登录态纯 Web 宠物首批可用链路：`/pet` 独立页面、`/me` 宠物摘要入口、基础照顾动作、基础成长状态和状态变化流水。
- 首次进入 `/pet` 时展示领取引导，用户确认后创建一只默认宠物；只读查询接口不做隐式写入。
- 宠物首版默认私域展示，不进入公开 sitemap；公开个人主页只在用户显式开启后展示宠物名片。
- 首页信息流不承载首批宠物组件，避免抢占 `/discover` 的公开内容分发职责；首页小组件待 `/pet` 稳定后再评估。
- 首版不接入萝卜币消耗、商城物品购买、评论 / 点赞 / 关注任务奖励或经验反向加成，只预留配置、流水和后续扩展点。

首版 MVP 切片：

| 范围 | 首批做法 | 明确不做 |
| --- | --- | --- |
| 宠物主档 | 每个用户一只宠物，包含 `PublicId`、名称、形态模板、成长阶段、心情、饱食度、清洁度、精力、成长值和当前装扮键 | 多宠物、繁殖、交易、排行榜、公开竞技 |
| 领取与命名 | `/pet` 提供领取和命名；名称走长度、敏感词和重复提交校验 | 自动批量创建、未登录体验内写入 |
| 日常照顾 | 喂食、清洁、互动、休息四类动作；后端控制每日次数、冷却和状态上限 | 复杂连招、在线倒计时压力、惩罚式流失机制 |
| 成长系统 | 成长值只来自首批照顾动作；阶段阈值来自配置或种子数据 | 经验等级反哺、评论刷成长、付费购买等级 |
| 私域入口 | `/pet` 为登录态主入口，`/me` 只展示宠物摘要和当前可执行动作 | 把宠物塞进 WebOS `/desktop`，把 `/discover` 改成宠物页 |
| 治理基础 | 所有状态变更写入宠物流水，支持后续审计和异常排查 | Phase B 直接做完整 Console 配置后台 |

## F4-H 公开宠物名片与隐私闭环

> **专题状态**：F4-H-A-D 已完成并关闭
>
> **正式消费面**：公开个人主页 `/u/:id`
>
> **私域控制面**：登录态 `/pet`

### 1. 专题裁决

F4-H 只完成现有“公开展示宠物名片”承诺，不扩展宠物玩法。当前 `PetProfile` 已持久化 `IsPublic`、`pet_` 前缀 `PublicId`、名称、物种、形态、成长阶段、心情与两项装扮预留键，`/pet` 也允许用户保存公开开关；但真实代码中仍存在以下闭环缺口：

- `PetController` 当前全部要求登录态，只能读取当前用户自己的完整宠物状态。
- `User/GetPublicProfile` 没有宠物公开投影，`/u/:id` 也没有宠物名片消费面。
- 现有 `PetProfileVo` 同时包含三项私域状态、成长值、最后照顾时间和动作资格，不能直接作为公开返回对象。
- `EquippedBackgroundKey / EquippedToyKey` 只是预留字段，当前没有正式装备来源、资源注册表或购买 / 发放闭环，不能把任意持久化 key 当作可信公开资源。

因此，F4-H 采用“公开用户资料聚合一个裁剪后的宠物名片”方案，不新增独立宠物公开页、分享页或 sitemap 项，也不把完整私域 `PetProfileVo` 暴露给匿名请求。

### 2. 用户路径与产品价值

唯一正式路径如下：

1. 宠物主人在 `/pet` 显式开启“公开展示宠物名片”。
2. 保存成功后，后续公开资料请求由服务端实时读取当前宠物公开状态。
3. 访客在该用户既有 `/u/:id` 公开个人主页看到一张只读宠物名片。
4. 主人关闭公开、宠物软删除、用户禁用 / 软删除或租户不匹配后，后续公开资料请求立即不再返回宠物名片。

该路径服务公开身份表达与社区关系复访，不提供照顾入口、排行、比较、任务或经济刺激。匿名访客不能通过名片推断宠物是否饥饿、主人何时照顾、消费记录或动作习惯。

### 3. 权威读取与隐私语义

公开宠物名片作为 `UserPublicProfileVo` 的可空投影 `VoPet` 返回，复用既有 `GET /api/v1/User/GetPublicProfile?identifier=...`，不新增由前端传入 `userId / petId / tenantId` 的第二套查询入口。

服务端处理顺序固定为：

1. 先按既有公开用户标识解析有效用户；兼容期 LongId 只属于公开用户路由的既有外层语义，不下传为宠物公开标识。
2. 使用服务端已解析的 `user.Uuid + user.VoTenantId` 查询宠物，不接受客户端提交租户或内部宠物 LongId。
3. 只接受 `TenantId` 与用户一致、`UserId` 一致、`IsPublic = true` 且 `IsDeleted = false` 的唯一有效宠物。
4. 任一条件不满足时统一返回 `VoPet = null`，不区分未领取、未公开、已删除、跨租户或失效，避免形成存在性探测接口。

`UpdateProfile(IsPublic=false)` 成功提交后，所有后续权威读取必须立即返回 `VoPet = null`。F4-H 不为匿名已打开页面建立实时撤回通道；但公开资料响应不得持久化到 localStorage、离线缓存或 head snapshot，刷新、重新进入或重新请求必须观察到最新隐藏状态。

宠物 `PublicId` 仅作为返回对象的稳定公开身份与后续契约预留，不在 F4-H 新增 `/pet/:id` 路由，也不允许从它反查主人私域信息。

### 4. 公开字段白名单

新增独立 `PetPublicCardVo`，只允许以下字段：

| 字段 | 公开语义 |
| --- | --- |
| `VoPublicId` | `pet_` 前缀稳定标识；不返回内部 LongId |
| `VoName` | 主人设置的宠物名称，按用户内容原文展示 |
| `VoSpeciesKey` | 服务端白名单内的物种展示键 |
| `VoShapeKey` | 服务端白名单内的形态展示键 |
| `VoGrowthStage` | 稳定数值阶段，由 client 按当前语言解析词元 |
| `VoMood` | 稳定心情键，由 client 按当前语言解析词元 |
| `VoAdornment` | 可空安全装扮摘要；只含服务端注册且可公开的展示键 |

明确禁止进入公开 VO：

- `VoId / VoUserId / TenantId` 等内部标识；
- `VoSatiety / VoCleanliness / VoEnergy / VoGrowthValue`；
- `VoLastCareTime / VoCreateTime`；
- `VoCareActions`、每日次数、冷却、动作资格；
- `PetStatLog`、幂等键、来源、消费、任务和审计信息；
- 服务端中文 display 字段或由页面自行推导的私域状态洞察。

装扮摘要采用白名单而不是透传数据库字段。F4-H-B 若尚无正式可验证的宠物装扮注册项，`VoAdornment` 必须为 `null`；不得为了让字段非空而提前创建商品、背包、装备或附件协议。未来正式装扮能力仍需独立专题建立拥有关系、资源 ACL 和公开资源注册后，才能进入该摘要。

### 5. 服务端边界

- `IPetService` 增加公开卡片读取方法；公开投影在 Pet Service 内构造，不在 `UserController` 手写字段映射或直接注入 Repository。
- 查询必须同时包含用户、租户、公开状态和软删除条件；不能先读取完整 `PetProfileVo` 再由 Controller 删除敏感字段。
- `UserController.GetPublicProfile` 继续负责公开用户聚合，只把已解析用户的服务端身份传给 Pet Service。
- 隐藏或不存在不是错误；公开用户仍返回 `200`，其中 `VoPet = null`。用户本身不存在继续沿用既有 `404`。
- F4-H 不新增数据库字段或 migration。若实现审计发现既有索引不足，只能在 F4-H-B 以真实查询计划和 SQLite / PostgreSQL 证据裁决，不能为架构对称性预建索引。
- `PetProfileVo` 继续服务本人私域，不能与 `PetPublicCardVo` 合并；两者共享稳定展示键，但公开 VO 永远保持字段白名单。

### 6. 正式 Web 边界

- 先更新 `public-web.pen` 中公开个人主页 PC / mobile 画板，再修改正式页面；不绕过 Pencil 新增卡片布局。
- 宠物名片位于公开身份摘要与内容列表之间，属于用户身份补充，不新增公开主页 tab，不抢占帖子 / 评论主内容区域。
- 无公开宠物时整块不渲染，不显示“该用户未领取”或“该用户选择私密”等可探测占位。
- 名片只展示名称、形态、成长阶段、心情与可用的安全装扮摘要；不显示照顾按钮、进度条、私域数值或跳转 `/pet` 的访客操作。
- 主人查看自己的公开主页时也遵守相同公开投影；管理公开开关仍回到 `/pet`，不能在公开页建立第二套写入口。
- `zh / en` 文案、成长阶段和心情复用现有稳定词元规则；宠物名称保留原文。四主题仅使用语义 token。
- public head、JSON-LD、sitemap 和公开主页 canonical 不加入宠物信息，避免关闭公开后仍由 head snapshot 或搜索索引保留旧状态。
- WebOS `/desktop` 不新增宠物名片功能；其打开公开个人主页时只复用正式 `/u/:id` 路径，不维护第二套组件或状态。

### 7. 不污染边界与停止线

F4-H 不产生：

## 经济与任务约束

`Phase B` 不直接消耗或发放萝卜币，也不把社区互动转换为宠物成长材料。这样可以先验证复访与照顾主链路，再把经济和防刷治理作为独立批次评审。

进入 `Phase C` 前必须补齐：

- 萝卜币消耗的流水类型、幂等键、失败回滚和重复提交语义。
- 宠物物品的价格、库存或不限量策略、发放目标和背包关系。
- 社区行为任务的每日上限、冷却、异常检测和封禁 / 冻结后的处理。
- 经验系统与宠物系统的单向关系：经验可以作为展示或低额任务输入，宠物成长不得反向放大经验收益。
- Console 查询口径：至少能按用户、宠物、动作类型和时间范围查看状态流水。

## 数据模型草案

后续实现前应先确认实体边界：

- `PetProfile`：用户宠物主档，包含名称、形态、成长阶段、状态数值和当前装扮。
- `PetStatLog`：状态变化流水，用于审计和异常排查。
- `PetItem`：食物、玩具、装饰等物品定义。
- `UserPetInventory`：用户拥有的宠物物品。
- `PetTaskProgress`：每日任务进度和奖励领取状态。

所有对外展示对象应预留 `PublicId`，不直接把内部 long 主键作为公开链接或分享标识。

### 首批后端契约草案

后端实现时优先保持以下边界：

- 实体放入 `Radish.Model/Models`，对外对象放入 `Radish.Model/ViewModels`，返回前端的类使用 `Vo` 后缀，字段使用 `Vo` 前缀。
- Controller 只注入宠物 Service，不直接注入 Repository；简单查询优先复用 `BaseService` / `BaseRepository` 能力。
- `/pet` 私域接口使用登录态用户身份定位当前宠物，不通过公开 URL 暴露内部 long 主键。
- 公开展示复用用户公开资料聚合：外层只解析用户公开标识，Pet Service 使用服务端已解析的用户与租户身份返回字段白名单；F4-H 不新增按宠物标识直查的独立接口。
- 状态变更接口必须由服务端计算结果，前端只提交动作类型或物品标识，不提交最终数值。
- 状态变化流水至少记录用户、宠物、动作类型、变更前后值、来源、幂等键和创建时间。

首批 API 候选：

| 接口 | 用途 | 备注 |
| --- | --- | --- |
| `GET /api/v1/Pet/GetMy` | 读取当前用户宠物状态 | 未领取时返回明确空态，不隐式创建 |
| `POST /api/v1/Pet/Claim` | 领取默认宠物 | 需要幂等处理，重复领取返回已有宠物 |
| `PUT /api/v1/Pet/UpdateProfile` | 修改宠物名称或展示设置 | 不处理装饰购买 |
| `POST /api/v1/Pet/Care` | 执行喂食、清洁、互动或休息 | 后端校验冷却、每日上限和状态边界 |
| `GET /api/v1/Pet/GetLogs` | 查询当前用户宠物流水 | Phase B 可先提供最近记录 |

## 阶段路线

### Phase A：方案冻结

- 明确宠物定位、数值范围、奖励来源和消耗边界。
- 明确 Web 入口：独立页面、首页小组件、个人主页展示的优先级。
- 明确是否接入萝卜币、经验和轻任务，以及各自上限。
- 明确不做范围：复杂养成战斗、交易市场、排行榜攀比、强付费成长。

### Phase B：Web 首版

- 实现宠物档案、日常照顾、基础成长和状态展示。
- 首批不接入首页信息流；后续如接入首页，只展示轻量宠物状态，不影响信息流阅读。
- 个人主页可选择展示宠物，但默认不暴露过多私域状态。
- 补后台配置或种子数据，避免关键数值散落在前端。

### Phase C：经济与治理

- 接入萝卜币消耗和宠物物品。
- 补状态变化流水、异常检测、每日上限和后台查询。
- 评估是否把部分任务与评论、轻回应、关注等社区行为联动。

### Phase D：跨端承接

- Flutter 只在 Web 首版稳定后承接日常照顾、状态提醒和轻量展示。
- 不把 Flutter 作为首发实现端。
- PC/Tauri 如有安装包价值，只复用纯 Web 能力，不单独做原生宠物系统。

## UI 与入口建议

- 首页：后续只放轻量状态入口，例如宠物头像、心情、一个当前可执行动作。
- 独立页面：首批承载完整照顾、装扮、任务和成长记录。
- 个人主页：展示宠物名片和可选装扮，不展示敏感行为记录。
- Console：后续承载数值配置、物品管理、异常用户查询和状态流水。

视觉方向应服从 [视觉主题规范](/frontend/visual-theme-spec) 和 [颜色参考](/frontend/visual-color-reference)，保持淡雅新中式，不做过重游戏化界面。

## 隐私与公开展示

- `/pet` 和 `/me` 宠物摘要均为登录态私域入口。
- 公开个人主页默认不展示宠物；用户开启后只展示宠物名称、形态、成长阶段、心情和装扮摘要。
- 饱食度、清洁度、精力、照顾时间、任务进度、消费记录和状态流水默认不公开。
- 分享链接和通知载荷优先使用 `PublicId`，不把内部 long 主键写入公开 URL。
- 来源返回语义沿用纯 Web 一次性来源转交，不把 `/desktop` 作为宠物新增入口。

## 风险与约束

- 防刷优先：任何把社区行为转化为成长材料的规则都必须有每日上限。
- 经济稳定优先：萝卜币消耗只做装饰和轻量便利，不做核心成长捷径。
- 用户压力控制：不做高频掉状态、高频通知或惩罚式流失机制。
- 隐私边界：默认不公开用户所有照顾行为，只展示用户主动选择公开的宠物状态。
- 公平性：不做强竞技排行榜，不把宠物系统变成攀比压力源。

## 当前不做

- 不在 Phase B 首批范围继续扩展完整游戏化养成系统。
- 不做战斗、繁殖、交易市场、抽卡或强付费成长。
- 不把宠物系统塞进 WebOS `/desktop` 作为新增主线。
- 不把首页信息流改造成宠物页。
- 不绕过 PublicId 规划直接暴露内部 long 主键。
- 不接入萝卜币消耗、商城物品、社区任务奖励、经验反向加成或 Console 配置 UI。
