## 为什么问答优先于抽奖

在“抽奖 / 问答”两个后续候选里，当前更适合优先做问答：

- **复用论坛能力更多**：可继续挂在帖子模型上，复用发帖、详情、评论、标签、分发和权限链路。
- **工程风险更低**：相比抽奖，不需要先解决公平性、防刷、开奖、奖品或奖励发放审计。
- **产品边界更清晰**：先做“提问帖 + 回答 + 采纳”就能形成闭环。
- **对当前阶段更友好**：现在刚结束一轮权限、聊天室、投票的收口，更适合继续做“小而快”的内容能力，而不是进入活动型系统治理。

因此，本方案最终选择了：

- **问答 MVP 先落地**
- **抽奖继续保留在后续候选，不作为当前先手**

## 目标

在现有论坛帖子主链路上增加一套**最小可用的问答能力**，满足以下目标：

- 用户可发布“提问帖”。
- 其他登录用户可针对提问帖发布回答。
- 提问者可从回答中采纳一个“最佳答案”。
- 列表页可识别问答帖，但不引入复杂聚合页或独立问答 App。
- 尽量复用现有帖子详情、评论、用户身份和多租户链路。

本轮目标是“**问答闭环最小成立**”，不是一次性做成完整 Stack Overflow 式系统。

---

## MVP 范围

### 本轮包含

- 提问帖标记
- 问题正文继续复用现有帖子内容
- 帖子详情页中的“回答列表”
- 登录用户提交回答
- 提问者采纳一个最佳答案
- 列表页显示“问答”标识与回答数
- 详情页显示“已解决 / 待解决”状态

### 本轮不包含

- 悬赏 / 奖励 / 萝卜币联动
- 多问题合帖
- 回答评论与楼中楼特殊规则
- 回答排序算法优化
- Console 问答管理页
- 独立问答频道 / 独立问答 App
- 回答编辑历史专属能力
- 最佳答案变更审计

## 产品形态

### 位置选择

问答不做独立应用，直接挂在论坛帖子体系内：

- **发布入口**：论坛发帖弹窗新增“作为问答发布”开关，并与“附带投票”互斥。
- **列表页**：帖子卡片显示“问答”徽标、回答数、是否已解决。
- **详情页**：正文下方增加“回答区”，下方保留“讨论区”，保持“答案”和“讨论”边界清晰。

### 用户流

#### 提问者

1. 打开发帖弹窗
2. 输入标题、问题描述、分类、标签
3. 开启“作为问答发布”
4. 发布提问帖
5. 在详情页查看回答
6. 选择一个最佳答案并采纳

#### 回答者

1. 在列表识别问答帖
2. 打开帖子详情
3. 查看问题正文与已有回答
4. 提交一条回答
5. 查看是否被采纳

## 业务规则

### 发帖规则

- 问答帖仍然要求标题、正文、分类、标签。
- 问答帖与普通帖互斥；MVP 阶段不支持“投票 + 问答”叠加。
- 一个帖子最多只代表一个问题主体。

### 回答规则

- 仅登录用户可回答。
- 提问者本人不可采纳自己的回答。
- MVP 阶段允许一个用户对同一问题多次回答，但前端可引导“优先补充已有回答而不是灌水”。
- 回答默认按“采纳优先，再按创建时间正序”展示即可，不引入复杂投票排序。

### 采纳规则

- 仅提问者本人可采纳最佳答案。
- MVP 阶段一个问题只允许存在一个最佳答案。
- 已采纳后默认允许更换，但本轮建议保持克制：
  - **默认不支持二次更换**，避免边界扩散。
- 采纳后帖子状态变为“已解决”。

## 后端设计

### 实现路径

优先采用“**帖子 + 问答扩展字段 + 回答实体**”方案，而不是直接把现有评论体系硬改成回答体系。

原因：

- 评论当前已有自己的交互语义，直接复用会让“回答”和“评论”边界变混。
- 问答的采纳状态、回答数、最佳答案标记天然更适合独立实体承载。
- 这样既能复用帖子主链，又能保持后续扩展空间。

### 建议新增实体

#### 1. `PostQuestion`

表示一个帖子是否为问答帖，以及问答状态。

建议字段：

- `PostId`
- `IsSolved`
- `AcceptedAnswerId`
- `AnswerCount`
- `TenantId`
- `IsDeleted`
- 审计字段

#### 2. `PostAnswer`

表示问题下的一条回答。

建议字段：

- `PostId`
- `AuthorId`
- `AuthorName`
- `Content`
- `IsAccepted`
- `TenantId`
- `IsDeleted`
- 审计字段

### 模型约束

- `PostQuestion`、`PostAnswer` 建议实现 `ITenantEntity`。
- 两者建议走软删除。
- 一个帖子最多有一个 `PostQuestion`。
- 一个 `PostQuestion` 最多有一个 `AcceptedAnswerId`。

## 接口设计

### 最小接口集

#### 1. 发布问答帖

`POST /api/v1/Post/Publish`

- 继续复用现有发帖接口
- 请求中增加 `isQuestion`

#### 2. 获取帖子详情（包含问答详情）

`GET /api/v1/Post/GetById/{id}`

- 若为问答帖，则返回 `VoQuestion`

#### 3. 提交回答

`POST /api/v1/Question/Answer`

请求：

- `postId`
- `content`

返回：

- 最新回答列表或最新问答详情

#### 4. 采纳答案

`POST /api/v1/Question/Accept`

请求：

- `postId`
- `answerId`

返回：

- 最新 `PostQuestionVo`

### 错误语义

建议统一返回明确失败消息：

- 当前帖子不是问答帖
- 请先登录后再回答
- 回答内容不能为空
- 回答不存在
- 只有提问者可以采纳答案
- 当前问题已采纳答案
- 不能采纳自己的回答

## 接口返回建议

### 帖子列表

列表页只返回轻量摘要：

- `VoIsQuestion`
- `VoIsSolved`
- `VoAnswerCount`

### 帖子详情

详情页返回完整问答信息：

- 问答状态
- 回答列表
- 最佳答案标记

这样可以继续保持“列表轻、详情重”的边界。

## 权限与安全

### 权限边界

- 问答帖发布权限继续复用现有发帖权限。
- 回答权限默认跟随“登录用户可发布内容”的最小规则，不额外引入新权限族。
- 采纳权限只由提问者本人持有。

### 风险点

#### 1. 回答与评论混淆

MVP 阶段应明确回答区与评论区的职责，不建议直接复用评论树作为回答主数据源。

#### 2. 采纳权限越权

必须在服务层校验提问者身份，不能只靠前端隐藏按钮。

#### 3. 回答数与采纳状态一致性

回答创建、删除、采纳时需要同步维护 `AnswerCount` 与 `AcceptedAnswerId`。

## 验收标准

满足以下条件即可认为问答 MVP 可进入验收：

- 可发布问答帖
- 列表可识别问答帖
- 详情可展示回答区
- 登录用户可提交回答
- 提问者可采纳最佳答案
- 采纳后状态显示正确
- 前后端字段口径一致，无匿名对象返回

### 当前验收结论（2026-03-14）

- 已完成一轮前后端联调，问答帖发布、列表识别、详情展示、回答提交、提问者采纳均已通过。
- 当前实现保持“问答答案独立于评论树”的模型，不与评论时间线直接混排。
- 回答输入、已采纳高亮、作者入口与成功反馈均已完成最小可用收口。

## 建议执行顺序

1. 先补文档与字段方案，确认“问答帖”和“普通帖 / 投票帖”的互斥规则。
2. 再做后端最小模型、DTO / Vo 与详情返回扩展。
3. 接着完成回答提交与采纳接口。
4. 最后做前端发帖开关、详情展示与最小回归。
