# 论坛轻回应墙设计草案

> 状态：设计中，尚未开发
>
> 最后更新：2026-04-07（Asia/Shanghai）

## 背景

用户反馈当前论坛帖子详情页里，正文之后到评论区之前缺少一种“轻量、低门槛、弱打扰”的即时互动层。

竞品常见做法是在每条回复下方持续堆叠短回应，但这会把互动入口分散到整棵评论树中，页面密度和治理成本都会快速上升。

本项目建议改为：

- 只在**帖子正文下方**放置一个独立的“轻回应墙”区域。
- 用户发送短内容后，以“头像 + 简短内容”的胶囊卡片形式滚动展示。
- 该区域强调“快速参与感”，不替代标准评论区、问答区或 Reaction。

## 目标

- 为帖子提供一层低门槛即时互动。
- 缩短用户从“想参与”到“留下痕迹”的路径。
- 保持帖子正文后的热度氛围，但不破坏评论区主链路。

## 非目标

- 不替代标准评论、楼中楼、问答回答或帖子 Reaction。
- 不承担长文本讨论、结构化回复、历史编辑等职责。
- 不在每条评论下重复渲染，避免界面失控。

## 建议命名

当前建议内部统一称为“轻回应墙”。

可选用户侧文案：

- 轻回应
- 路过留一句
- 帖内弹幕

首版建议优先使用“轻回应”，避免“弹幕”一词让用户误解为全屏飞行弹幕。

## 放置位置

放在帖子详情页的以下结构之间：

1. 帖子正文 / 标签 / 帖子主互动区之后
2. 标准评论树之前

这样可以明确传达三层语义：

- 正文：作者主表达
- 轻回应墙：围观用户的即时情绪与短反馈
- 评论区：正式讨论与回复

## 交互形态

### 展示

- 使用 2 到 3 行滚动胶囊卡片区域。
- 卡片内容包含：
  - 发送者头像
  - 昵称或昵称首字
  - 短内容文本
- 支持轻微自动滚动；悬停时暂停。
- 移动端降级为横向滑动列表，不强依赖动画。

### 发送

- 输入框固定为单行或紧凑双行输入。
- 字符上限建议 `16 ~ 24` 个中文字符。
- 发送成功后立即插入本地列表，再与服务端结果对齐。
- 未登录用户点击输入框时，沿用当前论坛登录引导。

### 内容节奏

- 默认展示最近一批内容。
- 可考虑“最新优先 + 同用户短时间去重”。
- 首版不做复杂筛选，不引入分页入口。

## 数据模型建议

不建议复用 `Comment` 或 `Reaction`。

建议新增独立模型，例如：

```text
PostQuickReply
- Id
- PostId
- AuthorId
- AuthorName
- AuthorAvatarUrl
- Content
- Status
- IsDeleted
- CreateTime
- DeletedAt
- DeletedBy
```

原因：

- `Comment` 的职责是正式讨论，字段与治理规则都更重。
- `Reaction` 适合表情/贴图聚合，不适合承载短文本。
- 轻回应墙未来可能需要独立限流、独立审核和独立排序。

## 后端接口建议

首版建议至少提供：

- `GET /api/v1/PostQuickReply/GetRecentByPostId?postId={id}`
- `POST /api/v1/PostQuickReply/Create`
- `POST /api/v1/PostQuickReply/Delete`

可选扩展：

- 批量拉取多个帖子轻回应摘要，用于列表热度预览。
- 管理端审核/删除接口。

## 内容治理建议

这是首版必须考虑的部分，不建议后补。

### 约束

- 单条字数上限：建议 `24`
- 单用户单帖发送频率限制：建议 `30s / 1 条`
- 同内容短时重复发送拦截

### 治理

- 接入现有敏感词/内容审核链路
- 支持举报
- 支持软删除
- 支持后台清理和最小审计留痕

## 前端实现建议

### 组件拆分

建议新增：

- `PostQuickReplyWall`
- `PostQuickReplyComposer`
- `PostQuickReplyMarquee` 或 `PostQuickReplyFlow`

### 状态策略

- 首屏进入帖子详情时加载最近数据
- 发送后本地乐观插入
- 如果后续需要实时感，可再接入轮询或 SignalR

### 视觉口径

- 维持论坛当前的浅色卡片风格
- 重点突出“轻”和“快”，不要做强侵入大动效
- 内容区域应有明显边界，但不能比评论区更重

## 分阶段建议

### Phase 1

- 独立数据模型
- 最近内容展示
- 文本发送
- 限流与软删除
- 基础举报/治理

### Phase 2

- 贴图或表情型轻回应
- 热门轻回应排序
- 列表页摘要预览
- 实时推送

原因：

- 该项属于论坛稳定维护期的产品增强，不是阶段主线路径变更。
- 在后端模型、治理边界和前端交互统一前，不适合直接用临时拼接方式落地。
