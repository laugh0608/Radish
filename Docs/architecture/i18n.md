# 国际化与多语言规范（i18n）

> 本文是 Radish 多语言长期契约，描述正式 Web、共享前端模块、API 与 Auth 的稳定职责。
>
> 当前只支持简体中文与英文；F3 的批次状态、迁移范围和验收矩阵见 [F3 i18n 完成度治理实施说明](/frontend/i18n-completion-governance)。

## 1. 总体约定

- 产品内部语言代码固定为 `zh | en`，不在状态层混用 `zh-CN`、`zh-Hans`、`en-US` 等地区标签。
- `zh-CN / en-US` 只用于 `Intl`、Ant Design 和 HTML `lang` 等展示层映射。
- 纯 Web 是唯一正式产品主线，覆盖 `radish.client`、`radish.console` 与 `Radish.Auth`。
- `/desktop` 只保持 WebOS 阻断级兼容；Flutter 条件式维护，Tauri 冻结，不随 Web 翻译资源同步扩张。
- 用户帖子、评论、聊天、商品自定义内容与 Docs 正文不自动翻译。
- 当前语言是同一浏览器设备的偏好，不写入 Profile、Claim 或数据库，也不承诺跨设备同步。

模块职责：

| 模块 | 职责 |
| --- | --- |
| `radish.client` | 产品端语言状态、正式 Header 入口、产品业务资源、页面格式化 |
| `radish.console` | 管理端独立资源、管理壳入口、路由与管理业务文案 |
| `@radish/ui` | 接收宿主 locale / labels，提供无宿主状态的共享格式化与组件契约 |
| `@radish/http` | 接收宿主语言与翻译函数，发送 `Accept-Language`，保留结构化错误字段 |
| `Radish.Api` | 生成稳定 HTTP status、`Code`、`MessageKey`、安全 `MessageInfo`、规范化 `MessageArguments` 和 `TraceId` |
| `Radish.Auth` | 通过 Request Culture、文化 Cookie 与 `.resx` 渲染 Razor 页面和校验反馈 |

## 2. 语言状态与持久化

### 2.1 client 与 Console

两端各自初始化 i18next，但共享同源设备偏好键：

```text
localStorage.radish_lang = zh | en
```

解析优先级固定为：

```text
?lang=zh|en -> localStorage.radish_lang -> navigator.language -> zh
```

实现入口：

- client：`Frontend/radish.client/src/i18n.ts`、`src/locales/language.ts`、`src/i18n/LanguageProvider.tsx`
- Console：`Frontend/radish.console/src/i18n.ts`、`src/locales/language.ts`、`src/i18n/LanguageProvider.tsx`

规则：

- 所有输入必须先归一化为 `zh | en`；`zh-*` 映射为 `zh`，`en-*` 映射为 `en`，其他值无效。
- i18next 使用 `fallbackLng: 'zh'` 与 `load: 'languageOnly'`。
- 显式切换使用 `i18n.changeLanguage(language)`，由 detector 写入 `radish_lang`。
- `LanguageProvider` 同步 `document.documentElement.lang`，并通过 `storage` 事件响应其他标签页的设备偏好变化。
- client 与 Console 不互相引用资源文件，只共享语言代码、存储键和行为契约。

### 2.2 Auth

Auth 的请求文化优先级为：

```text
Query String -> .AspNetCore.Culture Cookie -> Accept-Language -> zh
```

- `Account/SetLanguage` 将 `en*` 归一化为 `en`，其余输入退回 `zh`，不会把任意文化值写入 Cookie。
- 文化 Cookie 为一年有效、`HttpOnly`、`Essential`、`SameSite=Lax`；HTTPS 请求使用 Secure。
- `returnUrl` 必须是本地 URL，禁止语言切换形成开放重定向。
- client / Console 进入 OIDC 时可携带 `culture` 与 `ui_locales`；Auth 仍以自己的 Request Culture 和 Cookie 为页面真相源。

## 3. 翻译资源组织

### 3.1 前端资源

client 资源按“语言 × 业务域”组织，由单一 registry 合并：

```text
Frontend/radish.client/src/locales/
  resources.ts
  language.ts
  zh/{core,shell,discover,community,account,commerce,docs}.ts
  en/{core,shell,discover,community,account,commerce,docs}.ts
```

Console 使用相同 registry 形态，但维护自己的 `core / shell / dashboard` 及后续管理业务域。

```text
Frontend/radish.console/src/locales/
  resources.ts
  language.ts
  zh/{core,shell,dashboard,users,moderation,orders,products,settings,documents}.ts
  en/{core,shell,dashboard,users,moderation,orders,products,settings,documents}.ts
```

翻译键按语义和所有权划分：

- `common.*`：真正跨业务域且生命周期一致的动作或状态。
- `shell.* / lang.* / auth.* / oidc.*`：壳层、语言和认证。
- `discover.* / forum.* / messages.* / notification.* / profile.* / me.* / shop.* / wiki.*`：client 业务域。
- `console.*`：Console 壳层、路由、导航和跨业务管理动作。
- `dashboard.* / settings.* / users.* / moderation.* / orders.* / products.* / documents.*`：Console 自有业务域；后续域按同样的所有权边界扩展。
- `error.<domain>.* / info.<domain>.*`：需要与服务端 `MessageKey` 对齐的跨层消息。

同一句文案相同不代表应共用 key；只有语义、所有者和生命周期都相同时才能合并到 `common.*`。

资源测试至少阻断：

- 中英文缺 key；
- 同一语言或跨域重复 key；
- 非法顶级域；
- 资源重新膨胀为超大单文件；
- 已登记服务端 `MessageKey` 与前端错误键漂移。

### 3.2 稳定系统词元与业务内容

系统词元、本地化展示和业务内容必须分层处理：

- 页面控制、筛选、状态样式和本地化映射只读取稳定代码、枚举值或 key，例如经验 `voExpType`、萝卜流水 `voTransactionType`、商品能力 `voConfigurationRequirementKeys`、系统设置 `voKey`。
- `vo*Display`、服务端兼容说明和当前语言下的展示文本不得参与控制流；它们只可作为缺少稳定 key 时的安全展示回退。
- 已知系统词元映射宿主资源；未知稳定值按各业务域明确的 `Unknown` / 兼容策略处理，没有登记策略时显示原值，不能根据 display 文本猜测归类。
- 等级名称、商品名称与描述、设置覆盖值、管理员备注、冻结 / 撤销原因、文档标题 / 摘要 / 正文、用户资料等配置型或人工内容，在没有稳定本地化标识时保留原文。
- 同一字段不能同时承担稳定控制值和可翻译展示文案；新增跨层契约时应分别提供稳定字段与展示回退字段。

### 3.3 Auth 资源

Auth 使用：

```text
Radish.Auth/Resources/Errors.resx
Radish.Auth/Resources/Errors.zh.resx
Radish.Auth/Resources/Errors.en.resx
```

- 默认、中文、英文资源必须保持 key 完整一致。
- Login、Register、Consent 及 DataAnnotations 校验都通过 `IStringLocalizer<Errors>` 读取资源。
- Razor View 不维护 `isZh ? ... : ...` 双分支。
- 客户端名称、redirect host、用户输入等动态内容只做安全编码，不机器翻译。

## 4. HTTP 与错误翻译契约

### 4.1 稳定字段

前后端错误协作使用：

```text
HTTP status + Code + MessageKey + MessageInfo + MessageArguments + TraceId
```

- HTTP status 与 `Code` 决定控制流。
- `MessageKey` 是稳定翻译契约。
- `MessageInfo` 是服务端按请求文化生成的安全回退文案。
- `MessageArguments` 是可选的短位置参数数组，只用于 `MessageKey` 展示格式化，不得参与控制流。
- `TraceId` 用于诊断关联，不默认拼入普通提示。
- 禁止根据中文、英文或正则匹配错误消息决定 not-found、conflict、权限等业务状态。

### 4.2 `@radish/http`

宿主通过 `configureApiClient` 注入：

```ts
configureApiClient({
  getLanguage: () => 'zh',
  translateMessage: (key, args) => i18n.exists(key)
    ? i18n.t(key, Object.fromEntries((args ?? []).map((value, index) => [index, value])))
    : undefined,
});
```

统一客户端负责：

- 在调用方没有显式覆盖时设置 `Accept: application/json` 与 `Accept-Language`。
- `apiGet / apiPost / apiPut / apiDelete` 使用同一解析路径；失败响应存在 `MessageKey` 时才调用宿主翻译器，成功响应保留服务端消息字段，不额外改写。
- 解析结果保留 `ok / data / message / messageInfo / messageKey / messageArguments / code / statusCode / httpStatus / traceId`。
- 失败响应存在 `MessageKey` 时优先使用本地翻译；缺键时保留服务端安全 `MessageInfo`。
- 宿主翻译函数契约为 `translateMessage(key, args?)`；`@radish/http` 只传递位置参数，不解释参数语义或用它改变业务分支。
- 需要抛错时使用 `ApiResponseError` 保留结构化字段，页面通过 status、`Code` 或明确数据状态分支。

`@radish/http` 不依赖 React、i18next 实例或任一宿主资源。当前默认 `getToken` 仍为旧调用保留读取 `localStorage.access_token` 的兼容行为；client 与 Console 正式入口必须通过 `configureApiClient` 注入各自 `tokenService`，新增共享代码不得依赖该默认存储实现。

### 4.3 API 生产职责

- 全局认证、权限、验证、限流、依赖和系统异常必须使用稳定 status、`Code` 与 `MessageKey`。
- 业务域错误按正式链路成组迁移，不做全仓 Controller 机械资源化。
- 对外 `MessageInfo` 必须安全，不暴露内部异常、SQL、路径或凭据。
- 日志保留完整内部诊断并关联 `TraceId`；前端只展示安全消息。
- 动态消息参数统一使用 `MessageArguments` 位置数组，最多 8 项；只允许字符串、字符、布尔和有限数值等安全标量。
- 服务端对单个字符串最多保留 256 个字符，将控制字符规范化为空格；对象、非有限数值与其他不支持值降级为安全占位。
- `MessageArguments` 只用于用户可见文案的格式化，不允许传递任意对象、内部异常或借参数驱动前端流程。

## 5. 共享 UI 与格式化

### 5.1 `@radish/ui` 边界

- `ThemeProvider` 的 Ant Design locale 由宿主传入。
- 共享组件通过 `labels`、formatter 或显式 props 接收可见文案。
- 共享组件不得导入 client / Console i18n、读取 `radish_lang` 或自行决定产品术语。
- 为兼容既有调用保留的中文默认 labels 不能成为新增业务文案入口；正式宿主应显式提供翻译。

### 5.2 日期、数字与相对时间

统一优先使用 `@radish/ui` 提供的：

- `formatLocalizedNumber`
- `formatLocalizedDateTime`
- `formatLocalizedRelativeTime`
- `resolveIntlLocale`

约束：

- locale 决定展示格式，时区决定显示哪个时刻，两者是独立输入。
- 不在页面固定调用 `toLocaleString('zh-CN')` 或依赖浏览器隐式 locale。
- 金额和萝卜数量使用 `Intl.NumberFormat` 的明确 options；单位文案进入翻译资源，不在 formatter 内写死。
- 后端 `long / Int64` 标识与金额在前端保持十进制字符串；完整精度格式化、比较或换算不得先进入 JavaScript `number`，图表需要数值时只在适配边界投影。
- 无效日期保留可诊断的原始值，不伪造当前时间或默认日期。

### 5.3 复数

- 可计数英文文案使用 i18next plural 规则，不手工拼接 `item/items`。
- key 使用 i18next 支持的 `_one / _other` 形式，并至少验证 `0 / 1 / 2`。
- 传给 i18next 的 `count` 保持原始数值，格式化后的数字通过独立参数展示，避免千位分隔字符串破坏英文复数选择。
- 中文可以共享相同语义键，但不能迫使英文退化为单一字符串模板。

## 6. 中英文布局责任

资源完整不等于页面完成。正式 Web 业务域准备验收时，至少覆盖：

- PC `1920 × 1080` 与 mobile `390 × 844`；
- Header、底部导航、表格列头、筛选项、按钮组、弹层与错误说明；
- 英文长标题、长客户端名、redirect URI、动态用户内容混排；
- 加载、空态、成功、失败、无权限、not-found 与可复制诊断信息。

组件应根据职责选择换行、ellipsis、横向滚动或移动端重排，不通过缩小字号或隐藏必要内容掩盖布局问题。

## 7. 新增与修改流程

### 7.1 新增前端文案

1. 确认业务域和所有者，避免把业务文案放入 `common.*`。
2. 在宿主对应的 zh / en 域文件同时增加 key。
3. 使用 `useTranslation()` 或宿主 translator 消费；共享组件改为 labels 契约。
4. 数量文案补 plural，日期和数字使用统一 formatter。
5. 运行资源 parity、重复键和对应页面定向测试。

### 7.2 新增服务端错误

1. 先定义稳定 HTTP status、`Code` 与 `MessageKey`。
2. 在服务端 zh / en `.resx` 中增加安全文案。
3. 若正式 Web 需要本地翻译，在对应宿主资源中增加同名 key。
4. 前端控制流只读取 status / `Code` / 数据状态，用户提示读取本地翻译或 `MessageInfo`。
5. 测试同一结构化错误在中英文消息不同的情况下仍走相同业务分支。

### 7.3 新增共享组件

1. 共享 UI 不持有宿主语言状态。
2. 可见文本通过 labels 或 render props 输入。
3. Ant Design locale、日期与数字格式由宿主或共享 formatter 提供。
4. 默认文案只用于兼容，不代替正式宿主接线。

## 8. 验证与维护

开发中按风险执行：

- client / Console 资源 parity、重复键、归一化与跨标签同步测试；
- `@radish/http` 的 `Accept-Language`、翻译优先级和结构化错误测试；
- `@radish/ui` 的日期、数字、相对时间及组件 labels 测试；
- Auth 默认 / zh / en 资源完整性、文化 Cookie、安全回跳和表单校验测试；
- 四个前端 workspace 的 type-check / test，涉及宿主时执行 production build；
- 涉及后端后执行对应 .NET 测试与解决方案构建；
- repo hygiene 与 `git diff --check`。

真实 Gateway、PC / mobile 和 OIDC 浏览器往返只在专题或成组功能准备验收、且当前任务已确认服务启动或取得启动授权后执行。普通连续开发不沿用历史会话的运行状态。
