## 1. 文档结论

Radish 当前运行时仍保留多租户与 Snowflake 相关实现，但长期方向已经明确调整为：

1. 不再把 `Snowflake long` 视为长期统一对象标识方案
2. 不再把“租户”视为未来社区产品的核心公共语义
3. 未来公开社区与联邦能力以“公开域名 / 部署实例”为节点边界
4. 标识体系正式拆分为 `InternalId / PublicId / FederationId`

本页用于定义长期原则与迁移方向，不要求当前批次立即落地全部实现。

## 2. 当前事实

- 当前大多数核心实体仍以 `RootEntityTKey<long>` 作为数据库主键基础。
- API 与 SignalR 已经因为前端 `Number` 精度边界，将 `long / long?` 统一按字符串传输，说明 `long` 作为前端公开标识已出现契约失配。
- forum 公开直链已在当前主线中出现过真实故障：`/forum/post/:postId` 因前端把 Snowflake 大整数按 `Number` 解析，导致帖子实际存在却被误判为“不存在”。
- 现有 `Snowflake.WorkId / DataCenterId` 仍承担宿主内唯一 ID 生成职责，但它依赖部署编号、时钟与环境纪律，不适合继续扩展为长期公共身份方案。
- 现有多租户隔离、`TenantId` 与相关 QueryFilter 仍是当前系统的重要数据隔离机制，短期不应粗暴移除。

## 3. 目标与非目标

### 3.1 长期目标

- 建立可持续的对象标识体系，使数据库主键、前端公开标识、未来联邦对象 URI 不再耦合为同一个字段。
- 让公开社区、移动端、多壳层与未来联邦都建立在稳定字符串标识之上，而不是继续暴露内部 `long`。
- 为未来社区联邦保留标准协议所需的稳定对象 URI、本地对象与远端对象区分、节点级治理能力。

### 3.2 当前非目标

- 当前阶段不启动全量主键迁移。
- 当前阶段不启动 ActivityPub / WebFinger / 联邦收发实现。
- 当前阶段不删除现有 `TenantId`、`ITenantEntity`、多租户过滤与相关数据库结构。
- 当前阶段不把聊天室、商城、支付、后台治理等全部纳入联邦范围。

## 4. 标识体系分层

### 4.1 `InternalId`

`InternalId` 是数据库内部主键，只承担本地存储与高性能关联职责。

约束：

- 面向数据库、仓储与服务内部使用
- 不直接暴露给前端、公开 URL、第三方协议或联邦对象
- 优先追求 join、索引与写入性能，而不是可读性或公开稳定性

长期推荐：

- 类型：`bigint / long`
- 生成方式：数据库 `sequence / identity` 优先
- 对现存 Snowflake 主键实体，允许在过渡期继续保留，但不再作为未来统一方向扩散

风险判断：

- `bigint` 容量在现实工程尺度下可视为不会用尽
- 它真正的边界不是容量，而是不能再被当成对外公开身份

### 4.2 `PublicId`

`PublicId` 是面向前端、API、公开链接、跨端对象定位与日志排障的稳定字符串标识。

约束：

- 所有对外核心聚合最终都应提供 `PublicId`
- 前端路由、SignalR 契约、REST DTO、公开分享链接最终以 `PublicId` 为主
- `PublicId` 必须与 `InternalId` 解耦，允许后台内部换主键策略而不破坏外部契约

目标形态：

- 逻辑格式：`<type_prefix>_<encoded_body>`
- 示例：`usr_01JV2X6D8S8P4S9T4E4N8M5C0A`
- `type_prefix` 用于跨域日志、调试与人工辨识，例如 `usr / pst / cmt / att / chn`
- `encoded_body` 使用固定长度、大小写规则稳定的可排序字符串

推荐底层标准：

- `UUIDv7`
- 推荐编码：固定长度 `Base32`（Crockford 风格）或等价稳定编码
- 推荐保证：全局唯一、按时间近似有序、大小写规则固定、禁止依赖数据库自增泄露内部规模

实施建议：

- 数据库存储可采用固定长度字符串列并建立唯一索引
- 外部契约直接使用 `PublicId`
- 是否将 `type_prefix` 一并持久化，可按实际索引与可读性权衡决定；逻辑上 `PublicId` 仍以完整字符串形式对外承诺

风险判断：

- `PublicId` 不存在随业务增长而“越变越长”的问题
- 采用 `UUIDv7` 时，在现实产品寿命内可视为不会用尽
- 真正需要治理的是编码规范、唯一索引、线程安全实现和前缀口径一致性

### 4.3 `FederationId`

`FederationId` 是面向未来联邦协议的对象规范标识，语义上不等同于数据库主键，也不等同于本地 `PublicId`。

目标形态：

- 本地对象：规范 URI，例如 `https://node.example/users/alice`
- 远端对象：保留其原始 canonical URI
- Actor 标识可扩展为 handle 视图，例如 `@alice@example.com`

约束：

- `FederationId` 以 URI / 域名归属为核心，不以本地数据库主键为核心
- 远端对象进入本地系统后，不应重新发明一个替代其 canonical URI 的“伪主键”
- 本地 `PublicId` 可作为 URI 的一部分，但不能取代 URI 的联邦语义

## 5. `UUIDv7` 选型结论

### 5.1 对比结果

| 方案 | 优点 | 主要问题 | 结论 |
| --- | --- | --- | --- |
| `Snowflake / 类 Snowflake 64 位整数` | 短、索引友好、近似递增 | 依赖节点编号与时钟；前端精度差；容易把内部主键继续外溢 | 不再作为长期统一标识方案 |
| `ULID` | 字符串较短、可排序、可读性好 | 非通用标准；不同语言库的单调实现口径不完全一致 | 可作为备选，不作为首选 |
| `KSUID` | 可排序、碰撞概率极低 | 更偏社区方案，跨语言与通用工具链自然度不如 UUID 家族 | 可研究，不作为首选 |
| `NanoID / CUID2 / 自研短 ID` | 外观灵活、URL 友好 | 规范与长期维护成本更高，不适合成为核心统一协议 | 不作为核心标准 |
| `UUIDv7` | 标准化方向明确、按时间可排序、跨语言生态稳定、适合公共字符串标识 | 比 `bigint` 索引更大；会暴露粗粒度创建时间；实现需保证单调与线程安全 | 作为 `PublicId` 底层标准首选 |

### 5.2 选择 `UUIDv7` 的原因

- 它更适合作为对外长期对象标识，而不是继续把数据库内部主键外露。
- 它兼顾标准化、可排序与跨语言支持，适合未来移动端、多客户端与联邦扩展。
- 它不需要像 Snowflake 一样把部署节点编号变成长期运行前提。

### 5.3 `UUIDv7` 的边界与隐患

- 它包含时间信息，外部可推断对象大致创建时间，不适合把“创建时间绝对不可观察”作为前提的对象。
- 它的索引和存储体积大于 `bigint`，因此更适合作为 `PublicId` 而不是数据库内部热点主键。
- 若实现要求严格单调，需要选用支持 monotonic 行为且线程安全的生成器。
- 在 SQLite、PostgreSQL 等不同数据库上，需要提前统一字符串编码与比较规则，避免大小写和排序口径漂移。

### 5.4 会不会“算尽”

- `UUIDv7` 基于 `48` 位毫秒时间戳与额外随机位。
- 时间覆盖范围在现实工程尺度内可视为极其充足，可覆盖约数千年量级。
- 结合随机空间与正确实现，现实产品寿命内可视为不会因容量而失效。

结论：

- `UUIDv7` 不应被理解为“绝对无限”，但在 Radish 的现实生命周期内可视为不会算尽。

## 6. 租户语义的长期调整

### 6.1 当前结论

- 当前多租户实现继续保留，作为数据库隔离、查询过滤与历史兼容能力。
- 未来社区产品的公共语义不再以“租户”为中心组织。

### 6.2 长期口径

- 面向用户、公开页面与联邦协议时，优先使用 `instance / node / space / group / category` 这类语义。
- `tenant` 降级为当前实现阶段的数据隔离细节，而不是未来社区产品主语。
- 新文档、新功能命名和新协议设计中，避免把 `tenant` 当作“社区节点”或“公开空间”的同义词。

### 6.3 与现有实现的关系

- 当前 `TenantId`、`ITenantEntity`、QueryFilter、多租户日志与权限链路继续有效。
- 后续若社区产品逐步从“多租户语义”转向“单公开社区 + 联邦节点”，应优先做语义收敛，而不是先粗暴删除数据库字段。
- 长期若确需保留多租户能力，它也应服务于内部部署形态、托管模式或特定合规场景，而不是占据默认社区语义。

## 7. 联邦节点边界

Radish 未来联邦的最小公共节点定义为：

- 一个公开域名 / 一个部署实例 / 一套对外 HTTPS 能力

明确不采用以下口径作为默认联邦节点定义：

- 一个 `Tenant`
- 一个数据库
- 一个组织机构编号
- 一个本地后台分区

原因：

- 联邦协议天然围绕域名、HTTPS、签名、WebFinger 和对象 URI 运转。
- `TenantId` 与公开域名没有天然一一对应关系，混用会导致 URI、权限与治理边界混乱。

## 8. 联邦范围建议

### 8.1 未来优先考虑联邦的对象

- 用户公开身份
- 帖子 / 主题 / 文档等公开内容
- 评论 / 回复
- 关注关系
- 点赞 / Reaction
- 删除 / 封禁 / Tombstone 等状态传播

### 8.2 暂不纳入首批联邦的对象

- 商城
- 订单 / 资产 / 支付
- 后台治理数据
- 内部审计日志
- 大多数站内通知
- 实时聊天室

### 8.3 协议方向

- 公开社区对象联邦优先按 `ActivityPub + WebFinger` 方向预留
- 实时聊天若未来需要跨节点互通，应作为独立议题评估，不与公开社区联邦强行混用

## 9. 迁移原则

### Phase A：先冻结外部 `long`

- 新接口不再继续扩大公开 `long` 主键暴露面
- 新增核心聚合时优先预留 `PublicId`
- 公开路由、分享链接、通知 `extData`、窗口参数、深链查询参数中的对象 ID，当前一律按字符串解析、传递和回写
- 上述冻结要求不是“远期优化建议”，而是当前主线稳定性约束；forum 公开阅读链路已经因大整数精度丢失发生过真实回归

### Phase B：核心聚合双标识

- 对 `User / Post / Comment / Attachment / Channel / Notification / WikiDocument` 等核心聚合增加 `PublicId`
- 外部 API、前端路由、分享链接、实时消息契约逐步切向 `PublicId`
- 仓储与联表暂时仍可基于 `InternalId`

### Phase C：联邦前置准备

- 区分本地对象与远端对象
- 为公开对象建立 canonical URI
- 预留 `Inbox / Outbox / DeliveryQueue / Signature / Retry / Tombstone`

### Phase D：再评估内部主键生成策略

- 当 `PublicId` 已成为外部主契约后，再决定是否把现有 Snowflake 内部主键逐步替换为数据库 `sequence / identity`
- 内部替换不应再牵动前端与公开协议
