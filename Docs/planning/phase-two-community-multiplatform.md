## 1. 阶段结论

- `2026-04-06`，`v26.3.2-release` 已完成首版真实发布，第一开发阶段正式结束
- `2026-04-07`，官方规划口径正式切换到第二开发阶段：`社区深化与多端化`
- 发布治理、验证留痕、部署回滚与宿主运行基线继续保留，但从“当前产品主线”降级为“并行维护线”

## 2. 本阶段总目标

第二开发阶段不再回答“能不能发第一版”，而是回答三件事：

1. 社区是否开始具备持续使用与持续互动的产品深度
2. 用户是否可以在移动场景下自然访问，而不是被迫进入桌面窗口系统
3. 仓库是否为 Flutter 客户端建立清晰、可复用、可迭代的第二入口

## 3. 当前判断依据

- 论坛、关系链、聊天、通知、商城、个人能力都已有一版基础，不需要重开一条从零到一主线
- `radish.client` 当前仍是明显的桌面 / WebOS 优先架构，移动端不能简单理解为“把窗口缩小到手机里”
- 仓库当前没有真正落地的 `MobileShell` 实现
- 仓库当前也没有 Flutter 工程，因此 Flutter 必须作为独立产品线规划，而不是临时从 Web 侧分叉
- WebOS 的最佳角色是“桌面工作台”，而不是公开社区、移动端与原生客户端的唯一入口

## 4. 优先级配比

- `60%`：社区深化
- `20%`：移动 Web 形态
- `15%`：Flutter 客户端基础
- `5%`：维护线 / 文档 / 收口

### Phase 2-1：社区深化第一批 `已完成当前批次目标`

目标：先把社区从“能用”推进到“愿意多停留、多互动”。

当前固定执行顺序：

1. **论坛轻回应墙 Phase 1**
   - 在帖子正文下方新增独立的轻互动区域
   - 使用“头像 + 短内容”的滚动胶囊展示，不侵入标准评论树
   - 独立建模，保留独立限流、审核、软删除和最小治理边界
2. **帖子详情页轻互动层收口**
   - 明确正文、轻回应墙、正式评论区三层结构
   - 统一帖子详情页的互动密度与信息节奏，避免继续在评论树里堆入口
3. **论坛相关回流链路补齐**
   - 为轻回应、通知跳转、个人内容回看预留最小联动能力
   - 目标是让社区互动能回流，而不是只留在一次性页面行为里
   - 当前建议优先从“个人内容回看 -> 跳回帖子详情”切入，再评估是否补齐轻回应通知跳转

当前进度：

- 截至 `2026-04-08`，论坛轻回应墙 `Phase 1` 已完成基础链路落地，不再停留在专题设计阶段
- 当前已完成独立模型、独立服务 / 控制器、帖子详情页插入位、举报目标扩展、配置化治理与基础展示交互
- 当前帖子详情页结构已固定为“正文 -> 轻回应墙 -> 评论区”
- 治理边界已按配置集中收口，默认支持纯文本与系统默认 Unicode emoji，不支持贴纸、表情包、附件、Markdown 与 `@提及`
- 当前主线的下一落点不再是“继续证明轻回应墙能否成立”，而是“联调收口 + 最小回流链路补齐”
- 轻回应相关的“个人内容回看 -> 跳回帖子详情”最小闭环已落地
- 论坛现有帖子 / 评论类通知当前已具备“从通知中心跳回帖子详情”的最小回跳能力
- 论坛评论精确定位最小闭环已落地：通知中心、个人主页评论记录等带 `commentId` 的 forum 打开入口，当前已可在评论加载完成后自动定位到目标评论并给出一次性高亮提示
- 轻回应专属通知最小闭环已落地：他人在你的帖子下发布轻回应时，当前会给帖子作者发送一条最小论坛通知，并支持从通知中心跳回帖子详情
- 截至 `2026-04-08`，本批次当前已达到可收口状态，后续默认转入稳定维护，不再继续把论坛联调尾项作为主线

### Phase 2-2：移动 Web 形态 `当前主线`

目标：让移动端用户进入的是“社区产品形态”，不是桌面窗口系统的缩小版。

边界：

- 优先做公开内容浏览和高频轻互动链路
- 优先让帖子列表、帖子详情、文档阅读、个人公开页具备自然移动访问形态
- 登录后能力按价值逐步接入，不一次性把所有桌面 App 全搬到手机里
- 默认采用增量迁移：先增加公开直达路由，再决定是否调整根入口与 `/desktop`

首批冻结范围：

- 只从 forum 开始，不同时铺开文档、个人公开页和更多社区模块
- 首批只做 forum 公开列表与帖子详情的移动访问形态
- 详情页首批承载正文、轻回应墙与评论阅读，不同时迁移完整创作器、聊天或完整通知中心
- 首批不调整根入口，不把 `/` 直接切到公开社区首页

当前推进口径补充：

- forum 公开列表与帖子详情首批当前已达到可收口状态，后续转入稳定维护
- 文档阅读首批当前已完成 `/docs` 与 `/docs/:slug` 公开阅读入口落地，并在一轮人工验收后完成返回目录状态保持、正文内链与旧 `__documents__` 链接兼容等问题收口
- 文档阅读首批当前仍只做目录浏览与正文阅读，不同时迁移桌面文档应用里的编辑、版本历史、回收站或发布治理能力
- 个人公开页首批当前已完成 `/u/:id` 公开阅读链路收口，继续保持“公开资料 + 公开帖子 / 评论”边界
- `Phase 2-2` 当前下一批转到公开榜单首批，继续沿公开内容壳层补公开可读入口，而不是回头扩桌面窗口治理

明确要求：

- 移动 Web 应该有独立的移动壳层或移动路由模式
- 不直接复用完整 `WindowManager` 交互模型
- 桌面端与移动端允许共享数据、主题和组件能力，但不强求 UI 结构一致
- 公开内容壳层与桌面工作台壳层的边界，以 [前端多壳层策略](/frontend/shell-strategy) 为准

### Phase 2-3：Flutter 客户端 MVP `已规划，未启动`

目标：建立第二原生入口，优先覆盖 Android，再扩到 Windows / Linux。

首批建议范围：

- 登录 / 会话恢复
- 论坛列表 / 帖子详情 / 轻互动
- 通知中心基础能力
- 个人主页最小浏览链路

执行原则：

- Flutter 不直接复刻 WebOS 桌面
- Flutter 优先复用现有 API、认证契约和主题语义，而不是复刻前端实现细节
- Windows / Linux 客户端在 Android MVP 跑通后再进入同批次扩展
