## 阶段结论

- `2026-04-06`，`v26.3.2-release` 完成首版真实发布，第一开发阶段结束
- 第二开发阶段的目标不是继续证明“能不能发第一版”，而是推进社区深度、多端访问和产品分工
- 当前主线已从 Android MVP 与 Tauri 验证批次，收束为“多端路线收口后的产品功能开发推进”
- 发布治理、验证留痕、部署回滚与宿主运行基线继续保留，但只作为并行维护线

## 本阶段目标

1. 社区开始具备持续使用与持续互动的产品深度
2. 用户可以在移动场景下自然访问公开内容和高价值链路
3. 仓库形成清晰多端分工：Web 公开阅读、Flutter 移动安装包、Tauri + WebOS 桌面安装包

## 当前判断依据

- 论坛、关系链、聊天、通知、商城、个人能力都已有一版基础，不需要重开从零到一主线
- `radish.client` 仍是桌面 / WebOS 优先架构，移动端不能理解为“把窗口缩小到手机”
- WebOS 的角色收束为桌面工作台，不再是公开社区、移动端与原生客户端的唯一入口
- Flutter 是 Android / iOS 移动安装包路线，Android MVP 已完成第一轮，iOS 后续单独评估
- Tauri 是 WebOS 桌面工作台的安装包壳，不是移动端替代方案，也不是原生 UI 重写路线
- 公开内容壳层面向 PC / 移动浏览器响应式阅读，不承担完整工作台治理

## 当前优先级

- `35%`：产品功能开发推进，优先 WebOS 桌面工作台与已登录复访、启动续接、高价值应用回流
- `20%`：Android MVP 产品化深化，包括测试对象、反馈回收、已知问题列表、版本说明、release 前验收与分发留痕
- `20%`：公开内容壳层稳定维护，处理真实阅读、响应式、来源返回与分享问题
- `10%`：社区深化维护，只处理轻回应、通知回流、论坛阅读链路里的真实问题
- `10%`：维护线、验证基线、文档同步、发布 / 回滚 / 可观测性
- `5%`：Tauri + WebOS 桌面安装包维护观察

## 当前不做

- 继续扩张论坛轻回应、通知尾项或公开壳层页面细节
- 未经重新评估追加 Flutter 第 `24` 批及以后低增益体验微调
- 启动 Flutter iOS 产品化工程
- 把 Windows / macOS / Linux 作为 Flutter 默认扩平台方向
- 继续扩大 Capacitor Android 登录态或移动端产品化能力
- 把 Tauri 当作原生 UI 重写路线
- 做“移动版 WebOS”
- 完整 `PWA / Service Worker / 离线能力`
- 完整 Playwright / E2E 平台
- 完整可观测性平台、Tracing / Metrics 大阶段
