# 发布后维护与功能完成线

> 本页定义 `v26.7.1.1204-release` 完成生产部署之后 Radish 的常态开发节奏。
>
> 目标是让个人开发者可以持续发布、处理真实反馈并完成既有功能，不再把普通迭代重新积累成长期候选阶段。

## 一、进入条件

- `dev -> master` PR 已通过 required checks 并合并。
- `master` 已回灌 `dev`。
- 规范 `v26.7.1.1204-release` tag、五个正式镜像和部署后复核已经完成。
- 没有未处置的 `P0/P1` 发布阻断。

受控试用不再作为 tag 前置门禁。被动收到的明确生产故障和可追溯用户反馈直接进入维护或功能 backlog；主动生产使用数据采集不属于常态开发任务。

## 二、常态开发节奏

发布后同时维护两条线，但个人开发时只保持一个主要功能专题在进行：

1. **长期维护线**：处理用户反馈、bug、安全、依赖、迁移、部署和兼容问题。
2. **功能完成线**：按业务域完成已经存在但语义、状态或用户动作尚不完整的功能。

执行规则：

- `P0/P1` 可中断当前功能专题；P2/P3 进入同类问题批次。
- 功能专题先确认现有专题文档、数据模型、接口和页面边界，再实施代码。
- 每个功能专题完成后执行与风险匹配的测试并正常发布，不等待积累成超大型阶段。
- 已经收到的用户反馈与真实数据优先于继续做内部全站扫描，但不为取得这些材料暂停功能开发或主动建设生产采集专题。
- 新功能必须说明它与内容、关系、贡献、治理或复访主轴的关系。

## 三、长期维护线

默认覆盖：

- 用户反馈的页面错误、业务异常、失败状态和体验问题。
- 身份、权限、资产、订单、通知、可靠任务和数据库迁移问题。
- npm / NuGet 依赖安全、镜像漏洞、证书、域名和部署配置。
- 公开 head、动态 sitemap、生产域名与搜索分享质量。
- WebOS `/desktop` 阻断级兼容和 Flutter 既有 MVP 的条件维护。

维护批次应保留根因、影响面、修复范围和验证结果；不通过层层 fallback 隐藏契约错误，也不顺手扩大为无关重构。

首个真实生产 UX 维护项为首次管理员初始化统一入口门禁：

- 当前公开入口与 Workbench 未统一经过 `BootstrapGate`，私域入口各自包裹，因此首次部署会在点击“聊天”等私域入口后才显示管理员初始化页。
- 后端管理员存在性、创建事务、限流与并发保护保持不变；当前生产管理员已经创建，本项不构成运行阻断。
- 修复时把门禁上移到 `BrowserAppRouter` 统一编排，删除入口级重复包裹，并明确公开、Workbench、私域与 OIDC 回调行为。
- 补路由层契约测试，保证以后新增顶层入口不会再次漏接；该批次不重写已发布 tag，不触碰生产用户、角色或 bootstrap state。

首个发布后工程治理项为镜像漏洞门禁分层，目标是在保持真实高风险阻断的同时，避免个人开发节奏被无修复版本、不可达运行路径或上游漏洞库波动反复打断：

- `CRITICAL` 默认继续阻断，不因暂时无修复版本自动放行。
- 存在修复版本的 `HIGH / CRITICAL` 继续阻断，优先升级基础镜像、依赖或移除不必要运行资产。
- 无修复版本的 `HIGH` 改为生成可追溯报告和维护项，默认不阻断；若命中真实可达高风险路径，人工提升为阻断。
- 例外必须精确到 CVE，记录基础镜像、可达性判断、接受原因和复核期限；禁止全局关闭扫描或长期笼统忽略。
- 门禁实现应保留最终运行镜像扫描、SBOM、provenance 和固定 tag，不回退本次已完成的运行镜像精简。

该治理在 `v26.7.1.1204-release` 完成部署后进入维护批次，不反向修改已发布工作流结果。

## 四、功能完成线顺位

### F1. 商城商品效力与权益履约（已完成）

> 当前实施设计见 [商城商品效力与权益履约专题](/features/shop-product-effect-entitlement-fulfillment)。

当前购买、扣款、订单、背包与只读展示已经存在，下一步完成商品购买后的真实效力：

- 明确消耗品、权益商品和普通商品的状态模型。
- 完成订单成功后的发放、激活、使用、过期、撤销与审计语义。
- 复核改名卡、经验卡、双倍经验、帖子置顶、帖子高亮等现有或遗留商品效果。
- 明确重复购买、并发使用、过期、失败补偿和来源订单契约。
- 让背包、订单详情、真实生效页面和 Console 排障使用同一状态真相源。
- 对资产、权益和状态变化补幂等、事务、权限与高风险集成测试。

`F1-A -> F1-E` 已于 2026-07-14 完成实现与批次级验收，具体状态、迁移和证据以专题文档及验收记录为准；历史 Badge / Title 商品未自动上架。

### F2. 主题系统完成（已完成）

2026-07-14 已在 `default / guofeng` 基础上完成暗夜 / 樱花权益主题和真实用户状态：

- 用户当前主题与已拥有主题权益的关系。
- 登录后跨设备持久化，匿名态继续使用本地偏好。
- 主题商品的购买、激活、切换、失效和回退。
- client、`@radish/ui` 与 Console 新增页面的语义 token 一致性。
- 首屏主题初始化、可读性和切换稳定性。

正式 Web、共享 UI、Theme 服务端资源契约与 PC / mobile 页面族已完成验收；历史主题商品保持下架。详细实现和证据见[前端主题与 i18n 实施说明](/frontend/theme-i18n-implementation)与[F2 主题系统专题验收记录](/records/f2-theme-system-stage-acceptance-2026-07-14)。本专题未重做全站视觉，也未建立在线主题设计器。

### F3. i18n 完成度治理（已完成）

> 当前审计、首批实现状态与后续批次入口见 [F3 i18n 完成度治理实施说明](/frontend/i18n-completion-governance)。

现有 client 主路径资源化继续向正式 Web 边缘路径扩展：

- client、Console 与 Auth 的语言选择和持久化。
- 用户可见硬编码文案、错误码翻译和失败状态。
- 日期、数字、金额、相对时间和复数表达。
- 中英文布局、表格、长文本与移动视口稳定性。
- 翻译键按业务域维护，避免继续扩大单文件和重复文案。

`F3-A / F3-B / F3-C1-C9` 已完成实现与静态回归；2026-07-17 的 F3-D 已通过 Public / Private / Auth、Console 认证 / 无权限与管理员受权内页的 `zh / en × PC / mobile` 代表运行态矩阵。Dashboard、用户、治理、订单、应用、个人资料、Hangfire 与 not-found 的双语布局、功能抽屉和格式化证据齐备，F3 至此关闭。

用户内容与固定文档正文不自动翻译；本专题不建设翻译管理平台。

### F4. 既有功能持续完成（当前）

从 Docs、聊天、通知、治理、宠物或其他既有业务域中选择一个完整功能专题。选题优先依据当前产品定位、已有能力缺口、长期系统价值和可以稳定定义的边界，不等待主动生产数据采集。

专题进入代码前必须明确用户路径、长期价值、数据模型、接口 / 页面边界、权限与失败恢复、停止线和验证口径。个人开发时只保持一个主要专题在进行，完成设计、实现和成组验收后再切换下一项。

已经被动收到的明确生产故障、可追溯用户反馈或 `P0/P1` 仍直接进入维护线；处理具体问题不扩展为主动全链路采集，也不改变当前功能专题的长期边界。

### 最终收尾：生产使用证据采集（冻结）

2026-07-18 已完成首轮仓库证据盘点，并形成生产六链路采集说明与模板。该资产保留用于项目最终完成体复核，不是当前 F4 功能选题前置，也不按固定日期或观察周期自动重启。

只有同时满足以下条件，才能重新开启主动采集：

1. 计划内主线功能和既有功能完成专题已经全部关闭。
2. 当前没有明确功能、维护、设计补全或可执行技术债。
3. 产品进入最终完成体复核，确实需要生产证据作为最终判断材料。
4. 用户明确确认重新开启。

详细历史事实与冻结资产见 [F4-A 首批真实使用证据整理与反馈归因记录](/records/f4-a-first-real-usage-evidence-attribution-2026-07-18)、[真实使用证据采集说明](/guide/real-usage-evidence-collection)和[生产证据采集模板](/records/f4-a-production-evidence-capture-template)。

## 五、发布节奏

- 一个业务专题或一组成因相同的修复形成一个可审阅批次。
- 开发中执行定向 lint、type-check、test、build、repo hygiene 和 `git diff --check`。
- `dev -> master` 前执行 Repo Quality 对齐验证；高风险身份、数据库和资产批次追加专题验证。
- 版本、PR 合并、tag、镜像和部署仍保持可追溯顺序。
- 生产稳定期对高风险链路采用更小的变更范围和更明确的回滚目标。

## 六、停止线

- 产品仍有明确功能、维护或设计任务时，不主动建议、排期或执行生产使用数据采集，也不以缺少生产证据阻断功能开发。
- 不解冻 Tauri，不恢复 Flutter 与 Web 并行追赶。
- 不把完整 E2E、全仓 strict、大文件全量拆分或历史样式清零设为普通功能前置。
- 不在没有用户价值证据时启动推荐、ActivityPub、开放平台或大而全可观测性。
- 不因发布后进入维护期而跳过资产、权限、迁移和安全边界。
