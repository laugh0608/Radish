## 背景判断

`P3-9` 已把访客公开访问、Flutter 登录移动用户和 Console 管理员排障三类真实使用路径组织成可验收批次，并完成 `dev -> master` 合并。当前决定不创建发布 tag、不等待镜像、不进入 M15 测试 / 生产部署流程。

这不代表项目进入生产稳定运营。Web、Flutter、PC/Tauri 与 Console 的功能完整度、UI 一致性和任务归属仍然不足，继续开发应先回答“哪些能力应该在哪个端上承接”，再进入具体实现。

## 阶段目标

1. **明确跨端任务归属**
   - 纯 Web：承接公开访问、低门槛阅读、分享、登录后轻量链路。
   - Flutter：承接移动原生的复访、互动、通知、商城、资产和个人资料任务。
   - PC/Tauri：后置评估，只增强纯 Web 体验，不恢复 WebOS 扩张路线。
   - Console：承接治理、排障、权限、内容审核、订单和资产流水管理。
2. **形成下一批开发候选矩阵**
   - 每个候选任务必须说明用户价值、端归属、当前缺口、实现风险、验证入口和不做范围。
   - 优先选择能改善真实使用路径、跨端一致性、登录恢复、返回语义、错误态或治理效率的任务。
3. **避免继续按零碎缺口扩张**
   - 不把 Console 页面微调、移动 Web 逐页打磨或 WebOS 历史能力补齐当作默认主线。
   - 不一次性启动完整移动商城、完整通知中心、完整资产中心或完整 PC 分发能力。

## P3-10-A 初步评估结论

当前代码和文档口径显示，端能力已经具备基础分工，但入口命名和任务归属仍不够稳定：

| 端 | 当前事实 | P3-10-A 归属判断 |
| --- | --- | --- |
| 纯 Web | 根路径 `/` 默认进入 `/discover`；公开路由覆盖 `/discover / forum / docs / leaderboard / shop / u/:id`；`/desktop` 保留工作台入口 | 继续承担公开访问、移动 / PC 浏览器阅读、分享、SEO、登录后轻量回流，不扩成完整工作台 |
| Flutter | 当前底部栏是 `发现 / 论坛 / 文档 / 榜单 / 我的`；壳层已接入通知、商城、订单、背包、钱包、经验、最近访问和个人资料 | 需要评估是否收敛为 `发现 / 消息 / 更多 / 我的`，把移动复访、通知回流、私域资产和常用功能入口整理清楚 |
| PC/Tauri | Tauri 运行时根路径仍进入 `/desktop`；长期策略是只增强纯 Web，不绑定 WebOS | 暂不做分发、签名、自动更新或原生 UI；只记录未来增强触发条件 |
| WebOS `/desktop` | 仍承接历史工作台、forum / shop 私有深链和登录回流 | 作为保留入口维护，不承接新增功能主线 |
| Console | 路由集中在仪表盘、应用、商品、订单、用户、角色、分类、标签、内容治理、胡萝卜、经验、配置和定时任务 | 继续作为治理后台，下一批只挑影响排障效率、权限一致性或跨对象跳转的问题 |

建议下一批优先从 `P3-10-B1` 开始：先把 Flutter 移动信息架构的目标结构、入口归属和测试面收拢，再决定是否进入实际导航重组。这样能直接回应当前 Web、Flutter、PC/Tauri 和 Console 功能 / UI 差距，同时不让 PC 或 WebOS 抢走当前主线。

## 候选任务排序规则

优先级从高到低：

1. 阻断真实用户连续完成任务的问题。
2. 跨端任务归属不清导致入口混乱、返回异常或登录恢复丢失的问题。
3. LongId / PublicId、公开链接、通知回流、订单 / 背包 / 流水等契约风险。
4. 管理员无法快速定位用户、订单、商品、流水或权限问题的治理效率缺口。
5. 明显影响可读性、可操作性或主路径信任感的 UI 断裂。

低优先级：

- 纯样式偏好微调。
- 历史 WebOS 能力补齐。
- Console 低频页面迁移。
- 不影响主路径的移动 Web 边角页面打磨。
