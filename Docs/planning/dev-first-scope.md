> 本页用于冻结 **首版 dev 的范围、完成标准与明确后置项**，避免“某条子线已收口”被误读为“整个产品已经达到首版 dev 条件”。

## 1. 定义口径

首版 `dev` 不按“所有细节都完成”定义，而按“所有纳入范围的核心功能主线至少完成 70%”定义。

这里的“主线完成”采用以下判断标准：

1. **用户任务可完整走通**
   - 优先看“从入口到结果”的完整链路，而不是是否把所有筛选、增强、扩展与后台治理都补齐。

2. **无阻塞主线的已知问题**
   - 允许存在非阻塞体验问题，但不允许带着会直接打断主流程的 `P0 / P1` 缺陷进入首版 `dev`。

3. **无明确结构性返工风险**
   - 可以接受克制实现，但不能把明显会在首版 `dev` 后立刻返工的结构性坑、脆弱契约或长期技术债带入首版边界。

4. **工程交付链首次跑通**
   - 首版 `dev` 不仅要“本地可用”，还要具备第一版可重复验证、可构建、可镜像化的工程基础。

## 2. 首版通过标准

首版 `dev` 进入“可发内部开发版”状态，至少满足以下条件：

1. 所有纳入首版范围的核心功能主线达到 `70%` 闭环标准。
2. 当前没有阻塞主线的已知 `P0 / P1` 问题。
3. 当前没有明确会阻塞后续继续开发的结构性技术坑。
4. 最小验证基线可重复执行，且结果稳定。
5. 第一次 `CI/CD` 成功跑通，至少完成首版宿主所需镜像的构建链。

## 4. 体验主线的首版收敛规则

### 4.1 国风视觉 / 主题切换

首版纳入，但必须 **收敛范围**：

- 先做 `radish.client`，不把 `radish.console` 一并拉成当前主线；
- 先统一视觉基线与主题变量，不先做多套复杂皮肤包；
- 先覆盖高频主链路页面，不把全站历史角落页面一次性纳入；
- 先保证“切换后稳定可用”，再做更高精度的视觉打磨。

### 4.2 i18n

首版纳入，但也必须 **收敛范围**：

- 先覆盖 `radish.client` 用户端主链路；
- 先处理高频用户可见文案，不强求一次性清空所有次级提示；
- 不把 `Docs/` 固定文档全文翻译作为首版前置门槛；
- `radish.console` 的全量国际化默认不纳入首版第一批。

## 5. 当前明确不纳入首版

以下事项默认 **不属于首版 `dev` 必须完成项**：

- `P3-ext` / `P4-ext` / `P5-ext`
- `Console-ext Phase 2+`
- `Reaction Phase 2`
- 文档搜索 / 分享入口 / 更重目录治理
- 商城定时任务、压测、完备测试
- 经验值防刷高级规则、完整后台治理
- 开放平台第三方接入 / SDK
- 邮件通知系统
- 完整 CI / 完整容器交付 / 全量 E2E 这类超出首轮现实边界的大而全工程目标

## 6. 当前执行顺序建议

1. 先冻结本页范围，不再让首版边界在会话中反复漂移。
2. 再按 [首版 dev 功能矩阵状态表](/planning/dev-first-status-matrix) 补齐“待补齐主线”与“待联调复核主线”。
3. 并行推进 `radish.client` 的国风视觉基线、主题切换与 i18n 主线。
4. 在主线接近完成后，继续稳定首轮最小 `CI/CD` 门禁，并补齐 Docker 镜像构建链。
5. 最后做一轮按首版范围组织的总回归，而不是继续扩张局部增强项。

## 7. 使用规则

- 以后讨论“首版 `dev` 还差什么”时，以本页为准，不再以某个子专题的收口状态替代整产品首版判断。
- 任何想纳入首版的新项，都必须先回答：
  - 它是否属于核心用户主线？
  - 它是否会阻塞首版 `dev`？
  - 它是否会显著拖慢当前主线？
- 若某项不满足以上条件，默认进入 [Backlog](/planning/backlog)。
