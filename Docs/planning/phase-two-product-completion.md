## 背景判断

第二阶段已经完成公开内容壳层、Flutter Android MVP 与 Tauri + WebOS 桌面安装包路线的首轮验证。当前不应继续按端侧验证或分发产品化材料作为默认主线，而应回到产品本身，补齐首版发布后遗留的真实功能坑、治理能力和体验断点。

当前文档中部分模块被标记为“已完成”，其含义主要是首版主链路完成；代码和专题文档仍保留了模拟实现、无动作入口、治理缺口和后置增强项。后续应重新按产品价值和风险排序，而不是把所有后置项永久搁置。

截至 `2026-05-13`，后端 + Console 治理、商城管理验收、安全治理尾项、构建 warning、萝卜坑工作流和 `P2-C5-A` Flutter 首批补强已完成多轮小批次收口；第二阶段收口评审窄范围筛查未发现新的必须立即闭环 `P0/P1` 阻断项。第二阶段归档判断已 Go，其余事项转维护或后置。

## 优先级原则

1. 优先补“用户能点到但不能真正完成”的功能，例如模拟 API、空操作按钮、`alert` 占位和跳转 TODO。
2. 优先补会影响资产、安全、权益、经验和治理可信度的能力。
3. WebOS 工作台承载“做事情”的完整流程，公开壳层承载“看内容”的低门槛访问，Flutter 承载移动高频路径。
4. 不因为某能力在首版矩阵里写过“已完成”，就忽略当前代码中的真实缺口。
5. 不默认扩完整 PWA、完整 E2E、完整可观测性平台、开放平台或新 BFF。
6. 当前收口期不再无边界扫低收益按钮、提示、非阻断 warning 或视觉微调。

## 第二阶段收口评审

详见：[第二阶段收口评审](/planning/phase-two-closure-review)。

### 收口目标

- 形成第二阶段已完成、转维护、明确后置和仍需闭环事项清单
- 判断下一主任务是否切向 Flutter 移动端高价值已登录链路
- 避免把 WebOS / Console / 安全治理的零碎尾项继续当成默认主线

### 允许继续闭环的事项

- 资产、安全、登录、购买、转账、权限授权或主路径中断的 `P0/P1` 问题
- WebOS / PC 工作台中成片工作流无法完成的真实缺口
- 会阻断准备合并、发布或本地验证基线的构建 / 测试问题

### 默认转维护或后置的事项

- 非阻断 chunk warning、视觉微调、提示文案细节
- 未触发真实问题的 Console 权限扩展、审计扩张或共享接口平台化
- 完整 PWA、完整 E2E、完整可观测性平台、开放平台、BFF 深化

## 近期批次建议

### P2-C1 萝卜坑工作流补全（已完成首轮）

目标：让萝卜坑从“可展示”推进到更可靠的日常资产工作台。

- 已接通支付密码设置 / 修改真实 API，移除模拟 `setTimeout` 和 `alert`
- 已让快捷操作真正切换到转移、记录、安全、统计
- 已将最近交易“查看全部”切到交易记录
- 已将复制流水号改为统一 toast 反馈
- 已将交易记录导出从占位提示补为 CSV 下载，并按当前筛选条件导出交易流水
- 安全日志后端能力仍作为后续治理观察项，不在本轮扩展新审计平台

### P2-C2 Console 与后台治理补洞（已完成首轮）

目标：把已经存在的后端能力和 Console 页面补成可用管理面。

- 用户详情、萝卜币流水、订单列表已从模拟 API 切到真实治理数据
- 设置页与个人资料页的保存动作已接入真实能力
- `StatisticsController` 已从占位 / 随机统计切换到服务层真实聚合，并补充统计仓储、服务、权限种子与定向测试
- 内容治理审核台已继续完成筛选效率、手动处置和当前治理状态前置，转入稳定维护
- 商城管理前端已回拉首轮：商品 / 订单真实详情、商品删除拦截和订单 / 商品 / 用户治理回跳已接通
- 商城人工验收、历史构建 warning、安全治理尾项与 Console 权限种子一致性已完成当前小批次，Console 后续转入稳定维护

### P2-C3 经验 / 等级治理

目标：补齐社区长期运营需要的经验治理能力。

- 管理员调整、冻结、解冻从框架推进到真实逻辑
- 每日统计更新和等级配置缓存补齐
- 每日上限从硬编码转为配置读取
- 已完成经验页用户概览、每日统计观察、经验流水回看、最小可解释异常规则、人工复核联动与治理留痕首轮收口
- 经验值防刷和异常检测继续保持“最小可解释规则”边界，不做大而全风控平台，也不扩自动处罚

### P2-C4 商城权益效果收口

目标：避免用户可购买但权益效果不完整。

- 已收回未开放权益 / 道具在商品种子、存量上架态、公开展示、购买、激活与使用中的入口
- 已补订单超时取消、库存回补、订单 `StockType` 快照、`DbMigrate` 自动补列与商城消费扣款链路
- 已完成购买后资产闭环首轮：背包 `Vo` 契约与前端渲染、商城购买支付口令输入与服务端校验、通知中心点击购买成功通知后的白屏 / 循环更新风险已收口
- 已统一 6 位数字支付口令与六码格输入组件，并将旧支付口令明确废弃：安全设置改为直接重置，商城购买与萝卜转移命中旧口令会统一提示升级
- 商城管理前端缺口已完成首轮回拉：商品 / 订单真实详情、商品删除拦截，以及订单 / 商品 / 用户三向治理回跳已补齐
- 商城管理人工验收已于 `2026-05-12` 收口，商品相关订单、详情长 ID、删除确认、订单跳用户 / 商品返回链路和商品删除保护均已确认稳定
- 权益过期检查仍按真实风险排序，不直接做完整定时任务平台

### P2-C5 移动端高价值已登录链路

目标：让 Flutter 移动端继续补真实使用价值，而不是转向分发材料。

- 最小通知入口与 forum detail 回流已完成首批补强
- 个人复访中的轻操作已完成首批补强
- 不复刻桌面工作台，不扩完整通知中心、发帖、完整评论提交、点赞、投票或编辑治理

## 当前不做

- Android / Tauri 的正式分发产品化材料优先级不高于产品功能补全
- 不继续围绕 WebOS 复访入口做低收益微调
- 不把所有公开壳层页面继续扩细节
- 不启动 Flutter iOS 产品化工程
- 不启动完整 PWA / 离线能力 / Web Push
- 不启动完整 Playwright / E2E 平台
- 不启动完整可观测性平台或开放平台

## 文档同步规则

- `current.md` 只保留当前主线、占比和下一顺位摘要。
- `development-plan.md` 保留路线图总览和端侧任务占比。
- `backlog.md` 只保留明确后置项；被回拉的事项应迁出 backlog 或在本页列入近期批次。
- 功能实现细节、命令级验证记录和批次流水写入开发日志或专题文档，不堆到入口页。
