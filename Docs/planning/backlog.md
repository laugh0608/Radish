## 近期待进入

### 1. 聊天室完善（P1）

**目标**
- 基于现有聊天室 P0 主链路继续做增强，而不是重开新系统
- 优先补齐 `@mention`、图片上传 UI、引用回复、断线补拉、成员头像来源等高感知能力
- 控制范围，按 2-4 个小点分批推进

**进入条件**
- 权限冻结观察期内未出现新的权限裂缝
- 不需要为聊天室增强重新扩张权限主链路
- 当前主链路编译、联调与基础文档口径保持稳定

### 2. 开源软件清单扩展（P2-ext）

**目标**
- 在欢迎 App 最小首版基础上，扩展为更完整的第三方依赖公告能力
- 视需要补齐完整许可证文本、全量依赖清单或构建时自动生成
- 保持与当前产品规模匹配，避免过早引入重型构建治理

**进入条件**
- 当前欢迎 App 静态清单已不能满足发布或合规需要
- 聊天室与权限主链路进入稳定维护
- 不与更大治理项形成资源冲突

### 3. 论坛问答 MVP（P4）

**目标**
- 在现有论坛主链上增加“提问帖 + 回答 + 采纳最佳答案”的最小闭环
- 保持“列表轻摘要、详情重交互”的当前论坛设计边界
- 不拆独立问答 App，不并行引入悬赏、奖励与复杂排序

**进入条件**
- 权限治理、聊天室 P1 与论坛投票 MVP 均已进入稳定维护
- 团队希望继续推进一个“更贴近社区内容主线”的小而快能力
- 不希望当前阶段过早引入抽奖系统的公平性、防刷与审计边界

**参考方案**
- [论坛问答 MVP 设计方案](/features/forum-qa-mvp)

### 4. 抽奖最小首版（后置候选）

**目标**
- 在明确奖励、抽奖券、开奖与审计边界后，再评估最小可用抽奖方案

**后置原因**
- 需要先确认公平性、防刷、开奖、公示与奖励发放口径
- 相比问答，当前与论坛主链的最小复用面更小，治理成本更高

### 5. Later：国风 UI / 主题切换 / i18n / 邮件通知系统

**定位**
- 统一归类为影响面更广、复杂度更高的大功能或大治理项
- 暂不作为当前阶段的近期待进入事项

**后置原因**
- 主题切换与国风 UI 会牵动全局视觉和组件体系
- i18n 涉及键值治理、缺失检测与全链路文案收口
- 邮件通知系统涉及模板、触发、投递与运维边界，复杂度较高

**重新启动条件**
- 当前小而快的功能补齐路径完成
- 权限 / 聊天室 / 新功能首版不再频繁返工
- 团队可接受一次较大范围的横向治理改造
