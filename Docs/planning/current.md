## 当前已确认事实

- `v26.3.2-release` 已于 `2026-04-06` 完成首版真实发布，第一开发阶段正式结束
- 论坛、关系链、聊天、通知、商城、个人能力都已有一版基础，第二阶段不需要重新从零定义社区
- 发布、部署、回滚、验证和留痕仍然重要，但当前已降为并行维护线
- `radish.client` 当前仍是桌面 / WebOS 优先架构，仓库里没有真正落地的 `MobileShell` 实现
- WebOS 当前继续保留，但角色已经明确收束为“桌面工作台”，不再承担所有场景唯一入口
- 仓库里当前没有 Flutter 工程；Android / Windows / Linux 客户端需要作为独立产品线建设
- 论坛轻回应墙 `Phase 1` 当前已落下独立模型、独立接口、帖子详情页插入位、举报接入、配置化治理边界与首版前端展示闭环
- 论坛轻回应墙当前已不再是“设计是否成立”的问题，而是“基础链路已落地，是否能形成回流与复访闭环”的问题
- 轻回应相关的“个人内容回看 -> 跳回帖子详情”最小联动能力已落地到个人主页“我的轻回应”入口
- 轻回应相关的通知跳转当前仍未形成正式闭环，仍属于本批次应补齐项

## 当前批次目标

1. **多壳层边界冻结**
   - 正式采用“公开内容壳层 / 桌面工作台壳层 / Flutter 客户端壳层”分工
   - 第二阶段前半程默认采用增量迁移，不立即搬动根入口
   - 以后涉及移动端、公开路由或 Flutter 的规划，统一以 [前端多壳层策略](/frontend/shell-strategy) 为准
2. **论坛轻回应墙 Phase 1**
   - 在帖子正文下方建立独立轻互动区
   - 保持与标准评论区、Reaction、问答 / 投票 / 抽奖帖子兼容
   - 首版即纳入最小治理边界：限流、软删除、最小审核 / 举报接口预留
   - 当前基础链路已落地，重点转向交互打磨、真实使用反馈修正与最小验收收口
3. **帖子详情互动层收口**
   - 重新稳定“正文 -> 轻回应墙 -> 评论区”的结构顺序
   - 把轻互动从评论树里剥离，减少后续功能继续往树上堆叠
   - 当前结构顺序已固定，后续重点从布局收口转向回流链路收口
4. **论坛相关回流链路补齐**
   - 为轻回应、通知跳转、个人内容回看预留最小联动能力
   - 当前建议优先从“个人内容回看 -> 跳回帖子详情”切入，而不是立即扩展为完整通知体系改造
5. **为下一阶段多端化预留边界**
   - 轻回应墙组件和数据模型从一开始就预留移动端降级形态
   - 不把新能力写死在桌面窗口交互里
