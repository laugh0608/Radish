## 背景

当前文档系统已经形成稳定分层：

- 固定文档来自仓库 `Docs/`，运行时只读展示。
- 在线文档存储在数据库中，由正式 Web 作者入口创建 / 编辑，由 Console 治理发布、权限、导入导出和版本回滚。
- Markdown 导入导出定位为迁移桥梁，导入后生成在线文档与首个版本快照。

后续如果引入“本地文件转换的文档”“本地文件镜像文档”或“重新从本地文件覆盖在线文档”，需要先明确本地文件与数据库文档之间谁是写入权威，否则会出现在线编辑、本地文件更新、导入覆盖和版本回滚互相覆盖的问题。

## 核心结论

后续治理应坚持 **单一写入权威**：

- `BuiltIn / LocalMirror`：本地文件是权威，在线端只读展示和观察，不允许正文编辑、删除、发布状态覆盖、权限策略覆盖或回滚写入。
- `Online / ImportedSnapshot`：数据库是权威，本地文件只是导入材料或来源记录，后续在线编辑不回写原文件。
- “重新从本地文件导入”必须是显式治理动作，不能后台静默覆盖在线正文。

不建议做本地文件与在线文档的双向自动同步。Markdown 正文之外还涉及标题、slug、目录、附件、权限、发布状态、版本回滚和删除态，自动合并会让文档系统边界变得不可维护。

## 来源类型建议

后续可把当前 `SourceType` 语义收敛为更明确的来源类型：

| 建议类型 | 写入权威 | 在线正文编辑 | 说明 |
| --- | --- | --- | --- |
| `BuiltIn` | 仓库 `Docs/` | 否 | 现有固定文档，启动同步生成运行时索引。 |
| `LocalMirror` | 外部或本地文件源 | 否 | 后续若支持用户选择本地目录 / 文件镜像，应按只读镜像处理。 |
| `Online` | 数据库 | 是 | 正式 Web 作者入口新建的在线文档。 |
| `ImportedSnapshot` | 数据库 | 是 | 从 Markdown 文件一次性导入后的在线副本，不回写源文件。 |
| `Rollback` | 数据库 | 是 | 版本回滚产生的新版本来源标记，不改变文档写入权威。 |

当前实现中的 `Custom / Imported / BuiltIn` 可以先保持运行，未来进入专题时再评估是否迁移命名。若只改展示文案，可以先在前端把内部值映射成用户可读来源，不急于改数据库字段。

## 冲突策略

### 在线编辑冲突

在线正文保存应使用乐观并发：

- 详情接口返回当前 `Version`。
- 更新请求提交 `ExpectedVersion`。
- 服务端只在 `ExpectedVersion == 当前 Version` 时保存。
- 版本不一致返回 `409 Conflict`，提示用户刷新、查看差异或另存为新文档。

这项能力适用于 `Online / ImportedSnapshot`，不适用于只读来源。

### 本地源重新导入

从本地文件重新导入已有在线文档时，应作为治理动作处理：

- 需要显式选择目标文档和导入模式。
- 默认生成新版本，不静默覆盖。
- 必须提交 `ExpectedVersion`，避免覆盖在线端已经发生的编辑。
- 冲突时返回 `409 Conflict`，由治理人员选择取消、重新读取后导入或导入为新文档。

### 只读来源保护

`BuiltIn / LocalMirror` 的只读保护必须在服务层生效，前端禁用按钮只能作为体验优化：

- 正文编辑、删除、恢复、发布 / 下架、归档、权限策略、导入覆盖和版本回滚均应被拒绝。
- 提示应指向真实修改入口，例如“请修改源文件后重新同步”。

## 后续实施停止线

首批若回拉该专题，只建议做到：

1. 明确来源类型和写入权威文档口径。
2. 补在线编辑 `ExpectedVersion` 乐观并发。
3. 把本地文件一次性导入明确命名为“导入副本”。
4. 为只读来源补统一 `VoIsEditable / VoEditPolicy` 之类的返回字段，减少前端按 `SourceType` 猜测。

暂不纳入：

- 双向同步。
- 自动三方合并。
- 本地目录监听。
- WebDAV / Git / 云盘同步。
- 多人协同编辑。
- 富文本块级冲突合并。

## 回拉条件

只有出现以下情况之一时，才建议从 backlog 回拉：

- 需要支持用户把本地 Markdown 文件长期挂载为只读文档源。
- 需要允许 Console 重新从文件导入覆盖已有在线文档。
- 线上使用中出现在线编辑覆盖、导入覆盖或版本回滚边界不清的问题。
- 普通用户作者态开放前，需要先补 `ExpectedVersion` 防止并发覆盖。
