# P3-12-E 正式产品成熟度与质量硬化

> 状态：`P3-12-D67 已立项；P3-12-E1 成熟度矩阵与差距审计已完成；P3-12-E2/E3-A、E3-B、E3-C、E4-A、E4-B 与 E5-A 首批实现已完成；原正式版发布候选顺延到 P3-12-F`
>
> 启动日期：2026-07-05（Asia/Shanghai）
>
> 入口：以 [当前进行中](/planning/current) 为准。本专题承接 [P3-12 Web 完全化与 WebOS 收束](/planning/p3-12-web-completion-webos-retirement) 中 D61-D66 的 UI 与运行态补验结果，不直接创建 tag，不进入 M15 测试或生产部署。

## 背景

`P3-12-D61-D66` 已完成 Public Web、Private / Author 和 Console 当前发布前页面族首批实现、启动无关验证和运行态补验。D66 证明当前代表页面未发现新的阻断级运行态问题，但这仍不足以证明产品已经达到正式发布质量。

正式产品判断不能只看“无阻断”。还需要系统确认：

- 功能是否覆盖真实用户任务，而不是只有路由和页面。
- 权限、安全、写入可靠性和资产 / 订单 / Console 高风险动作是否足够稳。
- 产品路径是否容易理解，失败 / 空态 / 成功反馈是否像正式产品。
- PC / mobile 交互是否顺畅，尤其 Console 是否具备可接受的移动 Web 治理视图。
- 性能、刷新恢复、登录过期、慢接口和大列表是否能承受真实使用。
- 剩余后置项是否被清楚分类，而不是被“当前不做”掩盖。

因此新增 `P3-12-E 正式产品成熟度与质量硬化`，把原 `P3-12-E 正式版发布候选` 顺延为 `P3-12-F`。

## 总目标

让 Radish 的 PC / mobile Web 正式版在进入发布候选前，达到可向真实用户交付的产品成熟度：

- 核心任务可完成，入口清晰，反馈完整。
- 安全、权限、写入可靠性和数据边界有明确证据。
- Public / Private / Author / Console 的 PC 与 mobile 视图满足真实操作要求。
- 发布前必须补的缺口进入 E 批次实现；非发布前必需项进入后置产品 / API 专题。
- E 完成后才能进入 `P3-12-F 正式版发布候选整备`。

## 社区产品主轴

Radish 的正式产品定位应优先按“社区”判断，而不是按功能清单判断。帖子、聊天和社交关系是主线，Docs、商城、资产、宠物、经验、通知、Workbench 和 Console 都应服务于社区内容生产、社区互动、关系沉淀、信任治理和持续复访。

因此 E 期成熟度判断需要加入一条产品主轴：

- 用户为什么来：看帖子、参与讨论、提问回答、加入聊天、关注人和内容。
- 用户为什么留下：收到回应、形成关系、看到自己的贡献被记录、获得经验 / 宠物 / 权益反馈。
- 用户为什么信任：举报可追踪、治理可解释、资产和账号安全、公开与私域边界清楚。
- 用户为什么再回来：Workbench 和通知能把未完成互动、待回复聊天、被关注内容、订单 / 权益变化和宠物 / 经验反馈组织成可行动队列。

其他能力的发布前判断必须回到这条主线：

- Docs 是社区知识和长内容沉淀，不应成为孤立文档站。
- 商城、资产、背包和经验是社区激励与身份表达，不应成为脱离内容互动的独立交易模块。
- 宠物是复访、陪伴和社区任务反馈，不应只是单页养成工具。
- Console 是社区治理和运营后台，不应只是通用管理表格集合。
- Workbench 是个人社区活动中心，不应只是入口导航地图。

## 检查维度

### 1. 功能完整度

覆盖真实用户任务，而不是仅检查页面可访问：

- 公开浏览：发现、论坛、Docs、商城、榜单、公开主页、分享和来源返回。
- 登录与身份：注册、登录、登出、过期恢复、DisplayHandle / PublicId 展示与搜索边界。
- 内容互动：发帖、评论、回答、轻回应、编辑、历史查看、登录回流和幂等反馈。
- Docs 作者：创建、编辑、保存、修订、公开阅读回跳和只读边界。
- 交易与资产：商品浏览、购买、订单、背包、资产流水和通知回流。
- 私域复访：工作台、我的内容、历史、附件、经验、通知、消息、圈子、宠物。
- Console：内容治理、经验治理、商业运营、文档治理、用户、角色权限和系统设置。

社区主线下，E1 必须把以下用户任务单独拉出矩阵：

- 发帖前后：发现入口、分类 / 标签、草稿、预览、上传、发布反馈、编辑和历史查看。
- 讨论参与：根评论、楼中回复、轻回应、问答回答、最佳 / 高亮内容、登录回流和重复提交反馈。
- 聊天参与：公共频道、私域会话、@我、未读、定位消息、断线恢复、移动输入和从通知回到具体消息。
- 社交关系：关注、粉丝、公开主页、个人圈子、关系动态、从内容到人的跳转和取消关注反馈。
- 社区复访：通知、Workbench、历史记录、我的内容、被回复 / 被提及 / 被关注后的下一步。
- 社区信任：举报、审核、封禁 / 恢复、内容回看、处理记录、申诉或解释边界。

### 1.1 社区路径专项审核

E1 需要从用户视角额外审计这些事情，并把结论分成 `必须修复 / 发布前建议 / 后置专题 / 已满足`：

- **内容流是否像社区首页**：发现页、论坛首页、分类页、搜索和榜单是否能支撑“看看有什么新内容 / 热内容 / 与我相关的内容”，而不是只列出技术上能访问的对象。
- **发帖是否像正式创作器**：编辑器是否有清晰草稿、预览、附件上传失败、分类缺失、发布中、发布失败、发布成功后的下一步。
- **评论是否能承载真实讨论**：楼层、回复对象、提及、轻回应、回答、神评 / 高亮和实时状态是否让用户理解讨论脉络。
- **聊天是否有产品定位**：明确首发聊天是公共频道、私聊、订单沟通还是社区实时讨论；如果完整聊天平台后置，需要首发范围内的未读、回流、断线、移动输入和消息定位足够可靠。
- **社交关系是否形成循环**：从帖子 / 评论 / 聊天进入个人主页，再到关注、圈子动态、后续通知和复访入口是否顺。
- **通知是否是行动队列**：被回复、被提及、关注动态、审核结果、订单 / 权益变化是否按任务归类，而不是只提供时间线。
- **激励是否服务社区贡献**：经验、宠物、金币、背包权益应明确与发帖、回复、回答、治理贡献或复访行为的关系；不能只展示数值。
- **治理是否有证据链**：审核人员能否从举报进入原内容、上下文、用户历史、处理动作和审计记录；用户能否理解内容被处理的原因。
- **Console 是否服务社区运营**：后台首页和移动视图应优先暴露待审内容、异常互动、举报队列、用户风险和运营指标，而不是仅展示通用导航。
- **低频模块是否不抢主线**：商城、资产、Docs、宠物等页面应能回到社区内容、关系或激励，不应把用户带入孤立流程。

### 1.2 发布前补充审计面

除社区主线外，E1 还需要从完整产品视角补充以下审计面，避免只按当前已经实现的页面做自证：

- **首次使用与冷启动**：未登录用户、刚注册用户、无内容用户、无订单用户、无好友 / 关注用户分别能否理解 Radish 的主要价值；发现页、Workbench、个人中心和 Console 首页是否有面向真实任务的引导。
- **信息架构与全局查找**：公开内容、用户、标签、Docs、商品、通知和 Console 管理对象是否有清楚的查找路径；NotFound、空搜索、无权限和失效链接是否给出可继续行动的入口。
- **内容生命周期**：帖子、评论、回答、Docs、附件和聊天消息从创建、编辑、删除、恢复、审核、历史查看到公开分享的边界是否一致；不支持回滚或恢复的地方是否有明确解释。
- **用户资料与隐私边界**：公开主页、DisplayHandle、头像、关注关系、经验、宠物、资产和附件是否区分公开可见、登录可见、本人可见和 Console 可见；不能让用户误以为私域数据被公开。
- **搜索、推荐与内容分发**：即便正式版不做复杂算法，也需要明确发现页、论坛列表、榜单、分类 / 标签和搜索的排序依据、空态和低内容量策略。
- **反骚扰与反滥用**：发帖、评论、聊天、关注、举报、轻回应、抽奖 / 投票和通知都需要检查频率限制、重复提交、恶意刷屏、举报滥用和被拉黑 / 屏蔽后的体验边界。
- **通知偏好与打扰控制**：通知中心不能只按技术事件展示；需要审计哪些通知必须触达、哪些可归档、哪些可关闭，以及设置页里暂不生效的通知策略是否应隐藏、实现或标明边界。
- **帮助、反馈与问题恢复**：用户遇到错误、上传失败、购买失败、登录过期、权限不足、链接失效或 Console 异常时，是否有错误编号、重试、返回、安全退出、反馈 / 举报或联系支持路径。
- **可观测性与事故定位**：前端 ErrorBoundary、统一 logger、接口错误、Hub 断连、Console 高风险动作和后端审计日志是否能把用户问题定位到请求、用户、目标对象和时间点。
- **运营配置与默认值**：系统设置、角色权限、种子账号、初始分类 / 标签、默认频道、商城商品、经验规则、宠物配置和通知规则是否有可交付的默认状态；不能依赖开发者手工知道哪些开关可用。
- **法务、合规与用户承诺**：正式发布前至少需要明确隐私、用户协议、内容规则、社区公约、虚拟资产 / 虚拟商品说明、退款或不退款边界和未成年人 / 敏感内容处理口径。
- **可访问性与基础兼容性**：键盘可达、焦点状态、ARIA、颜色对比、表单错误读屏、触控尺寸、移动输入、长文本换行、低端网络和主流浏览器兼容性需要纳入 E1，不只看截图是否好看。
- **测试从组件走向用户旅程**：现有单元、契约和静态验证不能替代正式用户任务验证；E1 需要列出注册登录、发帖评论、聊天回流、关注复访、举报审核、购买到账、资产变化、Docs 发布、Console 处理等旅程级用例。

### 2. 安全与权限

重点检查：

- 公开页只读边界，不泄露内部 ID、管理动作或登录态私域数据。
- Console 路由、菜单、按钮和 API 权限一致性。
- 写入动作的幂等、频率限制、失败反馈和重复提交边界。
- 订单、资产、背包、奖励、支付口令、转账等高风险能力的发布前覆盖状态。
- 敏感日志、错误栈、接口返回、前端状态和本地配置是否泄露敏感信息。
- 高风险治理动作是否有确认、审计、撤销 / 恢复或明确不可逆提示。
- 用户删除内容、注销账号、附件访问、公开资料、聊天记录、订单 / 资产流水和 Console 审计日志的留存边界必须明确；正式版不能只依赖默认软删除语义。
- 前端展示的后置配置、不可用按钮、占位设置和暂不支持状态必须有产品解释，不能让用户以为动作已经生效。

### 3. 产品力与真实路径

每条主路径都要按真实用户意图复核：

- 新用户能否理解产品能做什么。
- 登录后能否找到继续处理的内容。
- 操作成功后是否知道下一步。
- 空数据、低权限、加载失败和网络错误是否给出可行动反馈。
- 低频入口是否有合理承接，不把用户送回历史 WebOS。

### 4. UI 与人机交互

重点不再是单纯美化，而是可用性和认知负担：

- 信息层级、标题、指标、列表、详情、rail 和操作区是否符合任务优先级。
- PC 与 mobile 按钮、表单、抽屉、弹窗、表格滚动和返回路径是否稳定。
- 文案是否是用户能理解的动作说明，而不是内部实现描述。
- 触控尺寸、移动端密度、首屏信息量和底部导航遮挡是否达标。
- Console 当前只有响应式后台基础和成组移动 CSS 视口复核，尚未证明具备正式移动治理视图；E 期必须单独盘点并补齐或明确产品边界。

### 5. 可靠性与性能

覆盖：

- 启动、健康检查、基础 baseline、identity、host runtime。
- 慢接口、大列表、分页、筛选、刷新恢复、重复点击和登录过期。
- SignalR / Hub 停连恢复、通知 / 消息回流和评论实时状态。
- 上传、Markdown、图片、文档版本、详情抽屉和 Console 大表格。
- 关键页面 console error / warning、全局横向溢出、长加载和错误页。

### 6. 发布就绪

E 期结束前必须形成：

- 正式产品成熟度矩阵。
- 必修复清单与已修复证据。
- 接受后置清单与理由。
- PR 到 `master` 前验证入口。
- 发布候选、部署 smoke 和回滚材料的输入边界。
