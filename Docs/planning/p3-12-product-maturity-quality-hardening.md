# P3-12-E 正式产品成熟度与质量硬化

> 状态：`P3-12-D67 已立项；P3-12-E 作为发布候选前的大专题，先做功能完整度、安全、产品力、UI / 人机交互、可靠性和发布就绪硬化；原正式版发布候选顺延到 P3-12-F`
>
> 启动日期：2026-07-05（Asia/Shanghai）
>
> 入口：以 [当前进行中](/planning/current) 为准。本专题承接 [P3-12 Web 完全化与 WebOS 收束](/planning/p3-12-web-completion-webos-retirement) 中 D61-D66 的 UI 与运行态补验结果，不直接创建 tag，不进入 M15 测试或生产部署。

## 背景

`P3-12-D61-D66` 已完成 Public Web、Private / Author 和 Console 当前发布前页面族首批实现、启动无关验证和运行态补验。D66 证明当前代表页面未发现新的阻断级运行态问题，但这仍不足以证明产品已经达到正式发布质量。

正式产品判断不能只看“无阻断”。还需要系统确认：

- 功能是否覆盖真实用户任务，而不是只有路由和页面。
- 权限、安全、写入可靠性和资产 / 订单 / Console 高风险动作是否足够稳。
- 产品路径是否容易理解，失败 / 空态 / 成功反馈是否像正式产品。
- PC / mobile 交互是否顺畅，尤其 Console 是否具备可接受的移动 Web 治理视图。
- 性能、刷新恢复、登录过期、慢接口和大列表是否能承受真实使用。
- 剩余后置项是否被清楚分类，而不是被“当前不做”掩盖。

因此新增 `P3-12-E 正式产品成熟度与质量硬化`，把原 `P3-12-E 正式版发布候选` 顺延为 `P3-12-F`。

## 总目标

让 Radish 的 PC / mobile Web 正式版在进入发布候选前，达到可向真实用户交付的产品成熟度：

- 核心任务可完成，入口清晰，反馈完整。
- 安全、权限、写入可靠性和数据边界有明确证据。
- Public / Private / Author / Console 的 PC 与 mobile 视图满足真实操作要求。
- 发布前必须补的缺口进入 E 批次实现；非发布前必需项进入后置产品 / API 专题。
- E 完成后才能进入 `P3-12-F 正式版发布候选整备`。

## 检查维度

### 1. 功能完整度

覆盖真实用户任务，而不是仅检查页面可访问：

- 公开浏览：发现、论坛、Docs、商城、榜单、公开主页、分享和来源返回。
- 登录与身份：注册、登录、登出、过期恢复、DisplayHandle / PublicId 展示与搜索边界。
- 内容互动：发帖、评论、回答、轻回应、编辑、历史查看、登录回流和幂等反馈。
- Docs 作者：创建、编辑、保存、修订、公开阅读回跳和只读边界。
- 交易与资产：商品浏览、购买、订单、背包、资产流水和通知回流。
- 私域复访：工作台、我的内容、历史、附件、经验、通知、消息、圈子、宠物。
- Console：内容治理、经验治理、商业运营、文档治理、用户、角色权限和系统设置。

### 2. 安全与权限

重点检查：

- 公开页只读边界，不泄露内部 ID、管理动作或登录态私域数据。
- Console 路由、菜单、按钮和 API 权限一致性。
- 写入动作的幂等、频率限制、失败反馈和重复提交边界。
- 订单、资产、背包、奖励、支付口令、转账等高风险能力的发布前覆盖状态。
- 敏感日志、错误栈、接口返回、前端状态和本地配置是否泄露敏感信息。
- 高风险治理动作是否有确认、审计、撤销 / 恢复或明确不可逆提示。

### 3. 产品力与真实路径

每条主路径都要按真实用户意图复核：

- 新用户能否理解产品能做什么。
- 登录后能否找到继续处理的内容。
- 操作成功后是否知道下一步。
- 空数据、低权限、加载失败和网络错误是否给出可行动反馈。
- 低频入口是否有合理承接，不把用户送回历史 WebOS。

### 4. UI 与人机交互

重点不再是单纯美化，而是可用性和认知负担：

- 信息层级、标题、指标、列表、详情、rail 和操作区是否符合任务优先级。
- PC 与 mobile 按钮、表单、抽屉、弹窗、表格滚动和返回路径是否稳定。
- 文案是否是用户能理解的动作说明，而不是内部实现描述。
- 触控尺寸、移动端密度、首屏信息量和底部导航遮挡是否达标。
- Console 当前只有响应式后台基础和成组移动 CSS 视口复核，尚未证明具备正式移动治理视图；E 期必须单独盘点并补齐或明确产品边界。

### 5. 可靠性与性能

覆盖：

- 启动、健康检查、基础 baseline、identity、host runtime。
- 慢接口、大列表、分页、筛选、刷新恢复、重复点击和登录过期。
- SignalR / Hub 停连恢复、通知 / 消息回流和评论实时状态。
- 上传、Markdown、图片、文档版本、详情抽屉和 Console 大表格。
- 关键页面 console error / warning、全局横向溢出、长加载和错误页。

### 6. 发布就绪

E 期结束前必须形成：

- 正式产品成熟度矩阵。
- 必修复清单与已修复证据。
- 接受后置清单与理由。
- PR 到 `master` 前验证入口。
- 发布候选、部署 smoke 和回滚材料的输入边界。

## 首批执行顺序

1. `P3-12-E1 正式产品成熟度矩阵与差距审计`
   - 建立 Public / Private / Author / Console 的真实任务矩阵。
   - 把功能、产品力、交互、安全、可靠性、移动视图分成 `必须修复 / 发布前建议 / 后置专题 / 已满足`。
   - 首批重点审计 Console 移动 Web：确认是响应式可用、需要任务化移动视图，还是应明确限制为桌面优先后台。

2. `P3-12-E2 Console 移动治理视图与高风险后台交互硬化`
   - 若 E1 判定 Console 移动 Web 不达正式产品要求，优先补移动导航、筛选、表格、详情、抽屉、权限矩阵和低风险治理动作。
   - 不默认开发独立移动 Console App；是否需要独立 App 作为产品决策记录。

3. `P3-12-E3 公开与私域真实任务链路产品力补强`
   - 复核公开浏览、登录回流、内容互动、商城订单、背包、资产、通知消息、圈子宠物和 Docs 作者链路。
   - 修正用户不知道下一步、状态反馈不完整、入口断裂或移动端不顺的问题。

4. `P3-12-E4 安全、权限与写入可靠性复核`
   - 汇总权限、公开只读边界、写入幂等、频率限制、高风险动作、日志脱敏和资产 / 订单边界。
   - 必要时补定向测试、静态扫描或小范围代码治理。

5. `P3-12-E5 性能、稳定性与异常恢复补强`
   - 处理慢加载、大列表、重复请求、Hub 恢复、上传 / Markdown / 文档版本、Console 表格滚动与错误态。

6. `P3-12-E6 发布候选进入判断`
   - 汇总 E1-E5 结果，判断是否进入 `P3-12-F 正式版发布候选整备`。
   - 不满足时继续按矩阵推进，不创建 tag。

## 当前已知重点

- Console 有响应式 CSS 和移动视口复核记录，但尚未建立正式移动治理视图的产品验收标准；这不是 D66 可用性补验能完全覆盖的问题。
- Public Web `P15-P16` 公开聊天室 / 移动聊天回复流仍是 Public 小专题后置项，E1 需要重新确认它是否影响正式产品首发。
- 完整聊天平台、完整钱包、退款 / 售后、支付口令、资产风控、独立移动 Console、内部 Jobs 平台和新的治理 API 不能继续用笼统“后置”掩盖，E1 需要逐项给出发布前判断。

## 不做

- 不直接创建发布 tag。
- 不进入 M15 测试或生产部署。
- 不把 E 做成无限功能扩张；每个缺口必须归类为正式版必需、发布前建议或后置专题。
- 不为移动端后台直接复制完整桌面 Console；先确认真实移动治理任务和权限边界。
- 不绕过现有 API / 权限 / 数据库边界直接做高风险功能；如 E 期确认必须新增后端契约，先补专题方案和验证口径。

## 验证口径

E 期按风险分层执行：

- 普通文档 / 规划批次：repo hygiene、`git diff --check`。
- 前端页面或交互批次：对应 workspace build、`lint:changed`、repo hygiene、必要单测。
- 后端 / 权限 / 写入批次：`dotnet test`、相关定向测试、`validate:baseline` 或 `validate:ci`。
- 阶段级补验：用户当轮确认前后端已启动后，执行 `check:host-runtime -- --details --report` 与 Gateway PC / mobile 真实页面复核。
