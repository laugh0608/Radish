## 背景判断

`P3-10` 已完成 Web-first 信息架构、公开入口治理、用户身份语义、系统设置治理、写操作可靠性、论坛内容写入可靠性和 Flutter 首批受控承接。当前 `dev -> master` PR 暂缓，项目仍处于第三开发阶段的功能建设期，不进入 Phase 4 稳定运营，也不启动 M15 测试 / 生产部署流程。

下一批不应继续在 `P3-10-D` 上追加第五批链接扫尾，也不应直接启动大功能。当前最有价值的工作是把已完成的大批量治理沉淀为可恢复合并、可发布候选整备、可分层验收的工程面，同时只补齐真实复访链路中的小范围缺口。

## 阶段目标

1. **冻结 P3-10 既有代码范围**
   - `master..dev` 的 P3-10 完整提交范围已进入阶段收束记录。
   - 后续只处理验证命中、编译失败、明确阻断、文档口径冲突或发布候选整备必须补齐的问题。
   - 不继续把同类链接、文案、样式边角作为默认主动深挖项。
2. **形成发布候选整备矩阵**
   - 把 P3-10 的 Web、Flutter、后端治理、数据库迁移、身份语义和运行态复核拆成可验收条目。
   - 明确哪些验证可以复用，哪些必须在恢复 PR、发布候选或部署前重跑。
   - 输出剩余风险和后续回拉条件。
3. **筛选轻量复访补齐项**
   - 只围绕纯 Web 登录后轻量链路、Flutter 已有成熟 API 工作流和 Console 必要排障能力做小范围补齐。
   - 每个补齐项必须先说明用户场景、数据契约、端归属、验证入口和不做范围。
   - 不把轻量复访扩展成完整社交、完整移动能力套件或完整运营平台。

## 轻量复访候选边界

优先候选：

- 纯 Web 登录后：`/notifications`、`/messages`、`/me`、`/circle` 的入口返回、空态、目标跳转和登录恢复。
- 论坛复访：公开详情到登录后参与、作者编辑、评论 / 回答写入重试后的可感知状态。
- Flutter 受控承接：当前已有 API 和 Web 契约支撑的论坛发帖、评论、回答、作者编辑和单商品购买重试。
- Console 必要排障：只处理发布候选或真实验收阻断的权限、用户、订单、系统设置可见性问题。

暂不纳入：

- Flutter 转账、完整移动商城、完整通知中心、完整资产中心或完整创作器。
- 电子宠物经济消耗、商城物品、社区任务奖励、经验反向加成和 Console 配置 UI。
- 首页瀑布流二次大改、推荐算法、ActivityPub / WebFinger、完整个人圈子或联邦社交。
- 完整 Playwright / E2E 平台、完整可观测性平台、Redis 分布式锁平台或通用反垃圾系统。

## 验证口径

开发中优先使用静态与代码侧验证：

```bash
npm run validate:ci -- --report
npm run validate:baseline
npm run validate:identity
flutter analyze
flutter test
git diff --check
```

真实页面复核只在恢复 PR、发布候选整备或用户可见页面明显变化时执行。执行前必须先要求用户明确前后端已经启动，不沿用历史会话或上一轮启动状态。

## 开工顺序

1. 先确认当前 Git 范围和 P3-10 阶段收束记录仍可复用。
2. 建立 `P3-11-A` 发布候选验收矩阵，复用已有验证事实并标出需要重跑的项目。
3. 对轻量复访候选做只读代码审计，筛出不超过一组的高信号缺口。
4. 如果命中明确缺口，再按最小必要范围进入代码；否则保持 P3-10 可 PR 状态，等待恢复合并或发布候选决策。
