# Redis 与缓存治理专题

> 状态：未来后置专题。当前不作为 `P3-6` 主线启动；只有真实部署、真实流量、多实例运行或高并发业务暴露明确问题时，再从本专题回拉小批次实施。

## 当前结论

- Radish 当前缓存入口已经统一为 `AddCacheSetup()`，由 `Redis.Enable` 决定使用 Redis 或内存缓存。
- 测试 / 生产 Compose 默认启用 PostgreSQL 与 Redis；本地开发和本地容器验证默认继续使用 SQLite 与内存缓存。
- 业务代码应优先依赖 `ICaching` / `IDistributedCache`，不直接 new Redis 连接。
- 当前 Redis 更多承担“分布式缓存”角色；尚未系统性承担原子计数、分布式锁、SignalR Backplane、在线状态或队列语义。
- `IRedisBasketRepository` 只适合确需 Redis List / Queue / PubSub 等语义时使用，普通缓存仍走 `ICaching`。

## 已有使用面

- 公开内容增长：动态 sitemap、公开详情 head snapshot 使用缓存降低重复生成成本。
- 通知中心：未读数缓存与通知去重已通过缓存层实现，Redis 开启时会共享到 Redis。
- 上传限制：并发上传、分钟速率与日上传大小当前通过缓存键记录。
- 经验 / 等级：等级配置和计算器支持缓存，后续治理可以继续复用当前缓存入口。
- 表情包、快捷回复、评论高亮、分类 / 租户等读多写少场景已有局部缓存或可复用 `QueryWithCacheAsync`。

## 后续回拉条件

满足以下任一条件时，可以把本专题拆成小批次回拉：

- API / Auth / Gateway 进入多实例部署，SignalR 推送、在线状态或缓存一致性出现跨实例问题。
- 通知未读数、上传限流、商城下单、萝卜币转账等出现并发读改写冲突。
- 公开 sitemap / head snapshot、排行榜、热门商品、VIP 状态等读多写少接口出现稳定性能压力。
- 测试 / 生产 Redis 连接、持久化、容量或故障恢复暴露真实运维问题。
- 业务需要 Redis 原子计数、分布式锁、Sorted Set、队列或 PubSub，而 `ICaching` 的普通 key-value 语义已经不够。

## 候选批次

### RCG-1 基线盘点与观测

- 梳理全部缓存键、TTL、失效路径、数据来源和调用链。
- 区分本地内存缓存、部署态 Redis 和必须跨实例一致的业务状态。
- 增加必要日志或最小诊断接口，避免只凭业务现象猜测缓存命中和失效。

### RCG-2 通知与限流原子化

- 将通知未读数从普通读改写升级为 Redis 原子计数或明确的数据库真值 + 缓存失效策略。
- 上传并发、分钟次数、日上传大小改为原子计数 / 集合语义，避免多实例下丢更新。
- 保留内存缓存降级路径，但明确它只适合单实例开发或轻量部署。

### RCG-3 SignalR 与在线状态

- 多实例部署时启用 SignalR Redis Backplane，保证通知和聊天室分组推送跨实例一致。
- 聊天室在线状态从进程内 `ConcurrentDictionary` 升级为 Redis Set / Hash，并补连接过期清理策略。
- 明确断线、重连、实例宕机时在线状态的过期与恢复口径。

### RCG-4 高价值读模型缓存

- 排行榜、热门商品、VIP 状态、公开 head snapshot、动态 sitemap 等读多写少接口统一评估缓存键、TTL、失效和预热策略。
- 排行榜如出现高频访问，再评估 Redis Sorted Set；未出现压力前继续保留数据库实时查询或普通缓存。

### RCG-5 商城与萝卜币一致性

- 商城订单幂等键、库存并发保护、购买频率限制按真实并发风险决定是否引入 Redis 锁或数据库行锁组合。
- 萝卜币余额缓存、每日转账计数、配额限制、风控去重等只在交易闭环进入真实高频使用前回拉。
- 涉及资金 / 资产类能力时，数据库仍是最终真值；Redis 只能做限流、幂等、缓存或辅助锁。

## 不做事项

- 不因为 Redis 已部署就把所有缓存都改成 Redis 专用 API。
- 不在没有多实例或真实并发压力前引入复杂 Redis 高可用、哨兵、集群或自研缓存平台。
- 不让缓存掩盖数据库索引、查询模型或业务契约问题。
- 不把 `P3-6` 扩成 Redis 平台化建设；每次回拉都必须对应真实问题或明确业务批次。
