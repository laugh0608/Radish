## 1. 目的

当前最终启动评审没有通过，不是因为仓库内还缺设计，而是因为：

- **仓库外兼容边界还没有被事实关闭。**

这份清单的目标，就是把“还没确认清楚”拆成实际可核的对象和证据，避免后续继续停留在模糊表述。

## 2. 适用范围

本清单只关注 **仓库外** 或 **部署态** 的兼容边界，不重复盘点仓库内官方客户端和仓库内 `.http` 资产。

重点检查以下四类对象：

1. 已部署环境中的历史脚本 / 运维脚本 / 联调脚本
2. 外部网关、反向代理或中间层映射规则
3. 仓库外只消费 Access Token 的调用方
4. 已存在或准备上线的第三方 / 开放平台客户端

## 3. 核心检查问题

只要下面任一问题不能回答为“已确认无依赖”，当前就不能把 Phase 4 写成“允许启动”：

1. 是否仍有仓库外对象直接读取 `ClaimTypes.NameIdentifier`、`ClaimTypes.Role`、`TenantId`？
2. 是否仍有仓库外对象把 `jti` 当成“用户 Id 语义字段”消费？
3. 是否存在只消费 Access Token、不走 `userinfo` 的仓库外调用方？
4. 是否存在还未事实化入库的第三方 / 开放平台客户端？

## 4. 确认清单

| 检查项 | 要确认的问题 | 证据来源 | 通过标准 | 未通过后的处理 |
| --- | --- | --- | --- | --- |
| 测试部署环境脚本 | 测试环境是否还有独立于仓库的登录、换 Token、接口联调脚本 | 部署机脚本目录、运维记录、测试同学常用脚本清单 | 已确认不存在，或确认存在但不读取历史 Claim 字段 | 记录脚本名、用途、依赖字段，评审保持不通过 |
| 生产部署环境脚本 | 生产环境是否有巡检脚本、运维脚本、网关辅助脚本直接解析 Token | 服务器脚本目录、运维手册、部署仓库 | 已确认不存在直接 Claim 字段依赖 | 记录脚本名、调用点和依赖字段，评审保持不通过 |
| 外部反向代理 / 网关映射 | 是否存在基于旧字段名的 Header 映射、鉴权规则、灰度规则 | Nginx / Traefik / Caddy / 其他网关配置 | 已确认只依赖标准 OIDC / Bearer 链路，不读历史字段 | 记录配置位置与依赖字段，评审保持不通过 |
| 测试 / 生产环境中的独立客户端 | 是否存在仓库外前端、小工具、调试页或内部系统直接消费 Access Token | 实际部署入口清单、运维口径、项目方确认 | 已确认不存在，或仅消费标准字段 | 记录客户端名称、入口、依赖字段，评审保持不通过 |
| `userinfo` 之外的调用方 | 是否存在仓库外调用方从不走 `userinfo`，只自己解 Access Token | 联调记录、第三方接入说明、调用方确认 | 已确认无此类调用方，或其仅依赖标准字段 | 记录调用方与依赖字段，评审保持不通过 |
| 第三方 / 开放平台真实接入 | 开放平台规划中的 `{custom}` 客户端是否已经有真实接入方 | 后台客户端列表、部署环境配置、项目方确认 | 已确认尚未真实接入，或已接入但不依赖历史字段 | 记录客户端与依赖字段，评审保持不通过 |
| 旧 Token 存量窗口 | 部署环境中是否仍有必须兼容的旧 Token 存量窗口 | 发布节奏、会话时长、运维确认 | 已明确旧 Token 兼容窗口，且不阻塞首轮输出收缩 | 若无法明确，实施窗口需继续推迟 |

## 5. 建议的取证方式

### 5.1 部署环境盘点

至少对以下位置做一次人工确认：

- 测试部署机
- 生产部署机
- 运维使用的脚本目录
- 可能独立于主仓库的部署仓库 / 配置仓库

建议重点搜索：

- `NameIdentifier`
- `ClaimTypes.Role`
- `TenantId`
- `jti`
- `access_token`
- `userinfo`

### 5.2 网关与反向代理盘点

至少确认以下配置是否存在“按字段名做映射 / 判断”：

- `Nginx`
- `Traefik`
- `Caddy`
- 其他接入层或内网网关

如果这些层只做标准 Bearer 转发，不自行解析 JWT，则可直接记为“无历史字段依赖”。

### 5.3 第三方 / 开放平台盘点

重点不是文档里“规划允许接入”，而是：

- 当前是否已经真的有人接入；
- 接入后是否真的在消费 Token 字段；
- 消费的是标准字段还是历史字段。

如果后台客户端列表里只有官方客户端，就应明确写成“当前未发现真实第三方接入方”。

## 7. 退出条件

只有同时满足以下条件，本清单才算“已完成”：

1. 测试与生产环境都完成一次盘点。
2. 外部脚本、网关映射、独立客户端、第三方接入四类对象都有明确结论。
3. 若发现依赖项，已经写清“对象是谁、依赖什么字段、是否阻塞”。
4. 最终启动评审能够基于这份清单把结论改成“通过”或“继续不通过”，而不是继续写“待确认”。

## 8. 结果分支

### 8.1 如果确认通过

满足以下条件时，可以回到 [身份语义 Phase 4 最终启动评审](/guide/identity-claim-phase4-start-review) 重新评审：

- 未发现仓库外历史字段依赖；
- 或虽有仓库外对象存在，但已确认它们只依赖标准字段；
- 旧 Token 存量窗口已明确，不阻塞首轮输出收缩。

此时可把评审结论从“暂不启动”更新为“允许启动 Phase 4”。

### 8.2 如果仍无法确认

如果短期内仍拿不到这些外部事实，当前主线就不应继续挂在 Phase 4 上等待。

此时应按既有规划切换为：

[M14 宿主运行与最小可观测性基线（重定义）](/guide/m14-host-runtime-observability-baseline)

### 8.3 如果明确发现阻塞依赖

如果确认某些外部对象仍依赖：

- `ClaimTypes.NameIdentifier`
- `ClaimTypes.Role`
- `TenantId`
- `jti`

则当前应继续维持：

`最终启动评审未通过，继续暂停 Phase 4 实施`

并把这些对象纳入后续迁移或替换计划，而不是强行推进输出收缩。

> 补充说明：截至 `2026-04-03`，当前实际部署范围已按“确认通过”分支完成事实关闭，见：[身份语义 Phase 4 仓库外兼容边界首轮执行记录（仓库资产侧）](/guide/identity-claim-external-compat-first-pass) 与 [身份语义 Phase 4 最终启动评审](/guide/identity-claim-phase4-start-review)。

## 9. 当前结论

截至 `2026-04-02`，这份清单的价值不在于“已经证明可以启动”，而在于：

- 它把最后缺失的外部事实确认工作拆成了明确对象；
- 它给出了“确认通过 / 仍无法确认 / 明确发现阻塞”三种出口；
- 后续不需要再重复讨论“到底还差什么”，只需要按这份清单逐项关闭即可。
- 截至 `2026-04-03`，仓库资产侧的首轮排查记录也已补齐，见：[身份语义 Phase 4 仓库外兼容边界首轮执行记录（仓库资产侧）](/guide/identity-claim-external-compat-first-pass)。
