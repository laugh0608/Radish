# 文档篇幅治理

本文定义 `Docs/` 固定文档的篇幅、拆分和索引规则，目标是降低开发者、用户和 AI / Agent 的阅读成本。

## 基本原则

- 文档篇幅不照搬代码文件行数上限，应按读者目标、阅读预算和文档类型约束。
- 一个文档只服务一个主要读者和一个主要目的。
- 入口文档只放当前状态、下一步、常用入口和关键约束，不承载完整历史。
- 历史过程、命令输出、验证流水和长背景应拆入日志、记录、归档或专题文档。
- 长文档顶部应提供简短摘要和必要目录，让读者先定位当前结论。

## 阅读预算

- 入口类文档应控制在 `5` 分钟内可读完。
- 专题理解类文档应控制在 `15` 分钟内可读完。
- 超过阅读预算时，应拆出概览、专题细节、历史归档或验证记录。

## 行数建议

| 文档类型 | 典型位置 | 软上限 | 硬上限 |
| --- | --- | ---: | ---: |
| 入口 / 索引 | `Docs/index.md`、`Docs/README.md`、目录 `index.md`、`Docs/planning/current.md` | `300` 行 | `500` 行 |
| 当前规划 / 当前进度 | `Docs/development-plan.md`、`Docs/planning/current.md` | `300` 行 | `500` 行 |
| 架构 / 规范 / 设计 | `Docs/architecture/`、`Docs/frontend/`、`Docs/guide/` 中的稳定规则文档 | `600` 行 | `900` 行 |
| 专题深度文档 | `Docs/features/`、专题方案、复杂模块设计 | `800` 行 | `1200` 行 |
| 日志 / 记录 / 归档 | `Docs/changelog/`、`Docs/records/`、`Docs/planning/archive.md` | 按日期、阶段或批次拆分 | 超过 `1200` 行应优先拆分或归档 |

## 拆分触发

- 超过软上限时，应检查是否混入历史过程、验证流水、实现细节或多个主题。
- 超过硬上限时，除日志、记录和归档类文档外，不应继续追加内容，应先拆分或归档。
- 日志、记录和归档类文档超过 `1200` 行时，应优先按月份、周次、阶段或批次拆分。

## 内容归位

- 当前结论、执行入口和必要约束保留在入口文档。
- 稳定规则和契约写入架构、规范或设计文档。
- 复杂模块背景和方案写入专题文档。
- 历史批次写入 `Docs/changelog/`。
- 验证证据、验收记录和发布留痕写入 `Docs/records/`。
- 过期阶段性内容归入 `Docs/planning/archive.md` 或更细的归档页。

## 索引要求

- 拆分文档后，应同步更新 `Docs/README.md`、对应目录 `index.md` 或专题入口。
- 短入口应链接到长文档，而不是复制长文档正文。
- 同一主题出现多个文件时，应明确哪个文件是当前真相源，哪个文件是历史记录或参考资料。
