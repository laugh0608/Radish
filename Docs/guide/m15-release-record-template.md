# M15 发布记录模板

> 本页用于把一次 `M15` 发布涉及的 tag、镜像、测试结论、生产结论与回滚目标收成统一记录。

## 适用场景

以下场景建议至少补一份发布记录：

- 一次 `v*-test` 或 `v*-release` 规范 tag 已创建
- 一次测试部署或生产部署已完成最小复核
- 需要在周志、PR、Release 或运维记录中快速复用同一套发布事实
- 需要明确“如果本次发布有问题，优先回滚到哪个版本”

## 记录目标

本模板只回答最小发布留痕问题：

- 这次发布对应哪个 Git tag
- 这次发布对应哪些镜像 tag
- 测试部署是否通过
- 生产部署是否通过
- 如果需要回滚，优先回到哪个已知可用版本
- 本次仍有哪些已知风险或后置项

本模板不替代：

- `M14` 宿主维护记录
- `M14` 部署后最小复核记录
- 生产环境真实回滚记录

它只负责把这些结论收成一份“发布视角”的摘要。

## 最小字段

建议每份发布记录至少包含：

1. 记录信息
   - 日期、记录人、发布类型
2. 发布标识
   - Git tag
   - 镜像 tag
   - 对应分支 / 提交
3. 测试部署结论
   - 是否已部署
   - 是否已通过最小复核
   - 关联记录
4. 生产部署结论
   - 是否已部署
   - 是否已通过最小复核
   - 关联记录
5. 回滚目标
   - 测试环境优先回滚目标
   - 生产环境优先回滚目标
6. 已知风险 / 后置项

## 推荐模板

```md
## M15 发布记录（<tag>）

- 记录日期：YYYY-MM-DD
- 记录人：<name>
- 发布类型：测试发布 / 正式发布 / 测试锚点发布

### 发布标识

- Git tag：`<tag>`
- 对应提交：`<sha>`
- 来源分支：`master` / `dev`
- 镜像 tag：
  - `radish-dbmigrate:<tag>`
  - `radish-frontend:<tag>`
  - `radish-api:<tag>`
  - `radish-auth:<tag>`
  - `radish-gateway:<tag>`

### 测试部署结论

- 部署情况：已部署 / 未部署
- 最小复核：通过 / 阻塞 / 未执行
- 关联记录：<link or none>
- 说明：
  - <fact 1>
  - <fact 2>

### 生产部署结论

- 部署情况：已部署 / 未部署
- 最小复核：通过 / 阻塞 / 未执行
- 关联记录：<link or none>
- 说明：
  - <fact 1>
  - <fact 2>

### 回滚目标

- 测试环境优先回滚目标：`<tag>` / 无
- 生产环境优先回滚目标：`<tag>` / 无
- 说明：
  - <why this target>

### 已知风险 / 后置项

- <none or item 1>
- <item 2>
```

## 推荐写法

### 1. 发布标识优先写可追溯事实

推荐：

- Git tag：`v26.3.2-release`
- 对应提交：`abc1234`
- 镜像 tag：五个服务统一使用 `v26.3.2-release`

不推荐：

- “最新版”
- “正式环境那套镜像”

### 2. 部署结论只写本轮真实执行范围

推荐：

- 测试部署：已部署，最小复核通过
- 生产部署：未部署，未执行

不推荐：

- “理论上生产也没问题”
- “和上次一样应该正常”

### 3. 回滚目标必须是已知可用锚点

推荐：

- 测试环境优先回滚目标：`v26.3.2-test`
- 生产环境优先回滚目标：无，当前仅有 1 个已知可用 `v*-release`

不推荐：

- “上一版”
- “出问题就回 latest”

## 当前建议

当前仓库的最小分工建议如下：

- `M15` 发布记录模板：负责记录一次发布的总事实
- `M14` 部署后最小复核记录模板：负责记录测试 / 生产部署后的最小复核
- `M15` 回滚演练记录 / 生产回滚预案：负责记录回滚动作或回滚边界

因此每次正式发布后，推荐最少保留三份材料：

- 一份发布记录
- 一份部署后最小复核记录
- 一份明确的回滚目标说明
