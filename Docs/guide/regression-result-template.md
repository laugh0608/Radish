# 回归结论记录模板

> 本页用于沉淀“已经执行完回归之后，如何记录结果”。
>
> 与 [人工验收模板](/guide/manual-acceptance-template) 的区别是：
>
> - 人工验收模板回答“应该怎么验”
> - 回归结论记录模板回答“验完之后怎么记”
> - [变更回归记录模板](/guide/change-regression-record-template) 回答“这次改动整体做了哪些验证、当前能下什么结论”

## 适用场景

以下场景建议至少保留一条结果记录：

- 新功能首轮收口
- 关键联调问题修复后复验
- `M13` 专题回归层补入口、补模板、补脚本后的首次落地
- 准备把某条主线从“当前进行中”转为“可维护”时

## 最小记录字段

建议每条记录至少包含以下内容：

1. **记录日期**
   - 使用绝对日期，避免“今天 / 昨天”口径漂移
2. **记录人**
   - 便于后续追溯是谁执行了这轮回归
3. **专题范围**
   - 明确到专题或功能闭环，不写成泛泛的“论坛回归”
4. **触发原因**
   - 说明为什么做这轮回归，例如“新功能首轮收口”“联调补丁复验”“验证入口收口后首次执行”
5. **执行入口**
   - 对应 `HttpTest`、前端页面、脚本或宿主命令
6. **身份语义影响面（按需）**
   - 若命中 `Identity Guard` 或追加 `validate:identity`，写清命中原因
7. **执行摘要**
   - 只列本轮实际执行的关键步骤，不必重复整份模板
8. **结果**
   - `通过` / `阻塞` / `部分通过`
9. **失败归类 / 环境边界（按需）**
   - 若本轮失败或受限，标注 `contract 漂移 / 默认执行面失败 / 身份语义专题失败 / 受限环境边界`
10. **结论**
   - 当前可得出的工程判断，例如“可转维护”“仍需继续观察”“阻断未解除”
11. **遗留项**
   - 若无可写“无”；若有则只写剩余阻塞或明确后置项

## 推荐格式

```md
### <专题名> 回归记录（YYYY-MM-DD）

- 记录日期：YYYY-MM-DD
- 记录人：<name>
- 专题范围：<topic>
- 触发原因：<why>
- 执行入口：<http / ui / script / host>
- 身份语义影响面：命中 / 未命中
- 命中原因：身份运行时入口 / Auth 协议输出 / 官方协议消费者 / Token 解析 / 默认执行面文档与门禁资产 / 无
- 执行摘要：
  - <step 1>
  - <step 2>
  - <step 3>
- 结果：通过 / 阻塞 / 部分通过
- 失败归类：无 / contract 漂移 / 默认执行面失败 / 身份语义专题失败 / 受限环境边界
- 结论：<current conclusion>
- 遗留项：<none or remaining items>
```

## 放置建议

建议按以下优先级落位：

1. 专题设计文档内，紧跟“人工验收模板”之后
2. 当周 `changelog`，记录阶段性结论
3. 若后续存在 PR 模板或发布清单，再从这些文档引用本模板
4. 若需要整理一次完整变更的验证结果，优先使用 [变更回归记录模板](/guide/change-regression-record-template)
