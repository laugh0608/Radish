# 变更回归记录模板

> 本页用于把一次实际变更的验证结果整理成可复用的记录。
>
> 三份模板的分工如下：
>
> - [人工验收模板](/guide/manual-acceptance-template)：定义“应该怎么验”
> - [回归结论记录模板](/guide/regression-result-template)：定义“验完之后怎么记结果”
> - 本页：定义“这次改动整体做了哪些验证、当前结论是什么”

## 适用场景

以下场景建议使用本模板整理一份记录：

- 一次跨层改动需要同时说明自动化、专题回归与人工验收结果
- 新功能首轮收口，准备转入维护
- 关键补丁修复后，需要给出“是否可合并 / 可继续推进”的工程结论
- 测试部署 / 生产部署复核后，需要把 `M14` 自动化前置与外部访问结论放进同一份记录
- 准备写周志、PR 描述或阶段性汇总时，不希望每次重新组织结构

## 默认粒度

本模板默认服务于“准备合并的一批改动”或“真实发布 / 部署动作”，而不是每一个开发中的本地 commit。

推荐口径如下：

- 本地开发中的连续小提交：
  - 默认不要求每次都写本模板
  - 只做必要验证即可
- 准备发起 `PR -> master`：
  - 对整个改动批次补一份记录
- 测试部署 / 生产部署 / 回滚：
  - 结合发布与部署记录按事实补充

如果把本模板压到每次本地提交，通常会明显拖慢开发效率，也会让记录迅速失去可读性。

## 最小字段

建议每份变更记录至少包含以下内容：

1. **记录信息**
   - 日期、记录人、专题 / 范围
2. **变更摘要**
   - 只写本次改了什么，不展开成 changelog
3. **影响专题**
   - 明确命中哪些专题回归，不写成“已全量验证”
4. **身份语义影响面（按需）**
   - 若命中 `Identity Guard` 或需要追加 `validate:identity`，写清“为什么命中”
   - 推荐直接复用 `check:identity-impact` 输出的原因类别
5. **自动化执行**
   - 例如 `validate:baseline:quick` / `full` / `host` / `validate:ci`
   - 写清结果是通过、阻塞还是未执行
6. **专题回归**
   - 对应 `HttpTest`、专题脚本、专题文档中的最小人工验收顺序
7. **人工验收**
   - 是否执行
   - 执行了哪些关键动作
8. **部署复核（按需）**
   - 若命中测试部署 / 生产部署、反代、证书或 `RADISH_PUBLIC_URL` 变更，补一段部署后最小复核结论
   - 推荐直接复用 [M14 部署后最小复核记录模板](/guide/m14-deployment-review-record-template)
9. **故障归类 / 环境边界（按需）**
   - 若出现失败或跳过，标注 `contract 漂移 / 默认执行面失败 / 身份语义专题失败 / 受限环境边界`
   - 若无可写“无”
10. **结论**
   - 当前工程判断，例如“可合并”“可转维护”“仍需继续观察”
11. **风险 / 后置项**
   - 若无可写“无”

## 推荐格式

```md
## <专题或改动名> 变更回归记录

- 记录日期：YYYY-MM-DD
- 记录人：<name>
- 变更范围：<topic>

### 变更摘要

- <change 1>
- <change 2>

### 影响专题

- <topic 1>
- <topic 2>

### 身份语义影响面（按需）

- 命中情况：命中 / 未命中
- 命中原因：身份运行时入口 / Auth 协议输出 / 官方协议消费者 / Token 解析 / 默认执行面文档与门禁资产 / 无

### 自动化执行

- `npm run validate:baseline:quick`：通过 / 阻塞 / 未执行
- `npm run validate:baseline`：通过 / 阻塞 / 未执行
- `npm run validate:baseline:host`：通过 / 阻塞 / 未执行
- `npm run validate:ci`：通过 / 阻塞 / 未执行

### 专题回归

- `Radish.Api.Tests/HttpTest/<file>.http`
- <manual acceptance or script>

### 人工验收

- 执行情况：已执行 / 未执行
- 摘要：
  - <step 1>
  - <step 2>

### 部署复核（按需）

- 执行情况：已执行 / 未执行
- 记录入口：`M14` 部署后最小复核记录 / 无
- 摘要：
  - <deployment check 1>
  - <deployment check 2>

### 故障归类 / 环境边界（按需）

- 归类：无 / contract 漂移 / 默认执行面失败 / 身份语义专题失败 / 受限环境边界
- 说明：<none or summary>

### 结论

- <current decision>

### 风险 / 后置项

- <none or remaining item>
```

## 推荐写法

### 1. 变更摘要只写本次动作

推荐：

- 新增身份语义防回归扫描并接入 baseline
- 收口 Console 入口权限扫描噪音

不推荐：

- 完整复述所有背景、里程碑和历史决策

### 2. 自动化执行写事实，不写推测

推荐：

- `npm run validate:baseline:quick`：通过
- `npm run validate:baseline:host`：通过（含 `DbMigrate doctor / verify`）
- `npm run validate:ci`：通过（本地已复现当前 `Repo Quality` 最小门禁）

不推荐：

- “理论上应该没问题”
- “应该都能跑”

### 3. 身份语义影响面写“为什么命中”，不要只写“跑了 `validate:identity`”

推荐：

- 命中情况：命中
- 命中原因：默认执行面文档 / 门禁资产

不推荐：

- “命中了，原因见上文”
- “因为改了点脚本”

### 4. 失败归类沿用统一口径，不要临时发明标签

推荐：

- 归类：默认执行面失败
- 说明：`validate:ci` 卡在 `validate:baseline:quick` 的权限扫描

不推荐：

- “CI 有点问题”
- “Identity Guard 好像坏了”

### 5. 人工验收只写本轮真正执行的动作

推荐：

- 已执行
- 摘要：
  - 发帖附带抽奖
  - 父评论参与
  - 浏览记录深链跳转

不推荐：

- 直接复制整份人工验收模板而不区分本轮实际执行内容

## 放置建议

建议按以下优先级使用：

1. PR / 评审描述
2. 周志或阶段汇总
3. 专题文档中的“回归记录”小节

## 当前建议

当前仓库已经有：

- 专题回归索引
- 人工验收模板
- 回归结论记录模板
- [M14 部署后最小复核记录模板](/guide/m14-deployment-review-record-template)

所以下一步优先不是继续补更多模板，而是让实际变更开始按本页结构输出记录，并把 `check:identity-impact` 的命中原因与 `Repo Quality` 失败归类默认写进记录。

当前也可以直接用命令先生成一份自动化初稿，再按本页补齐人工信息：

```bash
npm run collect:change-regression-record -- --title "当前批次" --scope "当前 PR / 改动批次"
```

该脚本当前会优先收口：

- `validate:ci` 的批次级报告
- 显式传入的 `validate:baseline` / `validate:baseline:host` 固定报告
- 显式传入的 `M14` 宿主维护记录
- 当前 changed files 的身份语义影响面命中原因

默认只自动读取 `.tmp/validate-ci-report.md`；其余报告需要通过参数显式传入，避免把历史残留 `.tmp` 文件误带进当前批次记录。

人工验收、专题回归摘要与最终工程判断仍需按本轮真实情况补充，不要把自动生成内容原样当成最终结论。
