# M14 宿主运行与最小可观测性基线（重定义）

## 背景

截至 `2026-04-05`，身份语义 Phase 4 的仓库内启动输入、仓库外兼容边界确认、首轮实施与官方顺序真实回归均已完成，且 `radish-client / radish-console / Radish.Api.AuthFlow.http / radish-scalar` 已完成本轮手工联调确认，当前结论已更新为“Phase 4 正式收口，转入稳定维护”。因此，`M14` 当前不再只是后续候选，而是已经切换为当前正式主线，用于整理宿主运行与最小可观测性基线。

在当前仓库真实资产下，最适合先被重定义并收口的下一阶段，是 `M14`：

- 它不是“完整运维平台建设”
- 也不是“补齐全量可观测性体系”
- 而是把已经存在但仍分散的宿主运行、自检、日志与部署复核能力，重新定义为一条可执行、可验收、可继续拆分的最小基线

## 当前为什么先做 M14

当前仓库已经具备以下事实基础：

- API / Gateway 已具备健康检查入口
- Serilog 已承担当前结构化日志落盘与排障入口
- `DbMigrate doctor` / `verify` 已形成宿主只读自检能力
- `npm run validate:baseline:host` 已把宿主类最小自检纳入统一验证入口
- `npm run check:host-runtime` 已补为运行态健康检查入口
- 测试 / 生产部署口径、镜像推送口径与 `DbMigrate -> Api/Auth -> Gateway` 启动顺序当前已有明确文档

真正缺少的不是“完全没有能力”，而是：

- 这些能力分散在验证基线、部署指南、架构说明和阶段规划中
- 当前还没有一份统一定义明确回答“什么算 M14 完成”
- `M14` 很容易再次被误读为“大而全的运维 / 观测平台工程”

因此，`M14` 本轮先做的是**重定义、入口收口与第一轮执行清单固化**，而不是直接启动新的大工程。

当前执行入口见：[M14 宿主运行首轮执行清单](/guide/m14-host-runtime-checklist)

## M14 的正式定义

`M14` 当前定义为：**围绕现有宿主，收束一套最小可运行、最小可自检、最小可排障的运行基线。**

它要解决的问题不是“如何建设完整观测平台”，而是下面这些更现实的问题：

- 一个宿主或环境异常时，是否有统一入口判断是配置、连接、数据前置、宿主启动还是网关链路问题
- API / Auth / Gateway 的最小运行观测口径是否一致，避免每个宿主各讲各的
- 测试部署与生产部署时，是否能用一套仓库内已有资产完成最小自检和基本复核
- 后续如果真的要做更完整的 Metrics / Tracing / OpenTelemetry，是否已经有清晰前置条件，而不是直接许诺平台化建设

## 范围

### 1. 宿主配置一致性与只读自检

重点是把“宿主能否安全启动、配置是否成型、连接定义是否合理、关键 `ConnId` 是否存在”收束为统一口径。

本阶段依赖并复用：

- `DbMigrate doctor`
- `DbMigrate verify`
- `npm run validate:baseline:host`
- 现有部署文档中的环境配置与启动顺序说明

本阶段输出应能明确回答：

- 当前环境是否具备最小启动前提
- 当前失败更像是配置问题、数据库问题、种子前置问题，还是宿主本身问题

### 2. 健康检查、启动日志与诊断入口

重点不是新增复杂监控系统，而是统一“运行中怎么看、启动失败先看哪”。

本阶段关注：

- API / Gateway / Auth 的健康检查入口是否被文档统一引用
- 启动日志、异常日志、SQL / 慢查询排查口径是否有单一入口
- 关键宿主的最小诊断动作是否能被文档明确描述，而不是继续依赖口口相传

### 3. 测试部署 / 生产部署的最小观测入口

重点是让测试环境与生产环境的复核动作有一致的最小骨架，而不是把部署问题继续拆散到多份零碎文档里。

本阶段关注：

- 部署前后的最小自检顺序
- 宿主启动顺序与镜像口径的回链关系
- 出现异常时优先查看的入口：健康检查、日志、`doctor` / `verify`、部署配置

### 4. 后续扩张边界的前置条件

本阶段要明确写清楚：哪些更重的能力不属于当前 `M14`，以及什么情况下才值得进入下一轮扩张。

## 不在本阶段内

以下内容当前明确**不**属于 `M14` 首轮重定义范围：

- 直接承诺接入完整 OpenTelemetry / Metrics / Tracing 平台
- 直接启动全量 CI、全量 E2E、完整值班 / 告警 / 仪表盘工程
- 把 `M14` 扩成“完整部署平台治理”或“完整运维体系建设”
- 顺手把 `Gateway & BFF` 深化、开放平台接入或额外基础设施专题并入同一阶段

换句话说，`M14` 当前要的是**最小基线**，不是**最终形态**。

## 与现有资产的关系

### 与验证基线的关系

- `M13` 解决的是“仓库里已有验证资产如何形成统一入口”
- `M14` 解决的是“宿主运行后如何形成最小自检与排障基线”
- `validate:baseline:host` 当前已作为 `M14` 的默认宿主验证入口，但不等于 `M14` 本身已经完成

### 与部署指南的关系

- 部署指南负责“怎么部署、怎么启动、怎么复核”
- `M14` 负责把部署后的最小观测、自检与排障边界定义清楚
- 两者应互相回链，但不能互相替代
- 第一轮排障与复核顺序统一收口到 [M14 宿主运行首轮执行清单](/guide/m14-host-runtime-checklist)

### 与 `DbMigrate` 的关系

- `DbMigrate doctor` / `verify` 是当前宿主最小只读自检的核心基础设施
- `M14` 不重复发明新的自检体系，而是先把现有命令纳入统一运行基线
- 只有在现有命令无法覆盖关键宿主判断时，才考虑下一轮补能力

## 首轮验收标准

当以下条件同时成立时，可认为 `M14` 的“重定义”已经完成，并可以继续拆成后续具体工作项：

- 已存在一份单独文档，明确 `M14` 的目标、范围、非范围与验收口径
- `planning/current.md`、`development-plan.md`、`planning/backlog.md`、`architecture/framework.md`、周志等主要入口已统一回链到同一口径
- 已明确 `validate:baseline:host`、`DbMigrate doctor` / `verify`、部署复核入口之间的分工关系
- 已明确“当前不做什么”，避免再次把 `M14` 扩写成完整观测平台或完整运维平台
- 已形成下一轮可继续拆分的主线入口，而不是继续停留在抽象阶段名

## 当前进展与下一步建议

截至 `2026-04-05`，`M14` 的第一轮主路径已完成首轮收口：

- `planning/current.md`、`development-plan.md`、验证基线说明与本页当前已统一回链到同一口径
- 默认执行顺序已固定为“启动前 `validate:baseline:host` -> 启动后 `check:host-runtime`”
- 启动前与启动后两段检查当前都已支持 `--report`、`--report-file`，并统一为 `Summary / Actions` 报告格式
- `Gateway` 当前已明确拆分 `/health` 与 `/healthz` 语义：前者用于最小后端宿主链检查，后者用于更完整的扩展观测
- `Api / Auth / Gateway` 当前已进一步把健康检查判断与启动日志摘要收口到同一批运行时 helper：`Api` 会输出 JWT 验签模式 / Issuer / signing 证书摘要，`Auth` 会输出 OIDC Issuer / 密钥模式 / 证书摘要，`Gateway` 会输出最小探活与扩展观测目标摘要，便于在人工查看 `/health`、`/healthz` 前先核对宿主到底按哪套模式启动
- `Gateway` 当前已把下游观测目标进一步结构化：`console-service` 继续保留在扩展观测层，探活目标固定为 `/healthz`，且失败状态固定记为 `Degraded`，避免前端未启动时再次干扰默认后端宿主链判断

在这个前提下，后续不再重复解决“入口是什么”，而是继续沿着已固定的主路径往下补细节。

完成本页后，`M14` 后续应按以下顺序继续推进：

1. 继续按 [M14 宿主运行首轮执行清单](/guide/m14-host-runtime-checklist) 维持默认执行顺序，不再回退到口头决定先跑什么。
2. 沿着启动前与启动后两段主路径，继续补配置自检、健康检查 / 启动日志、测试 / 生产复核三条线的细项，而不是重新发明阶段入口。
3. 在真实维护记录中继续使用统一的 `Summary / Actions` 报告口径，降低后续排障继续依赖零散终端输出的成本。
4. 只有当这些最小基线已经稳定、且真实暴露出进一步观测缺口时，再决定是否进入更重的 Metrics / Tracing / 平台化建设。
