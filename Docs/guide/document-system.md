# 文档系统设计与实施方案

> 本文定义 Radish 在 M12 阶段的统一文档体系，用于指导 **固定文档挂载**、**在线文档管理** 与 **Markdown 导入导出** 的分阶段落地。
>
> **状态**：方案确认，已进入实施阶段
>
> **最后更新**：2026-03-08

## 1. 目标

Radish 的文档系统分为两部分：

1. **固定文档**
   - 文档源统一放在仓库 `Docs/` 目录
   - 属于项目随仓库交付的固定内容
   - API 启动时自动扫描、同步并挂载到“文档”应用中
   - 默认只读，不允许在应用内直接编辑

2. **在线文档**
   - 由管理员或用户在应用内新建
   - 内容存储在数据库中
   - 适合教程、手册、精品帖子沉淀与迁移内容二次整理

## 2. 命名与产品口径

- 前端统一使用 **“文档”** 作为应用名称
- 不再保留独立的 `radish.docs` 文档站项目
- 不再对外区分 `docs` / `wiki` 两个应用
- `Docs/` 是固定项目文档的唯一真相源

## 3. 固定文档机制

### 3.1 目录约定

固定文档直接放在 `Docs/` 下，按目录分类：

- `Docs/architecture/`
- `Docs/guide/`
- `Docs/frontend/`
- `Docs/features/`
- `Docs/deployment/`
- `Docs/changelog/`
- `Docs/images/`

### 3.2 启动同步

API 启动时执行固定文档同步：

- 扫描 `Docs/**/*.md`
- 读取标题、摘要、正文
- 自动写入 / 更新固定文档索引
- 将站内链接重写为文档应用可跳转链接
- 将图片等资源重写为 `StaticAssetsRequestPath` 可访问路径

### 3.3 配置项

统一配置节为 `Document`：

```json
{
  "Document": {
    "ShowBuiltInDocs": true,
    "BuiltInDocsPath": "Docs",
    "StaticAssetsRequestPath": "/docs-assets"
  }
}
```

说明：

- `ShowBuiltInDocs=false`：关闭固定文档展示，仅保留在线文档
- `BuiltInDocsPath`：固定文档目录
- `StaticAssetsRequestPath`：图片、附件等静态资源访问前缀，当前仅暴露 `Docs/images/` 下的固定文档静态资源，不直接暴露 `Docs/**/*.md` 源文件

## 4. 在线文档机制

在线文档继续使用现有数据库文档模型承载，但产品口径统一为“在线文档”：

- 支持创建、编辑、发布、归档
- 支持 Markdown 导入 / 导出
- 支持版本历史与回滚
- 用于教程、手册、精品内容与外部内容迁移

## 5. Markdown 导入导出定位

Markdown 导入导出不是固定文档主存储方式，而是 **迁移桥梁**：

- **导入**：把外部平台帖子 / 文档迁移进在线文档
- **导出**：把在线文档导出为 Markdown，便于备份与迁移

## 6. 当前实施顺序

1. 删除 `radish.docs` 工程与相关启动脚本
2. 将文档源统一迁移到 `Docs/`
3. 增加 `Document.ShowBuiltInDocs` 配置
4. 打通 API 启动时固定文档同步
5. 前端统一收口为“文档”应用
6. 保留在线文档编辑与 Markdown 导入导出能力

## 7. 非目标

当前阶段不包含：

- 多人协同编辑
- 全文检索引擎
- ZIP 批量导入导出
- 固定文档在线编辑回写到 `Docs/`

## 8. 备注

当前实现阶段允许内部模块继续沿用 `Wiki` 命名的代码实体与接口，以降低改造成本；但对外产品口径、文档与前端界面统一使用“文档”。
