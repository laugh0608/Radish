# 写操作可靠性与并发保护治理

> 状态：`规划专题 / 下一批候选评审`
>
> 记录日期：`2026-06-19`（Asia/Shanghai）
>
> 适用范围：用户或管理员发起的会改变业务状态、资产、权限、内容、通知、关系链或运行时配置的写操作。

## 背景

P3-10 后置安全治理已经完成前端敏感日志脱敏、支付口令哈希升级，以及商城购买 / 萝卜币转账的首批幂等治理。当前需要把这些点状治理沉淀成统一判断口径：以后遇到新的写操作，不应临时决定是否要幂等、并发保护、限流、审计或重试语义。

本专题负责回答：

- 哪些写操作需要服务端幂等。
- 哪些写操作只需要唯一约束、业务去重或事务保护。
- 哪些写操作进入多实例或高并发后才需要 Redis / 分布式锁 / 原子计数。
- 哪些能力属于发布候选前回归，哪些可以作为后续产品开发。

本专题不替代 [支付与转账幂等治理](/guide/payment-idempotency-governance) 和 [Redis 与缓存治理专题](/planning/redis-cache-governance)。支付幂等是首批已落地实现，Redis 治理是多实例 / 高并发后的后置工具箱；本页是二者之间的写操作分级与决策入口。

## 治理目标

1. 写操作的重复点击、网络重试、刷新重提和客户端超时重发，不应造成重复扣款、重复发放、重复关系变更或重复配置变更。
2. 写操作的并发冲突必须由服务端最终保护，前端按钮禁用、确认弹窗和本地状态只作为体验层辅助。
3. 数据库仍是业务真值；Redis、缓存、分布式锁和队列只能作为限流、协同、加速或跨实例协调手段。
4. 每个新治理批次必须先说明适用操作、数据真值、失败重试口径、验证入口和不做范围。
5. 不用层层兜底掩盖契约不清；优先明确对象边界、唯一键、事务边界和终态响应。

## 写操作分级

| 分级 | 典型操作 | 主要风险 | 默认治理 |
| --- | --- | --- | --- |
| A. 资产 / 库存 / 交易写入 | 商城购买、萝卜币转账、管理员调账、订单取消退款、权益发放 | 重复扣款、重复发放、余额或库存不一致 | 服务端幂等、请求摘要绑定、数据库事务、唯一约束或条件更新、审计 |
| B. 内容发布与互动 | 发帖、评论、轻回应、点赞、投票、抽奖参与、编辑历史 | 重复提交、刷屏、重复投票、状态抖动 | 业务唯一键、频率限制、内容去重窗口、必要时幂等 |
| C. 身份 / 安全写入 | 注册、登录刷新、支付口令设置、资料展示名、绑定邮箱 | 凭证混用、重复注册、会话状态漂移 | 规范化、唯一约束、审计、确认参数和定向测试 |
| D. 管理与治理写入 | 系统设置、权限、内容审核、商品管理、订单治理 | 高风险配置误改、审计缺失、并发覆盖 | 风险等级、修改原因、确认参数、审计历史、乐观并发或版本号 |
| E. 玩法与奖励写入 | 经验奖励、神评 / 沙发奖励、宠物照顾、每日任务 | 重复奖励、冷却绕过、日上限失效 | 奖励幂等、冷却 / 上限校验、状态流水、定向测试 |

## 决策规则

### 需要服务端幂等的情况

满足以下任一条件时，应优先评估幂等 key 和请求摘要：

- 用户确认一次操作后，客户端可能因为网络失败、超时、刷新或重试重复提交。
- 写入会产生订单、交易流水、权益、奖励、库存扣减、余额变化或外部可见审计记录。
- 操作成功后客户端需要拿到同一个业务编号或终态响应。
- 未来 Flutter、Web、Console 或开放 API 会复用同一个写入口。

不满足这些条件时，不应为了形式统一强行增加幂等表；优先使用业务唯一键、事务和定向测试。

### 需要数据库唯一约束或条件更新的情况

以下场景优先用数据库保护真值：

- 同一用户对同一对象只能存在一条关系，例如关注、点赞、收藏、投票。
- 库存、余额、次数或状态需要在单个事务里判断并更新。
- 写操作可以通过自然业务键表达唯一性，例如 `UserId + TargetType + TargetId`。

前端禁用按钮不能代替这些保护。

### 需要版本号或乐观并发的情况

以下场景应评审版本语义：

- 管理员编辑同一配置、商品、权限或内容审核结果，后提交者可能覆盖前提交者。
- 用户编辑资料、帖子、文档或设置时需要提示“内容已被更新”。
- 历史记录和审计需要知道覆盖发生在什么版本之后。

首批不默认给所有表加版本号；只在治理对象确实存在覆盖风险时引入。

### 需要 Redis / 分布式锁的情况

Redis 只在以下条件满足后进入实现评审：

- 已进入多实例部署，进程内锁或内存计数不能保证一致。
- 高频计数、限流、在线状态、SignalR 分组或短窗口去重已经超出数据库普通查询承载。
- 数据库事务和唯一约束仍是最终保护，但需要 Redis 辅助削峰或跨实例协调。

不因为 Redis 已经部署就把所有写操作迁到 Redis。资产类操作不能把 Redis 当作最终真值。

## 后续候选推进

`WOG-1` 已完成盘点矩阵，后续代码批次按以下顺序筛选：

1. 已有用户主路径且重复提交会造成资产或权益错误。
2. 已经有多端承接计划，需要统一接口契约。
3. 已有业务去重但缺少测试或文档说明。
4. 发布候选前回归会反复命中的高风险写入口。

下一批建议先评审 `WOG-2 内容互动关系写入与计数一致性`。如果候选进入代码并触达接口、数据库结构、运行时行为或跨端契约，应先输出方案并等待确认。

## 不做范围

- 不启动完整钱包、经济扩展、资产风控平台或开放 API 签名平台。
- 不把浏览器通用 `sign`、字段级加密、安全会话或设备绑定并入本专题。
- 不把 Redis 分布式锁作为所有写操作的默认前置。
- 不把前端按钮禁用、loading 状态或本地防抖写成服务端可靠性方案。
- 不一次性改完所有写入口；每个代码批次必须有明确对象、风险、验证和停止线。
- 不为了治理感新增空泛抽象；只有能稳定契约、减少重复或降低真实风险时才新增通用组件。
